# 一、模块概述

文件情报模块用于解析分析在其余模块扫描过程中发现的，敏感文件等的解析分析，实现自动下载，自动解析输出的功能。
需要解析：例如.git等的敏感文件

# 二、功能拆分

## 2.1 文件下载
可以发现泄露的文件信息（其他模块发现后发送连接过来），直接尝试下载文件，并保存到指定目录，并严格管理下载文件的权限，禁止执行文件，仅允许读取分析，并在项目完成后执行删除。
## 2.2 文件解析
针对不同的泄露的文件，采取不同的解析提取逻辑。
## 2.3 格式化输出报告
对文件提取出的信息，进行格式化输出。
## 2.4 处理输入信息
可以正确的处理总控agent下发的信息并执行任务。

# 四、提示词设计

文件情报模块的提示词设计目标，不是让模型“自由总结文件内容”，而是让模型在明确边界下完成文件识别、类型路由、敏感信息提取、风险判断和结构化输出。提示词必须服务于第五部分的解析逻辑，因此应固定为“主提示词 + 类型补充提示 + 输出约束 + 降级提示”的组合模式，而不是每次临场拼接。

## 4.1 设计目标

文件情报模块提示词需要满足以下目标：

1. 能理解总控agent下发的任务上下文，明确当前文件来自哪里、属于哪个目标、由哪个模块发现。
2. 能根据文件名、路径、扩展名、文件头和内容片段判断文件真实类型，避免只按后缀名处理。
3. 能按照不同文件类型切换不同提取重点，例如 `.git` 重点恢复仓库结构，`.env` 重点提取凭证，日志文件重点提取异常与线索。
4. 能将提取结果统一输出为结构化 JSON，便于后续报告模块、复核模块和状态存储模块直接消费。
5. 能在证据不足、文件损坏、格式异常时主动降级，保留可追溯结果，而不是输出幻觉式结论。

## 4.2 提示词设计原则

文件情报模块提示词建议固定遵循以下原则：

1. 只解析文件，不执行文件，不运行脚本，不连接数据库，不使用提取出的凭证进行任何访问尝试。
2. 只基于输入中给出的文件内容、元数据和上下文做判断，不把推测写成事实。
3. 高风险信息可以识别和标记，但默认不回显完整密钥、完整密码、完整 Token。
4. 先判断文件类型，再调用对应解析逻辑，不允许“所有文件都按同一种方式总结”。
5. 输出必须结构化、可审计、可落库，不输出长篇散文式分析。

## 4.3 主提示词

主提示词用于文件情报模块处理任意待解析文件时的通用约束，负责完成类型判断、解析任务选择和统一输出。

建议主提示词正文如下：

```text
你是“文件情报模块agent”的文件解析单元。你的职责是对已经下载到本地、且被明确授权分析的文件做只读解析，提取其中的结构化情报，并输出统一结果。

你的唯一任务是：
1. 识别文件真实类型；
2. 按文件类型选择对应解析重点；
3. 提取敏感信息、线索信息、结构信息和风险信息；
4. 输出结构化 JSON 结果；
5. 在格式异常、内容损坏或证据不足时输出降级结果。

你必须遵守以下规则：
1. 禁止执行文件、禁止运行脚本、禁止连接外部服务、禁止使用文件中的凭证做验证。
2. 禁止把猜测写成已确认事实；不确定内容必须标记为“疑似”或“待复核”。
3. 禁止输出散文式报告，只允许输出结构化结果。
4. 必须优先判断文件类型，再进入对应解析逻辑。
5. 若文件内容与扩展名不一致，应以内容特征和结构特征为准。
6. 若文件只解析出部分结果，也必须输出“部分解析成功”，不能静默失败。
7. 对密钥、密码、Token、证书内容默认做脱敏展示，只保留摘要、指纹、前后缀或长度信息。

你需要优先识别以下文件类别：
1. 版本控制文件：.git、.svn、.hg
2. 源码映射文件：.js.map
3. 配置文件：.env、.ini、.properties、.yaml、.json、config.php、settings.py、web.config
4. 备份与临时文件：.bak、.old、.backup、.swp、*~
5. 日志文件：.log、debug.log
6. 数据库文件：.sql、.db、.sqlite
7. 接口文档：swagger.json、openapi.json、openapi.yaml、/api-docs
8. 系统索引文件：.DS_Store、Thumbs.db
9. 密钥证书文件：.pem、.key、id_rsa
10. 压缩包与制品：.zip、.tar.gz、.tgz
11. CI/CD 与部署配置：.gitlab-ci.yml、Dockerfile

你的处理步骤固定如下：
第一步：读取输入中的文件元数据，包括文件名、来源URL、内容类型、大小、哈希值、文件头、文本片段。
第二步：判断文件真实类型，并给出类型判断依据。
第三步：根据文件类型切换提取重点。
第四步：提取结构化发现，包括账号、密码、Token、密钥、接口、路径、仓库信息、数据库信息、部署信息等。
第五步：判断风险等级，并给出风险标签。
第六步：按照统一 JSON 结构输出。

若文件无法完整解析，则按以下顺序降级：
1. 保留文件元数据和类型猜测；
2. 提取可见字符串和关键片段；
3. 保留错误原因和待复核标记；
4. 输出部分结果，不允许直接输出空结论。
```

## 4.4 类型补充提示

主提示词确定“怎么工作”，类型补充提示确定“当前文件重点看什么”。建议将第五部分的解析逻辑固化为类型补充提示模板，在路由到具体文件类型后拼接到主提示词后面。

### 4.4.1 版本控制文件补充提示

```text
当前文件被判定为版本控制相关文件。

你的提取重点是：
1. 仓库结构、分支、标签、修订号；
2. 远程仓库地址、用户名、邮箱、子模块配置；
3. 可恢复的源码路径、配置路径、部署文件路径；
4. 提交历史、作者信息、对象索引、工作副本信息；
5. 可能继续补抓的路径线索。

若是 .git，优先关注 HEAD、config、index、refs、logs、objects。
若是 .svn，优先关注 entries、wc.db、text-base、修订号和仓库 URL。
若是 .hg，优先关注 hgrc、branch、store、requires。
```

### 4.4.2 配置文件补充提示

```text
当前文件被判定为配置相关文件。

你的提取重点是：
1. 数据库连接信息；
2. 用户名、密码、Token、Secret、API Key；
3. Redis、MQ、SMTP、OSS、LDAP 等中间件与外部服务配置；
4. 运行环境标识，如 dev、test、staging、prod；
5. 调试开关、管理后台路径、日志路径、上传路径。

若为代码式配置，只允许做静态文本提取，不得执行。
若同一文件内可拼出完整连接信息，应标记为高危。
```

### 4.4.3 日志文件补充提示

```text
当前文件被判定为日志文件。

你的提取重点是：
1. 时间范围、日志级别、请求ID、线程ID；
2. 异常堆栈、类名、方法名、源码路径；
3. SQL 语句、表名、字段名、数据库报错；
4. Cookie、Session、JWT、手机号、邮箱等敏感数据；
5. 可形成漏洞线索的错误模式，如鉴权失败、SQL报错、上传报错、路径泄露。

日志文件不要只做关键词搜索，要尽量按事件切片提取。
```

### 4.4.4 数据库文件补充提示

```text
当前文件被判定为数据库导出或数据库文件。

你的提取重点是：
1. 数据库名、表名、字段名、索引、视图、触发器；
2. 用户表、配置表、权限表、日志表等高价值数据对象；
3. 密码哈希、盐值、邮箱、手机号、Token、权限字段；
4. 数据量级和敏感表识别；
5. 数据库版本、导出路径、注释信息。

禁止执行 SQL；只允许静态解析或只读结构读取。
```

### 4.4.5 密钥证书文件补充提示

```text
当前文件被判定为密钥或证书文件。

你的提取重点是：
1. 文件是私钥、公钥还是证书；
2. 算法类型、位数、是否加密、证书用途；
3. Subject、Issuer、SAN、有效期、指纹；
4. SSH 私钥注释和用途说明；
5. 该文件是否足以构成高危接管风险。

禁止输出完整密钥内容，只输出类型、摘要、指纹和风险判断。
```

### 4.4.6 压缩包与制品文件补充提示

```text
当前文件被判定为压缩包或发布制品。

你的提取重点是：
1. 文件清单、目录结构、核心入口文件；
2. 是否包含源码、配置、SQL、证书、日志等高价值内容；
3. 是否存在路径穿越、恶意构造或超大解包风险；
4. 解包后二次分发到哪些解析器；
5. 根目录 README、部署文件、配置文件中的关键信息。
```

## 4.5 输入提示负载设计

为了让模型稳定工作，建议文件情报模块传给模型的提示负载固定包含以下对象：

1. `task_context`：任务 ID、目标主体、发现来源模块、证据链引用。
2. `file_meta`：文件名、来源 URL、本地路径、大小、哈希、Content-Type、下载时间。
3. `file_probe`：文件头、编码、首段文本、魔数识别结果、扩展名判断结果。
4. `file_content`：文件正文或分块内容。
5. `parse_policy`：允许的解析模式、禁止的行为、脱敏策略、输出字段要求。

建议输入示例如下：

```json
{
  "task_context": {
    "task_id": "task_20260417_0001",
    "target": "example.com",
    "discovered_by": "基础发现模块",
    "evidence_ref": "evidence://file/001"
  },
  "file_meta": {
    "file_name": ".env",
    "source_url": "https://example.com/.env",
    "local_path": "/data/files/task_20260417_0001/.env",
    "size": 2481,
    "sha256": "abc123",
    "content_type": "text/plain"
  },
  "file_probe": {
    "extension_guess": "env",
    "magic_guess": "text",
    "encoding": "utf-8",
    "head_preview": "DB_HOST=127.0.0.1"
  },
  "file_content": "DB_HOST=127.0.0.1\nDB_PORT=3306\nDB_USERNAME=admin\nDB_PASSWORD=***",
  "parse_policy": {
    "allow_execute": false,
    "mask_secrets": true,
    "required_output": "structured_json"
  }
}
```

## 4.6 输出提示约束

文件情报模块提示词应强制模型输出统一结构，避免不同文件类型输出风格漂移。建议统一输出以下结构：

```json
{
  "parse_result": {
    "file_type": "",
    "type_confidence": 0,
    "type_evidence": [],
    "parse_status": "",
    "risk_level": "",
    "risk_tags": [],
    "key_findings": [],
    "secrets": [],
    "assets": [],
    "artifact_refs": [],
    "followup_paths": [],
    "summary": "",
    "need_review": false,
    "review_reason": "",
    "error": {
      "has_error": false,
      "error_code": "",
      "error_message": ""
    }
  }
}
```

其中字段约束建议如下：

1. `file_type` 必须是明确类别，不允许写“未知但像某种文件”这种自然语言。
2. `type_confidence` 用 0 到 1 表示类型判断置信度。
3. `parse_status` 只能从 `success`、`partial_success`、`failed` 中选择。
4. `key_findings` 用于承载结构化结论摘要。
5. `secrets` 中每条敏感信息必须包含 `secret_type`、`masked_value`、`location`、`risk_reason`。
6. `need_review` 用于标记损坏文件、冲突证据、低置信判断或高危发现。

## 4.7 降级与复核提示词

当文件损坏、格式异常、内容过大或解析器无法确定文件类型时，不应直接失败，而应切换到降级提示词。

建议降级提示词正文如下：

```text
当前文件无法按标准路径完整解析。你现在进入“降级提取模式”。

你的目标不是给出完整结论，而是最大化保留可追溯线索。

你必须执行以下动作：
1. 保留文件元数据和来源信息；
2. 给出最可能的文件类型猜测和依据；
3. 提取可见字符串、关键路径、URL、域名、邮箱、账号名、Token 特征串；
4. 明确说明失败原因、损坏位置或结构异常原因；
5. 判断是否需要人工复核。

禁止因为无法完整解析就输出空结果。
禁止虚构不存在的配置项、表名、接口或凭证。
输出格式仍然必须与标准 parse_result 结构一致，只是 parse_status 改为 partial_success 或 failed。
```

## 4.8 落地约束

为了让第四部分真正可落地，建议固定以下约束：

1. 主提示词固定不变，类型补充提示按文件类别动态拼接。
2. 文件类型判断结果必须回写，并且要保留判断依据。
3. 所有高危文件解析结果默认进入脱敏输出。
4. 提示词中不得出现“如果你觉得合理可以自由发挥”之类开放表述。
5. 提示词输出结构必须与第六部分的输入输出设计保持一致，不允许一版一套格式。
6. 提示词版本需要可配置、可审计，建议记录 `prompt_version` 和 `parser_profile`。

# 五、文件解析逻辑

文件情报模块进入解析阶段后，不直接按“后缀名”粗暴处理，而是先完成统一预处理，再根据泄露文件类型进入不同解析器。统一预处理流程如下：

1. 识别文件类型：结合 URL 路径、文件名、扩展名、Content-Type、文件头特征判断真实类型，避免被伪装扩展名误导。
2. 生成文件指纹：计算 `sha256`、记录文件大小、编码格式、下载时间、来源 URL，作为去重和证据链基础。
3. 只读安全打开：所有文件仅允许只读解析，禁止执行、禁止解释器直接运行、禁止自动加载依赖。
4. 选择解析器：按照“版本控制类、源码映射类、配置类、备份类、日志类、数据库类、接口文档类、系统索引类、密钥证书类、压缩包类、CI/CD 类”进入对应处理流程。
5. 结果标准化：不同解析器最终统一输出 `file_type`、`source_url`、`local_path`、`parse_status`、`key_findings`、`secrets`、`artifact_refs`、`risk_tags` 等字段，便于后续格式化报告模块消费。

## 5.1 版本控制文件

### 5.1.1 .git文件

`.git` 泄露优先级最高，解析目标不是只读取单个文件，而是尽可能恢复仓库结构、提交历史和敏感配置。解析逻辑如下：

1. 优先探测核心文件：`.git/HEAD`、`.git/config`、`.git/index`、`.git/packed-refs`、`.git/refs/*`、`.git/logs/HEAD`、`.git/objects/*`。
2. 若 `HEAD` 和 `refs` 可读，则提取默认分支、当前提交、历史分支名、标签名。
3. 若 `config` 可读，则提取远程仓库地址、用户名邮箱、子模块配置、凭证残留、代理配置。
4. 若 `index` 可读，则解析暂存区文件清单，恢复仓库中文件路径、文件模式、文件哈希。
5. 若 `objects` 可读，则按 `commit/tree/blob` 递归还原源码树；优先恢复配置文件、脚本文件、密钥文件、部署文件。
6. 若只能获取部分对象，则保留“部分恢复”状态，输出已恢复文件列表和缺失对象列表，供后续补抓。
7. 对恢复出的文本文件继续做二次解析，提取账号、口令、Token、数据库连接串、内部域名、接口路径等高价值信息。

输出重点包括：仓库远程地址、分支与标签、提交人信息、源码目录树、敏感文件路径、疑似密钥/口令、可继续下载的对象路径。

### 5.1.2 .svn文件

`.svn` 泄露需要兼容不同版本工作副本格式，不能只按一种结构解析。解析逻辑如下：

1. 优先识别 `.svn/entries` 与 `.svn/wc.db`，判断是旧版 SVN 还是新版工作副本。
2. 若存在 `entries`，解析仓库 URL、版本号、目录结构、已检出文件列表。
3. 若存在 `wc.db`，只读打开 SQLite，提取 `NODES`、`REPOSITORY`、`ACTUAL_NODE` 等表中的仓库地址、本地路径、修订号、文件节点信息。
4. 尝试定位 `.svn/text-base/` 或工作副本缓存，恢复原始源码内容。
5. 提取用户名、仓库根路径、分支路径、历史修订号，识别是否可进一步推导出完整仓库目录。

输出重点包括：仓库地址、工作副本路径、修订号、可恢复文件列表、源码内容、残留认证信息。

### 5.1.3 .hg文件

`.hg` 泄露通常不如 `.git` 常见，但仍可能暴露完整源码和仓库配置。解析逻辑如下：

1. 探测 `.hg/requires`、`.hg/hgrc`、`.hg/branch`、`.hg/store/*`。
2. 从 `hgrc` 提取远程仓库地址、认证配置、路径别名。
3. 从 `branch`、`cache`、`store` 中提取分支名、索引文件、版本对象。
4. 若能恢复 revlog 或文件清单，则尽量还原源码目录与关键源码文件。
5. 对恢复出的源码继续执行配置提取、接口提取和密钥扫描。

输出重点包括：仓库地址、分支信息、恢复文件、敏感配置、可疑凭证。

## 5.2 源码映射文件

### 5.2.1 `.js.map` / SourceMap 文件

SourceMap 的价值在于恢复前端源码和隐藏接口。解析逻辑如下：

1. 校验文件是否为合法 JSON，并识别 `version`、`file`、`sources`、`sourcesContent`、`names`、`mappings` 字段。
2. 若存在 `sourcesContent`，直接恢复原始源码；若不存在，则根据 `sources` 反推原始源码路径。
3. 提取源码中的接口路径、请求方法、环境变量名、第三方服务地址、调试开关、鉴权逻辑、前端埋点配置。
4. 识别构建框架特征，如 `webpack://`、`vite://`、`src/`、`node_modules/`，判断项目技术栈。
5. 将恢复出的源码按文件维度存档，并对其中的 `.env` 引用、API Key、调试账号、GraphQL Query、隐藏路由做重点标注。

输出重点包括：恢复源码列表、接口清单、隐藏页面路径、前端框架信息、可疑密钥与调试参数。

## 5.3 配置文件

### 5.3.1 `.env`、`.ini`、`.properties`、`.yaml`、`.json`

纯配置型文件以键值提取为主，核心是识别“密钥类字段”和“连接类字段”。解析逻辑如下：

1. 自动识别文件编码和分隔格式，按键值对、段落、对象树展开为标准字典结构。
2. 对键名做语义分类，如 `password`、`token`、`secret`、`apikey`、`access_key`、`db_host`、`redis_url`、`smtp`、`oss`、`bucket`。
3. 对值做模式匹配，识别数据库连接串、云服务密钥、JWT、SMTP 凭证、LDAP 参数、Kafka/RabbitMQ 地址、对象存储配置。
4. 若同一文件中同时出现主机、端口、用户名、密码，则自动拼接可用连接描述，标记为高危。
5. 对多环境配置进行区分，识别 `dev/test/staging/prod`，避免把测试配置和生产配置混淆。

输出重点包括：配置项列表、凭证类字段、连接类字段、环境标识、服务依赖清单。

### 5.3.2 `config.php`、`settings.py`、`application.py` 等代码式配置

代码式配置不能直接执行，必须做静态解析。解析逻辑如下：

1. 仅做只读文本解析，不通过 PHP/Python 解释器执行文件。
2. 优先匹配数组、字典、常量定义、赋值语句中的配置项。
3. 提取数据库连接、缓存配置、Session 配置、上传路径、日志路径、调试开关、管理员邮箱、第三方 OAuth 参数。
4. 若文件包含函数或条件分支，仅提取静态可见字面量，无法确定的值标记为“动态配置”。
5. 与源码路径、框架目录特征联动，识别 Laravel、ThinkPHP、Django、Flask、Spring Boot 等框架场景。

输出重点包括：框架类型、静态可见配置项、动态配置提示、疑似生产配置。

### 5.3.3 `web.config`

`web.config` 属于 XML 结构配置，重点是站点路径和连接串。解析逻辑如下：

1. 解析 XML 节点，重点提取 `appSettings`、`connectionStrings`、`system.webServer`、`authorization`、`handlers`。
2. 提取站点根路径、重写规则、目录映射、默认文档、错误页、自定义处理器。
3. 识别数据库连接串、Windows 身份认证配置、IIS 模块暴露信息。
4. 标记是否开启调试、是否存在备份目录映射、是否存在危险 handler。

输出重点包括：连接串、物理路径、访问控制信息、重写规则、调试状态。

## 5.4 备份文件与临时文件

### 5.4.1 `*.bak`、`*.old`、`*.backup`、`*~`

备份文件的关键不是文件名本身，而是推断其原始文件身份。解析逻辑如下：

1. 根据去除备份后缀后的名称推断原始文件类型，例如 `config.php.bak` 按 PHP 配置文件解析。
2. 若存在同名在线文件，则对比差异，提取“只在备份中出现”的敏感配置、旧密码、下线路由、废弃接口。
3. 对纯文本备份继续按源码、配置、脚本三类进入二级解析器。
4. 对二进制备份仅提取元数据和字符串片段，避免误执行。

输出重点包括：原始文件推断结果、差异内容、残留密钥、废弃接口或路径。

### 5.4.2 `*.swp`

Vim 交换文件通常保留了尚未保存的编辑内容。解析逻辑如下：

1. 识别 Vim swap 文件头，提取原始文件名、编辑时间、用户信息。
2. 尝试恢复可见文本片段，提取尚未落盘的配置、代码、注释。
3. 若无法完整恢复，则保留字符串扫描结果，提取 URL、账号、口令、SQL 语句。

输出重点包括：原始文件路径、编辑用户、文本恢复片段、敏感字符串。

## 5.5 日志文件

### 5.5.1 `*.log`、`debug.log`

日志文件的重点是按事件切片，而不是整文件暴力搜索。解析逻辑如下：

1. 按时间戳、请求 ID、线程 ID、异常堆栈边界切分日志事件。
2. 识别日志级别，优先处理 `ERROR`、`WARN`、`DEBUG` 中包含认证、数据库、文件路径、内部地址的信息。
3. 提取异常堆栈中的源码路径、类名、方法名、中间件信息、SQL 语句、表名、字段名。
4. 识别日志中泄露的 Cookie、Session、JWT、Access Token、手机号、邮箱、身份证号等敏感数据。
5. 统计高频报错点，形成“疑似漏洞线索”，如 SQL 报错、权限报错、文件上传报错、SSRF 回显。

输出重点包括：时间范围、异常类型、内部路径、数据库信息、认证数据、漏洞线索摘要。

## 5.6 数据库导出与数据库文件

### 5.6.1 `*.sql`

SQL 导出文件解析分为结构信息和数据样本两层。解析逻辑如下：

1. 提取数据库名、表名、字段名、索引、视图、存储过程定义。
2. 识别 `INSERT`、`COPY` 等数据导出语句，统计每张表的数据量级。
3. 对账号表、权限表、日志表、配置表优先抽取样本数据，识别密码哈希、盐值、邮箱、手机号、Token。
4. 对 SQL 中出现的注释、主机地址、导出路径、数据库版本进行额外记录。
5. 不执行 SQL，只做静态解析，避免对本地环境产生副作用。

输出重点包括：库表结构、敏感表识别、账号样本、哈希算法线索、数据库版本信息。

### 5.6.2 `*.db`、`*.sqlite`

SQLite 类数据库文件可直接只读查询结构，但禁止修改。解析逻辑如下：

1. 只读打开数据库文件，提取 `sqlite_master` 中的表、索引、触发器、视图。
2. 识别高价值表名，如 `users`、`admin`、`config`、`token`、`orders`、`logs`。
3. 对高价值表做有限样本抽取，不全量导出超大表，避免内存和报告膨胀。
4. 识别密码字段、权限字段、API Token、设备标识、用户隐私字段。
5. 若数据库损坏，则退化为字符串扫描，尽量提取残留库表名和明文片段。

输出重点包括：库表清单、敏感表、字段分类、抽样数据、损坏状态说明。

## 5.7 接口文档文件

### 5.7.1 `swagger.json`、`openapi.json`、`openapi.yaml`、`/api-docs`

接口文档的核心是恢复完整 API 资产清单。解析逻辑如下：

1. 识别 OpenAPI/Swagger 版本，解析 `servers`、`paths`、`components`、`securitySchemes`、`tags`。
2. 按接口维度提取路径、方法、参数、请求体、返回体、鉴权方式、示例值。
3. 标记无需鉴权、调试接口、上传接口、文件下载接口、管理接口等高风险接口。
4. 从 schema 中提取实体字段，补充用户、订单、权限、日志等业务对象模型。
5. 与 JS 解析结果联动，对文档中存在但页面未显式暴露的接口做“隐藏接口”标注。

输出重点包括：接口总数、接口列表、鉴权方式、管理接口、隐藏接口、高危操作接口。

## 5.8 系统索引与目录泄露文件

### 5.8.1 `.DS_Store`

`.DS_Store` 主要用于恢复目录中文件名。解析逻辑如下：

1. 解析目录项记录，提取文件名、目录名、显示属性。
2. 还原当前目录下可能存在但未直接暴露的文件和子目录。
3. 对提取出的文件名进入“二次探测队列”，尝试组合下载敏感文件。

输出重点包括：目录清单、可疑文件名、建议补抓路径。

### 5.8.2 `Thumbs.db`

`Thumbs.db` 可用于恢复缩略图及文件索引。解析逻辑如下：

1. 提取缩略图索引、原始文件名、图片类型。
2. 标记可能存在的后台截图、文档截图、源码截图、身份证明图片。
3. 如可恢复缩略图，则仅输出缩略图元数据与引用，不在解析阶段做图像识别执行链。

输出重点包括：文件名索引、图片数量、缩略图线索。

## 5.9 密钥与证书文件

### 5.9.1 `*.pem`、`*.key`、`id_rsa`

密钥类文件风险极高，解析时只做元数据识别，不做任何连接尝试。解析逻辑如下：

1. 识别 PEM 块类型，如 `PRIVATE KEY`、`RSA PRIVATE KEY`、`CERTIFICATE`、`PUBLIC KEY`。
2. 判断是否为私钥、证书、公钥，识别算法类型、位数、是否加密。
3. 从证书中提取 `Subject`、`Issuer`、`SAN`、有效期、用途、指纹。
4. 对 SSH 私钥识别注释、用途、是否设置口令保护。
5. 将密钥材料标记为最高危，默认不展示完整内容，仅输出摘要、指纹和类型。

输出重点包括：密钥类型、算法、指纹、是否加密、证书主体、适用域名。

## 5.10 压缩包与发布制品

### 5.10.1 `*.zip`、`*.tar.gz`、`*.tgz`

压缩包解析重点是安全解包和递归分发。解析逻辑如下：

1. 先列出压缩包内文件清单、层级深度、总大小，再决定是否解包。
2. 解包前检查路径穿越风险，禁止出现 `../`、绝对路径覆盖等危险条目。
3. 解包后按内部文件类型再次分发到对应解析器，例如源码、配置、SQL、证书、日志。
4. 优先解析根目录下的 `README`、配置文件、部署文件、源码入口文件。
5. 对超大压缩包限制递归层数、解包总量和单文件上限，避免被恶意压缩包拖垮。

输出重点包括：压缩包目录树、核心文件列表、解包风险、二次解析结果汇总。

## 5.11 CI/CD 与部署配置文件

### 5.11.1 `.gitlab-ci.yml`、`Dockerfile`

部署配置文件的价值在于暴露镜像、环境变量、内部仓库和发布流程。解析逻辑如下：

1. 解析构建阶段、部署阶段、镜像名称、构建参数、缓存路径、脚本入口。
2. 提取环境变量名、私有镜像仓库地址、制品路径、部署目标、云服务引用。
3. 标记硬编码凭证、私有 Registry、SSH 部署脚本、Kubernetes 配置引用、容器暴露端口。
4. 对 `Dockerfile` 额外提取基础镜像、工作目录、复制路径、启动命令、包管理命令，辅助判断技术栈和攻击面。
5. 不执行任何脚本，只做静态解析和命令字符串提取。

输出重点包括：发布流程、镜像信息、环境变量、部署目标、硬编码凭证、服务端口。

## 5.12 解析失败时的降级策略

不同类型文件在解析失败时需要统一降级，避免整个模块中断。降级规则如下：

1. 文本类文件解析失败时，退化为字符串扫描，至少提取 URL、域名、邮箱、IP、Token 特征串。
2. 结构化文件解析失败时，保留原始片段、报错原因和文件指纹，标记为“待人工复核”。
3. 二进制文件无法完整解析时，仅提取文件头、元数据、可见字符串和熵值特征。
4. 对高危文件如密钥、数据库、版本控制目录，即使只解析出部分信息，也必须输出高危告警，不允许静默忽略。
5. 所有降级结果都要回写统一结构，保证格式化输出报告模块仍可消费。

# 六、输入与输出

文件情报模块的输入与输出设计目标，是让总控agent、文件下载阶段、文件解析阶段、报告阶段之间可以稳定对接。该部分不追求“自然语言兼容”，而是追求结构稳定、字段明确、可审计、可扩展。

## 6.1 输入设计

文件情报模块建议统一接收一个 `file_intel_input` 对象，作为唯一标准输入。这样可以避免上游模块直接传散乱字段，也便于后续状态存储和问题回放。

### 6.1.1 输入对象 `file_intel_input`

| 参数名 | 是否必须 | 参数解释 | 示例 |
| --- | --- | --- | --- |
| task_id | 必须 | 当前任务唯一标识，用于和总控任务、证据链、报告关联 | `task_20260417_0001` |
| file_id | 必须 | 当前文件对象唯一标识，用于去重、引用和结果回写 | `file_000231` |
| source_module | 必须 | 发现该文件的上游模块名称 | `基础发现模块` |
| target | 必须 | 该文件所属目标，如域名、URL、站点或资产标识 | `example.com` |
| source_url | 必须 | 文件原始下载地址 | `https://example.com/.env` |
| local_path | 必须 | 文件下载后在本地的只读路径 | `/data/files/task_20260417_0001/.env` |
| file_name | 必须 | 文件名，用于初步类型判断 | `.env` |
| file_size | 必须 | 文件大小，单位建议为字节 | `2481` |
| file_hash | 必须 | 文件哈希，建议使用 `sha256` | `sha256:abc123...` |
| content_type | 可以缺省 | HTTP 响应头或探测结果中的内容类型 | `text/plain` |
| extension_guess | 可以缺省 | 基于扩展名的文件类型初判 | `env` |
| magic_guess | 可以缺省 | 基于文件头或魔数的文件类型初判 | `text` |
| encoding | 可以缺省 | 文件编码格式 | `utf-8` |
| discovered_at | 必须 | 文件被发现或下载完成的时间 | `2026-04-17T11:20:35+08:00` |
| file_content | 可以缺省 | 文件原文或分块后的文本内容；超大文件可不直接内嵌 | `DB_HOST=127.0.0.1` |
| content_chunks | 可以缺省 | 大文件分块内容，便于模型分段处理 | `["chunk_1","chunk_2"]` |
| parse_policy | 必须 | 解析策略对象，定义是否脱敏、是否允许分块、最大解析长度等 | `{...}` |
| evidence_ref | 必须 | 上游证据引用，便于审计和报告追踪 | `evidence://file/001` |
| parent_event_ref | 可以缺省 | 触发本次文件解析的上游事件引用 | `event://obs/8891` |

### 6.1.2 解析策略对象 `parse_policy`

| 参数名 | 是否必须 | 参数解释 | 示例 |
| --- | --- | --- | --- |
| allow_execute | 必须 | 是否允许执行文件，固定应为 `false` | `false` |
| mask_secrets | 必须 | 输出时是否对密钥、密码、Token 做脱敏 | `true` |
| enable_chunk_parse | 必须 | 是否允许分块解析大文件 | `true` |
| max_inline_content_length | 必须 | 允许直接内嵌到输入中的最大文本长度 | `20000` |
| max_extract_items | 必须 | 单次最多提取的敏感项或发现项数量 | `100` |
| require_structured_output | 必须 | 是否强制返回结构化 JSON | `true` |
| file_type_hint | 可以缺省 | 上游已知的文件类型提示 | `config_file` |
| review_threshold | 可以缺省 | 低置信、高危或损坏时触发复核的阈值策略 | `high_risk_or_low_confidence` |

## 6.2 输出设计

文件情报模块建议统一输出一个 `file_intel_output` 对象。该对象应同时服务于报告模块、状态存储模块、复核模块和前端展示层，因此输出必须同时包含“解析结果”、“风险结论”和“错误状态”。

### 6.2.1 输出对象 `file_intel_output`

| 参数名 | 是否必须 | 参数解释 | 示例 |
| --- | --- | --- | --- |
| task_id | 必须 | 关联的任务 ID | `task_20260417_0001` |
| file_id | 必须 | 关联的文件 ID | `file_000231` |
| parse_started_at | 必须 | 本次解析开始时间 | `2026-04-17T11:21:02+08:00` |
| parse_finished_at | 必须 | 本次解析结束时间 | `2026-04-17T11:21:08+08:00` |
| parser_profile | 必须 | 本次使用的解析器画像或类型模板 | `config_file_v1` |
| prompt_version | 必须 | 本次使用的提示词版本 | `file_intel_prompt_v1` |
| parse_result | 必须 | 核心解析结果对象 | `{...}` |
| source_refs | 必须 | 本次输出关联的证据引用、文件引用、事件引用 | `["evidence://file/001"]` |
| next_actions | 可以缺省 | 建议下游执行的动作，如补抓、复核、报告标记 | `["mark_high_risk","queue_review"]` |

### 6.2.2 核心结果对象 `parse_result`

| 参数名 | 是否必须 | 参数解释 | 示例 |
| --- | --- | --- | --- |
| file_type | 必须 | 最终判定的文件类型 | `config_file` |
| type_confidence | 必须 | 文件类型判断置信度，0 到 1 | `0.97` |
| type_evidence | 必须 | 支撑类型判断的依据 | `["file_name=.env","text key-value format"]` |
| parse_status | 必须 | 解析状态，只能是 `success`、`partial_success`、`failed` | `success` |
| risk_level | 必须 | 风险等级，建议为 `low`、`medium`、`high`、`critical` | `critical` |
| risk_tags | 必须 | 风险标签列表 | `["db_credentials","redis_exposure","config_leak"]` |
| key_findings | 必须 | 结构化核心发现列表 | `[{...}]` |
| secrets | 可以缺省 | 敏感信息列表，默认脱敏输出 | `[{...}]` |
| assets | 可以缺省 | 从文件中提取出的资产对象，如域名、IP、URL、库表、镜像仓库等 | `[{...}]` |
| artifact_refs | 可以缺省 | 解析过程中产生的中间产物或恢复文件引用 | `["artifact://recovered/env_fields"]` |
| followup_paths | 可以缺省 | 建议继续尝试抓取或补解析的路径 | `["/.env.bak","/config.php"]` |
| summary | 必须 | 对本文件解析结果的简短摘要 | `发现数据库和 Redis 凭证，风险极高` |
| need_review | 必须 | 是否需要进入人工复核或复核模块 | `true` |
| review_reason | 可以缺省 | 触发复核的原因 | `发现高危凭证，建议人工确认是否为生产环境` |
| error | 必须 | 错误对象；成功时也需保留空结构 | `{...}` |

### 6.2.3 发现对象 `key_findings[]`

| 参数名 | 是否必须 | 参数解释 | 示例 |
| --- | --- | --- | --- |
| finding_type | 必须 | 发现类型，如 `credential`、`repo_info`、`api_endpoint`、`db_schema` | `credential` |
| title | 必须 | 发现标题 | `数据库连接信息泄露` |
| value | 必须 | 发现值或描述信息 | `mysql://admin:***@127.0.0.1:3306` |
| location | 必须 | 发现所在位置，如行号、字段名、路径 | `line:1-4` |
| confidence | 必须 | 当前发现置信度 | `0.98` |
| risk_reason | 可以缺省 | 该发现为什么危险 | `可直接用于数据库登录` |

### 6.2.4 敏感信息对象 `secrets[]`

| 参数名 | 是否必须 | 参数解释 | 示例 |
| --- | --- | --- | --- |
| secret_type | 必须 | 敏感信息类型 | `db_password` |
| masked_value | 必须 | 脱敏后的值 | `Pa***d!` |
| raw_length | 可以缺省 | 原始值长度 | `9` |
| location | 必须 | 敏感信息位置 | `DB_PASSWORD` |
| related_asset | 可以缺省 | 关联的资产或服务 | `mysql://127.0.0.1:3306` |
| risk_reason | 必须 | 风险解释 | `泄露数据库密码` |

### 6.2.5 资产对象 `assets[]`

| 参数名 | 是否必须 | 参数解释 | 示例 |
| --- | --- | --- | --- |
| asset_type | 必须 | 资产类型，如 `domain`、`ip`、`url`、`db`、`repo` | `db` |
| asset_value | 必须 | 资产值 | `mysql://127.0.0.1:3306` |
| source_field | 可以缺省 | 该资产来自哪个字段或片段 | `DB_HOST + DB_PORT` |
| confidence | 必须 | 资产识别置信度 | `0.96` |

### 6.2.6 错误对象 `error`

| 参数名 | 是否必须 | 参数解释 | 示例 |
| --- | --- | --- | --- |
| has_error | 必须 | 是否发生错误 | `false` |
| error_code | 可以缺省 | 结构化错误码 | `UNSUPPORTED_BINARY_FORMAT` |
| error_message | 可以缺省 | 错误摘要 | `文件为未知二进制格式，无法完整解析` |
| degradated | 可以缺省 | 是否已进入降级模式 | `true` |

## 6.3 落地约束

为保证第六部分长期稳定，建议固定以下约束：

1. 文件情报模块只接收 `file_intel_input`，不直接消费前端原始输入。
2. 输出必须统一为 `file_intel_output`，禁止不同文件类型使用不同协议。
3. `parse_status`、`risk_level`、`need_review` 必须始终有值，不允许省略。
4. 高危敏感信息默认脱敏，不允许在普通输出中直接回显完整明文。
5. 即使解析失败，也必须返回结构化 `error` 和最小可用的 `parse_result`。
6. `task_id`、`file_id`、`evidence_ref` 必须可回溯，便于总控agent、报告模块和审计模块串联。
