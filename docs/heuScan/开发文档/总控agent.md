# 一、agrnt任务描述

    作为项目的核心总控agent，需要承担对接用户输入输出，拆解任务，处理错误等核心功能，并且具体调用那一部分agent，也需要该总控agent控制；因此该agent处于绝对的核心。

    使用PLAN范式设计总控agent：需要在开始任务之前，明确用户目标，以及信息收集的边界；同时明确用户信息收集的目的（例如，需求仅为收集企业信息等，则不需要启动诸如JS解析等agent，节省Token）；
    使用框架：
    引入状态机：在plan确定后，需要生成状态管理（分布任务），用于管理整个项目的生命周期。

# 二、具体功能拆解

## 1.用户输入与任务建模

### 1.1 输入接收与标准化

    总控agent首先只接收统一输入，不直接把前端原始表单透传给下游agent。这里要先完成输入标准化，生成统一的TaskCommand。标准化内容至少包括：企业名称标准化、域名去协议与去路径、IP/CIDR格式校验、URL拆分、批量目标去重、别名归并、用户意图标签化（企业情报/资产发现/应用分析/图谱聚合/价值评估/报告输出）。

    工程上建议把输入拆成三层：原始输入raw_input、标准化输入normalized_input、任务契约task_contract。raw_input只做留痕；normalized_input作为后续状态机和路由的唯一输入；task_contract则补充范围、约束、优先级、预算、风险等级，成为整个任务生命周期的固定锚点。

### 1.2 澄清机制与默认策略

    总控agent不是对所有不确定项都追问，而是只对会影响执行边界的问题做澄清。建议将字段分成两类：
    1. 必须澄清项：目标主体不明确、扫描对象类型不明确、收集边界不明确、输出目标不明确；
    2. 可默认项：并发等级、输出风格、非关键模块开关、报告详细程度。

    如果命中必须澄清项，总控agent应中断plan生成，返回结构化澄清问题；如果仅缺少可默认项，则写入默认策略继续执行。默认策略建议固化为profile，如enterprise_only、surface_scan、deep_app_scan、graph_first、report_first，避免每次由LLM自由发挥。

### 1.3 任务契约生成

    总控agent需要在任务开始前生成可持久化任务契约，至少包含：task_id、目标集合、目标类型、执行目的、启用agent列表、禁用agent列表、最大并发、最大token预算、证据保留策略、允许的状态跳转、人工复核阈值、最终输出格式。后续所有agent都只能在该契约范围内执行，不能自行扩大范围。

## 2.返回输出与结果聚合

### 2.1 输出不是直接返回报告文本

    总控agent返回前端的内容不能只是报告生成agent的原始文本，而应统一经Result Aggregator组装成TaskSnapshot。TaskSnapshot至少包含：全局状态、当前阶段、阶段摘要、关键发现、图谱摘要、价值排序、待复核项、最终报告引用、证据引用、错误与告警列表。

### 2.2 面向前端的输出分层

    返回前端建议分三层：
    1. 实时层：任务进度、当前运行agent、最近事件、失败重试信息；
    2. 分析层：发现列表、置信度、证据片段、关联实体、图谱快照；
    3. 产物层：报告正文、报告块结构、导出链接、审计回放入口。

    前端不直接消费底层ObservationEvent，而是消费总控agent输出的聚合视图，避免前端自己拼状态导致不一致。

### 2.3 渲染安全与可回放

    返回输出时要同时保存render_payload和source_refs。render_payload用于前端渲染，source_refs用于回放和审计；任意前端展示块都必须能追溯到证据引用或上游事件。这样既能保证报告展示稳定，也能支持后续“某结论为何出现”的排查。

## 3.任务分析拆解与能力路由

### 3.1 规划阶段职责

    总控agent在plan阶段要把任务转换成可执行的阶段图，而不是简单列出agent名单。建议固定映射到项目文档中的状态机阶段：界定范围 → 发现 → 分析 → 扩展 → 关联 → 确定优先级 → 报告 → 评审。

### 3.2 能力路由规则

    总控agent应维护一份显式的能力路由表，而不是靠prompt临场判断。例如：
    1. 企业主体、股东、备案、历史变更类需求，路由到Enterprise Intel Agent；
    2. 子域、IP、端口、路径发现类需求，路由到Asset Discovery Agent；
    3. 页面技术栈与中间件识别，路由到Fingerprint Agent；
    4. JS接口、参数、隐藏路径、外联域名提取，路由到JS Semantic Agent；
    5. 图关系拼装与拓扑构建，路由到Graph & Topology Agent；
    6. 资产优先级、价值排序，路由到Value Assessment Agent；
    7. 最终对外可读产物，路由到Report Agent；
    8. 高风险、低置信或冲突结论，升级到Review Agent。

### 3.3 跳过、补跑与动态加派

    总控agent必须支持三类调度决策：
    1. 跳过：如任务仅为企业情报收集，则不启动JS Semantic Agent和Passive Traffic Audit Agent；
    2. 补跑：如资产发现阶段发现大量JS资源，再补跑JS Semantic Agent；
    3. 加派：如JS解析发现新域名、接口或文件线索，则触发Dynamic Dictionary Agent、File Intel Agent或Chain Hint Agent。

    这部分要基于显式规则和事件触发条件实现，不能完全依赖LLM自由决定，否则很容易造成链路不可控和token浪费。

### 3.4 依赖图与执行顺序

    总控agent内部建议维护DAG而不是单一串行列表。典型依赖关系如下：Enterprise Intel与Asset Discovery可并行；Fingerprint依赖基础HTTP结果；JS Semantic依赖页面/JS采集结果；Graph & Topology依赖实体与发现沉淀；Value Assessment依赖图谱和发现；Report依赖前述摘要；Review插在高风险分支和最终产出前。这样既能提高吞吐，也便于失败隔离。

## 4.状态机与状态存储

### 4.1 全局状态存储模型

    总控agent不能只把“当前状态”写进一张表，而要同时保存当前态和状态流转历史。建议至少拆分以下对象：
    1. tasks：任务主表，保存task_contract、当前状态、当前阶段；
    2. task_state_transitions：状态跳转流水，记录from_state、to_state、trigger_event、operator；
    3. agent_runs：每个agent执行实例，记录开始时间、结束时间、输入摘要、输出摘要、token消耗、失败原因；
    4. observation_events：标准事件表，作为状态推进和回放依据；
    5. task_snapshots：前端查询快照，避免每次现算；
    6. context_capsules：阶段摘要与上下文压缩结果；
    7. review_tickets：复核工单；
    8. evidence_refs：证据对象索引。

### 4.2 存储职责划分

    PostgreSQL适合保存任务、事件、状态流转、报告元数据；Redis适合保存队列、短期锁、去重键、心跳、短时上下文缓存；MinIO适合保存HTML、JS、流量包、文件、报告原文；Neo4j适合保存实体图与拓扑图。总控agent只通过统一repository写入，不允许各agent自己决定往哪存、怎么存。

### 4.3 状态一致性策略

    当前状态建议采用“事件驱动更新 + 快照查询”模式：状态流转以append-only事件为准，tasks.current_state和task_snapshots仅作为查询加速层。发生异常时，可以根据task_state_transitions和observation_events重建最新状态，避免因为缓存或单次写入失败导致任务不可恢复。

## 5.多agent异步调用、调度与并发控制

### 5.1 异步调度基本模型

    总控agent不直接同步串行调用所有agent，而是生成AgentTask并交给AgentExecutor。每个AgentTask至少包含：task_id、stage、agent_type、input_ref、context_ref、priority、retry_policy、deadline、parent_event_id。执行结果统一回写ObservationEvent，再由总控agent消费事件推进状态机。

### 5.2 任务领取与租约机制

    为防止重复执行，建议对每个AgentTask使用claim lease机制：执行器领取任务时写入lease_owner、lease_expire_at，并定期心跳续租；超时未续租则回到待执行队列。这样即使worker异常退出，也不会出现任务永久卡死。

### 5.3 并发控制

    并发至少分三层控制：
    1. 任务级并发：单个task允许同时运行多少个agent；
    2. agent级并发：某类agent的全局并发上限，避免高成本模块拖垮系统；
    3. 目标级并发：同一域名、同一IP或同一证据对象的并发限制，防止重复抓取和热点拥塞。

    默认建议为：企业情报模块低并发、基础发现中并发、JS解析与文件抽取低并发、图谱与报告单任务单实例。具体数值不写死在prompt，而写在runtime配置中。

### 5.4 去重、幂等与防循环

    多agent协同时最容易出现事件风暴和重复调用，所以总控agent必须内置幂等控制。建议每个ObservationEvent都生成dedupe_key，构成方式可为task_id + agent_type + normalized_target + event_type + content_hash。若同一个dedupe_key在有效窗口内已被消费，则只更新计数，不重复触发下游。

    对于容易形成闭环的链路，如“JS解析发现新路径 -> 扩展词典生成新词 -> 再次基础发现 -> 再次触发JS解析”，必须增加最大展开深度、最小增益阈值和阶段循环次数上限。若新增信息增益低于阈值，应停止扩展并进入下一阶段。

### 5.5 暂停、恢复与人工介入

    总控agent要支持任务级pause/resume/retry。pause时不再下发新任务，但保留已运行任务结果；resume时从最近一个稳定状态重新调度，不从头执行；retry分为agent级retry和阶段级retry，避免因为单个子任务失败导致整任务完全重跑。

## 6.上下文装配与上下文剪裁

### 6.1 上下文不是“把所有历史拼进去”

    总控agent的上下文装配必须先分层再裁剪。建议固定为五层：
    1. 不可裁剪层：task_contract、范围约束、当前阶段目标、明确禁用项；
    2. 核心事实层：已确认实体、关键资产、已验证结论、Review结论；
    3. 当前工作层：本阶段待解决问题、最近上游摘要、直接依赖事件；
    4. 证据片段层：与当前agent最相关的HTML/JS/文件/流量/图谱片段；
    5. 冗余历史层：旧日志、低置信发现、重复片段、细粒度执行日志。

    裁剪顺序必须从第5层往上删，前3层原则上只允许压缩，不允许直接删除。

### 6.2 token预算计算标准

    上下文裁剪必须以模型窗口为硬约束。建议总控agent维护model_registry，记录不同模型的最大上下文窗口W。每次组装上下文时按以下方式预留预算：
    1. 输出预算reserved_output = max(预估输出token, W * 15%)；
    2. 系统提示与工具协议预算reserved_system = W * 10%；
    3. 安全缓冲reserved_safety = W * 10%；
    4. 可用上下文budget_context = W - reserved_output - reserved_system - reserved_safety。

    若budget_context为负，则必须降低任务粒度、拆分任务或切换更大窗口模型，不能强塞。工程上建议把实际注入上下文控制在W的65%~70%，75%作为硬上限，超过即触发剪裁或拆包。

### 6.3 裁剪评分标准

    对证据与历史事件不应按时间粗暴截断，而应按综合分排序后保留。建议评分维度如下：
    score = 相关性 * 0.35 + 置信度 * 0.25 + 新鲜度 * 0.15 + 依赖度 * 0.15 + 多样性收益 * 0.10。

    其中：
    1. 相关性：与当前agent目标是否直接相关；
    2. 置信度：上游agent或规则打分是否可靠；
    3. 新鲜度：是否来自最近阶段、是否已被后续结论覆盖；
    4. 依赖度：是否是当前推理链的必要前置；
    5. 多样性收益：是否提供新的实体、接口、路径或异常模式。

    重复内容要先做归并再打分，避免同类HTML片段、相似JS函数或重复事件占满token。

### 6.4 不同内容类型的分块标准

    为保证上下文连续性，裁剪之前要先按内容类型做结构化分块：
    1. HTML：按DOM区块、script块、form块或接口相关区域分块，每块建议400~800 token，重叠80~120 token；
    2. JS：按文件模块、函数、对象或AST节点簇分块，每块建议300~600 token，重叠50~100 token；
    3. 文本文件：按标题/段落分块，每块建议500~1000 token；
    4. 流量日志：按会话或请求链分块，保持request/response成对；
    5. 事件历史：按阶段和实体聚合，不按单条日志逐条注入。

    这样可以避免“从中间截断一个函数”或“把同一请求链拆开”导致语义断裂。

### 6.5 保证上下文连续性的机制

    为了在裁剪后仍保证连续性，建议总控agent在每个阶段结束后生成Context Capsule，作为下一阶段的稳定输入锚点。一个Context Capsule至少包含：
    1. 本阶段200~500 token摘要；
    2. 已确认事实列表；
    3. 待验证假设列表；
    4. 关键实体ID与证据ID；
    5. 下阶段建议动作；
    6. 已排除路线与禁止重复工作项。

    下游agent优先读取capsule，再按需附加少量原始证据片段。这样即便原始历史非常长，也能保持语义连续，而不是完全依赖长上下文记忆。

### 6.6 超预算时的降级策略

    当上下文仍然超预算时，建议按以下顺序降级：
    1. 删除低分冗余事件；
    2. 用阶段摘要替代原始执行日志；
    3. 将长证据替换为“摘要 + 关键片段 + 对象引用”；
    4. 将任务拆分成更小的AgentTask；
    5. 必要时切换到更大窗口模型。

    禁止的做法是：直接截断最后N个token后发送给模型，因为这会造成证据断裂、JSON结构损坏和上下游推理不一致。

## 7.复核、质量控制与人工介入

### 7.1 自动复核触发条件

    总控agent应自动将以下情况升级到Review Agent：高价值资产判定但证据不足、多个agent对同一实体给出冲突结论、链路提示带有较强推断性质、报告中出现高风险表述、同一发现置信度波动过大。

### 7.2 输出前质量闸门

    在进入REPORTING和REVIEWING前，总控agent应做统一质量检查：字段完整性校验、证据可追溯校验、置信度下限校验、重复发现归并校验、图谱节点孤立度检查、报告块渲染校验。任何一项失败都不能直接进入最终输出，而应回退到对应阶段补算或发起复核。

### 7.3 人工介入点设计

    即使一期不做复杂人工工作台，也要预留人工介入点，包括：手动确认任务范围、手动忽略误报、手动确认图谱合并、手动触发补跑、手动批准最终报告。这些介入动作也要写入状态流转和审计日志。

## 8.错误处理与降级恢复

### 8.1 错误分类

    总控agent建议把错误分为五类：
    1. 输入错误：参数不合法、目标不可解析、范围不清；
    2. 工具错误：扫描器超时、外部API失败、解析器异常；
    3. agent错误：输出格式不合法、上下文不满足、推理失败；
    4. 状态错误：非法状态跳转、重复消费、事件顺序错乱；
    5. 数据错误：写库失败、对象丢失、快照构建失败。

### 8.2 重试与补偿策略

    不同错误不能统一重试。建议：工具类错误采用指数退避重试；agent格式错误先进入自检修复一次，再失败则升级Review或标记FAILED；状态错误禁止盲目重试，应进入隔离队列；数据写入错误要优先保证事件不丢，可先落本地补偿日志或补偿表，随后异步回放。

### 8.3 降级策略

    当某一agent不可用时，总控agent应支持降级而不是整任务直接失败。例如：JS Semantic Agent不可用时，仍可继续资产发现、指纹识别、图谱构建和基础报告，但在TaskSnapshot中标记“应用层解析缺失”；Value Assessment不可用时，可先返回未排序发现列表并标记待补评估。

### 8.4 部分完成与隔离

    如果非核心分支失败且不影响主链路，可进入PARTIAL_DONE；如果发现异常循环、严重冲突、可疑数据污染，则进入QUARANTINED，停止自动扩散，由Review或人工确认后再恢复。这样可以保证系统在复杂任务下仍具备可控失败能力。

## 9.观测性、安全边界与工程治理补充

### 9.1 可观测性

    总控agent至少要输出四类观测数据：状态流转日志、agent执行指标、工具调用记录、token与成本统计。核心指标包括：任务总耗时、各阶段耗时、各agent成功率、重试次数、上下文大小、平均token消耗、快照构建延迟、队列积压量。

### 9.2 审计与回放

    审计不是最终报告的附属功能，而是总控agent的基础职责。每个最终结论都要能回溯到ObservationEvent、EvidenceRef、AgentRun和StateTransition；每次重试、跳过、补跑、人工干预都必须留痕。这样后续做问题追查、质量评估和模型调优时才有数据基础。

### 9.3 安全边界

    总控agent要严格受task_contract约束，只进行授权范围内的信息收集与分析。工程上应显式校验目标范围、禁用越权扩展、限制高风险工具调用，并对外部对象下载、文件抽取和流量样本处理设置大小上限与类型白名单，防止任务被异常输入拖垮。

# 四、提示词设计

## 1.用户输入标准化提示词

    本提示词用于总控agent接收用户输入后的第一步处理，只负责“理解、清洗、归一、补齐默认值、识别缺失项”，不负责执行扫描、不负责生成报告、不负责扩大收集范围。其目标是把自然语言输入、表单输入或批量目标输入，转换成后续状态机与能力路由可直接消费的标准化任务输入。

    建议提示词正文如下：

```text
你是“总控agent”的用户输入标准化单元。你的唯一职责是把用户原始输入转换为后续可执行的标准化任务输入。你不能执行信息收集，不能调用任何下游模块，不能补充用户未授权的目标，不能把模糊猜测写成确定事实。

你的处理目标如下：
1. 识别用户想分析的主体、目标、范围、目的和输出偏好。
2. 将混乱输入清洗为统一结构，便于后续任务拆解与状态机推进。
3. 提前识别会影响执行边界的缺失信息，并区分“必须反问”和“可以默认”。
4. 为后续任务规划输出稳定、可落库、可审计的标准化结果。

你必须遵守以下规则：
1. 只处理用户已经给出的信息，不得自行扩展目标范围。
2. 企业名称、域名、网址、地址段、单个地址必须分别标准化，不得混用。
3. 多个目标输入时必须去重，并保留用户原始表达与标准化结果之间的对应关系。
4. 对明显无效、冲突或不完整的输入要标记原因，但不要私自修正为新目标。
5. 只有会影响执行边界的问题才允许标记为“必须反问”。
6. 并发等级、输出风格、报告详细度、非关键模块开关等信息，若缺失可写入默认策略，不要升级为必须反问。
7. 所有意图标签必须从以下范围中选择：企业情报、资产发现、应用分析、图谱聚合、价值评估、报告输出。
8. 如用户目的仅为企业情报整理，应明确标记“不建议启用脚本解析、流量审查、文件提取等高成本模块”。

你需要重点完成以下标准化动作：
1. 企业名称标准化：清理括号说明、地区修饰、冗余描述，保留主体全称与别名。
2. 域名标准化：去掉协议、端口、路径、查询参数，仅保留主机名，并识别是否为根域、子域或无效域名。
3. 网址标准化：拆分为协议、主机、端口、路径、查询参数，并提取可继续用于路由的主目标。
4. 单个地址与地址段标准化：校验格式是否合法，区分单点目标与网段目标。
5. 批量目标标准化：去重、归类、标记来源顺序，避免同一目标重复进入后续调度。
6. 用户目标理解：识别用户是要做企业背景梳理、外围资产盘点、应用层线索提取、图谱整合、价值排序，还是报告输出。
7. 边界识别：识别哪些信息缺失会直接阻塞执行，哪些只影响体验但不阻塞执行。

当你完成处理后，只允许输出一个结构化结果，字段必须完整，不能输出解释性散文。输出结构如下：
{
  "原始输入摘要": {
    "输入方式": "",
    "用户原文概括": "",
    "是否为批量输入": ""
  },
  "标准化结果": {
    "主体名称": "",
    "主体别名": [],
    "目标列表": [
      {
        "原始值": "",
        "标准化值": "",
        "目标类别": "",
        "是否有效": "",
        "说明": ""
      }
    ],
    "目标类型汇总": [],
    "用户意图标签": [],
    "期望输出": "",
    "范围说明": ""
  },
  "缺失信息判断": {
    "是否存在必须反问项": "",
    "必须反问项": [
      {
        "字段": "",
        "原因": "",
        "建议提问": ""
      }

    "可默认项": [
      {

补充要求如下：
1. 若用户输入同时包含企业名和域名，要保留二者，不要只保留其中之一。
2. 若用户输入出现“帮我查一下”“看看这个公司”“分析这个站”这类口语表达，要提炼出可执行目标，但不得擅自增加未出现的资产。
3. 若用户输入只有一句非常模糊的话，且无法判断目标主体或执行目的，必须输出必须反问项。
4. 若用户输入已经足够清晰，则“必须反问项”应为空数组，并给出默认策略。
5. 输出中所有判断必须可追溯到用户输入，不允许出现无依据推断。
```

## 2.用户输入任务拆解与反问提示词

    本提示词用于接收“标准化结果”后生成任务规划、能力路由和反问结论。该提示词的重点不是自由发挥，而是基于既定状态机、能力分工和边界约束，把任务转换成可执行阶段图；只有在确实会影响执行边界时，才允许向用户发起反问。

    建议提示词正文如下：

```text
你是“总控agent”的任务拆解与边界确认单元。你的职责是在已经完成输入标准化的前提下，决定任务是否可以直接进入执行规划；如果可以，则输出阶段计划、能力路由、默认策略和任务契约草案；如果不可以，则输出最小化、结构化、可直接返回前端的反问结果。

你的工作原则如下：
1. 先判断边界，再决定任务路径，最后才生成阶段计划。
2. 反问必须克制，只有影响执行边界的问题才允许提出。
3. 不能因为信息不完整就默认启用所有模块，必须按任务目标做最小必要规划。
4. 不能把下游模块当作兜底垃圾桶，任务拆解必须在总控层完成。
5. 不能生成超出用户授权范围的目标、路径或扩展动作。
6. 若仅缺少非关键配置，直接采用默认策略继续规划，不要中断执行。

你必须按照以下步骤思考并输出：
第一步，判断任务是否具备可执行前提：
1. 目标主体是否明确；
2. 目标类型是否明确；
3. 收集边界是否明确；
4. 输出目标是否明确到足以规划；
5. 是否存在明显冲突输入。

第二步，选择执行画像。执行画像只能从以下几类中择一或组合，但必须给出理由：
1. 企业情报优先；
2. 外围资产优先；
3. 深度应用分析优先；
4. 图谱聚合优先；
5. 报告输出优先。

第三步，生成阶段计划。阶段名称必须从以下固定集合中选择，不得擅自发明新阶段：
1. 界定范围；
2. 发现；
3. 分析；
4. 扩展；
5. 关联；
6. 优先级判定；
7. 报告；
8. 评审。

第四步，生成能力路由。能力路由必须遵守以下映射：
1. 企业主体、股东、备案、历史变更、关系梳理，路由到企业情报模块；
2. 子域、地址、端口、路径发现，路由到基础发现模块；
3. 页面技术栈、中间件、组件识别，路由到指纹识别模块；
4. 脚本接口、参数、隐藏路径、外联域名提取，路由到脚本语义解析模块；
5. 企业别名、技术栈、路径词扩展，路由到扩展词典模块；
6. 实体合并、资产关系、拓扑构建，路由到图谱聚合模块；
7. 价值分级、优先级排序，路由到价值评估模块；
8. 最终可读产物生成，路由到报告输出模块；
9. 高风险、低置信、冲突结论，路由到复核模块；
10. 流量样本中的线索识别，路由到流量审查模块；
11. 下载文件与文本抽取，路由到文件情报模块；
12. 基于图谱事实生成推断性链路提示，路由到链路提示模块。

4. 高风险结论、冲突结论、低置信结论，必须进入复核模块，不得直接写入最终报告。

第六步，决定是否需要反问。只有以下情况允许反问：
1. 目标主体不明确；
2. 目标类型冲突，无法判断是企业、域名、网址还是地址；
3. 收集边界不明确，可能导致越权扩展；

1. 一次最多提出三个问题；
2. 每个问题必须说明“为什么必须问”；
3. 问题必须短、准、可直接作答；
4. 不得提出“你还想要什么”“还有别的要求吗”这类开放式问题；
```

# 五、输入表转化与边界鉴定

## 1.设计目标

    本部分用于定义总控agent在接收到用户输入后，如何将输入由“原始表单/自然语言”稳定转化为“可执行、可审计、可路由”的结构化输入，并在进入任务契约生成前完成一次明确的边界鉴定。

    为避免前端字段、LLM理解结果和后端执行对象之间出现歧义，输入转化固定采用三层表设计：
    1. 原始输入层：只保留用户提交内容与前端上下文，负责留痕，不参与直接调度；
    2. 标准化输入层：负责清洗、归一、去重、分类，是后续路由与状态机的唯一输入；
    3. 边界鉴定层：负责判断任务是否越界、哪些参数必须明确、哪些参数可以模糊默认。

## 2.三层输入表设计

### 2.1 第一层：原始输入表 `raw_input`

    该层只做留痕，不做推理，不做裁决，不作为下游agent直接输入。其目标是完整记录“用户当时说了什么、前端提交了什么、系统收到的原文是什么”，便于后续审计与回放。

| 参数名           | 是否必须 | 是否允许模糊 | 参数解释                                     | 实例                                                               |
| ---------------- | -------- | ------------ | -------------------------------------------- | ------------------------------------------------------------------ |
| request_id       | 必须     | 否           | 本次前端请求唯一标识，用于链路追踪           | `req_20260408_0001`                                              |
| submitter_id     | 必须     | 否           | 提交用户或系统账号标识                       | `user_admin`                                                     |
| submit_source    | 必须     | 否           | 输入来源，如页面表单、批量导入、接口调用     | `web_form`                                                       |
| raw_text         | 可以为空 | 是           | 用户自然语言原文，若来自表单可为空           | `帮我分析一下华星科技和它的官网`                                 |
| raw_form_payload | 可以为空 | 是           | 前端表单原始结构，未经标准化                 | `{"company_name":"华星科技","domain":"www.hxtech.com"}`          |
| raw_target_list  | 可以为空 | 是           | 用户直接提交的目标列表，保留原顺序           | `["www.hxtech.com","hxtech.com","https://www.hxtech.com/login"]` |
| raw_intent_hint  | 可以为空 | 是           | 前端显式选择的任务意图，如资产发现、企业情报 | `资产发现`                                                       |
| raw_output_hint  | 可以为空 | 是           | 用户对输出形式的口头要求或前端选项           | `生成简要报告`                                                   |
| attachments_refs | 可以为空 | 是           | 附件、截图、补充文件引用                     | `["minio://upload/task1/brief.pdf"]`                             |
| submitted_at     | 必须     | 否           | 提交时间戳                                   | `2026-04-08T10:15:32+08:00`                                      |

### 2.2 第二层：标准化输入表 `normalized_input`

    该层是总控agent后续plan、能力路由、状态机推进的唯一输入层。所有企业名、域名、网址、地址、意图、输出偏好，都必须在这一层完成归一化；后续模块不得再回头直接读取`raw_input` 进行自由解释。

| 参数名                  | 是否必须 | 是否允许模糊 | 参数解释                                                                     | 实例                                                             |
| ----------------------- | -------- | ------------ | ---------------------------------------------------------------------------- | ---------------------------------------------------------------- |
| normalized_subject      | 必须     | 条件允许     | 标准化后的目标主体名称；若任务不以企业为中心可为空，但必须由目标集合补足主体 | `华星科技有限公司`                                             |
| subject_aliases         | 可以缺省 | 是           | 主体别名、简称、历史名、英文名等                                             | `["华星科技","HXTech"]`                                        |
| normalized_targets      | 必须     | 否           | 标准化后的目标集合，每个目标必须带类型、有效性、来源映射                     | 见下方示例                                                       |
| target_types            | 必须     | 否           | 本次任务包含的目标类型汇总，如企业、域名、URL、IP、CIDR                      | `["企业","域名","URL"]`                                        |
| primary_target          | 必须     | 否           | 后续优先分析的主目标，只允许一个                                             | `hxtech.com`                                                   |
| normalized_intent_tags  | 必须     | 否           | 标准化后的任务意图标签，只能从既定标签集选取                                 | `["企业情报","资产发现","报告输出"]`                           |
| expected_output_types   | 必须     | 是           | 期望产物类型，可为快照、图谱、报告等                                         | `["任务快照","简要报告"]`                                      |
| execution_profile       | 必须     | 是           | 默认执行画像，用于决定模块启停                                               | `surface_scan`                                                 |
| explicit_constraints    | 可以缺省 | 是           | 用户明确提出的限制条件                                                       | `["仅限官网和相关子域","不做深度JS解析"]`                      |
| normalization_notes     | 可以缺省 | 是           | 标准化过程中产生的备注，如去重、冲突、疑似无效项                             | `["www.hxtech.com 与 hxtech.com 归并","login URL 提取出主域"]` |
| missing_required_fields | 必须     | 否           | 当前仍缺失的必须字段列表，若为空表示可继续进入边界鉴定                       | `[]`                                                           |

    其中`normalized_targets` 的子结构建议固定如下：

| 子参数名         | 是否必须 | 是否允许模糊 | 参数解释                               | 实例                             |
| ---------------- | -------- | ------------ | -------------------------------------- | -------------------------------- |
| raw_value        | 必须     | 否           | 用户提交的原始目标值                   | `https://www.hxtech.com/login` |
| normalized_value | 必须     | 否           | 标准化后的目标值                       | `www.hxtech.com`               |
| target_type      | 必须     | 否           | 目标类型                               | `域名`                         |
| target_role      | 必须     | 是           | 目标角色，如主目标、补充目标、候选目标 | `主目标`                       |
| is_valid         | 必须     | 否           | 格式是否合法                           | `true`                         |
| canonical_group  | 可以缺省 | 是           | 归并组标识，用于目标去重               | `domain:hxtech.com`            |
| derived_from     | 可以缺省 | 是           | 由哪个原始值拆解或归并而来             | `https://www.hxtech.com/login` |
| note             | 可以缺省 | 是           | 对该目标的额外说明                     | `URL 去路径后归一为域名`       |

### 2.3 第三层：边界鉴定表 `boundary_input`

    该层用于回答“能不能做、做到哪里、哪些条件必须先明确”这三个问题。它不是任务契约本身，但它会直接决定后续是否允许生成任务契约。

| 参数名                   | 是否必须 | 是否允许模糊 | 参数解释                                                  | 实例                                        |
| ------------------------ | -------- | ------------ | --------------------------------------------------------- | ------------------------------------------- |
| scope_subject            | 必须     | 条件允许     | 本次任务边界绑定的主体，可以是企业，也可以是域名/地址集合 | `华星科技有限公司`                        |
| allowed_target_scope     | 必须     | 否           | 允许进入后续执行的目标范围白名单                          | `["hxtech.com","*.hxtech.com"]`           |
| denied_target_scope      | 可以缺省 | 是           | 明确禁止扩展的范围                                        | `["合作方域名","第三方登录回调域名"]`     |
| task_goal                | 必须     | 否           | 用户授权的任务目标                                        | `企业情报梳理 + 外围资产盘点`             |
| data_collection_boundary | 必须     | 否           | 允许的信息收集边界，防止模块自由蔓延                      | `仅收集公开可访问的企业与外围资产信息`    |
| output_boundary          | 必须     | 是           | 最终允许返回的结果形态                                    | `任务快照 + 简要报告，不输出深度推断链路` |
| must_clarify_items       | 必须     | 否           | 必须反问的字段及原因列表                                  | `[]`                                      |
| defaultable_items        | 必须     | 否           | 允许走默认值的字段列表                                    | `["并发等级","报告详细度"]`               |
| forbidden_modules        | 可以缺省 | 是           | 在当前边界下禁止启动的模块                                | `["流量审查模块","文件情报模块"]`         |
| conditional_modules      | 可以缺省 | 是           | 仅在满足事件条件后才能补跑的模块                          | `["脚本解析模块","扩展词典模块"]`         |
| boundary_confidence      | 必须     | 否           | 当前边界判断的置信等级，高/中/低                          | `高`                                      |
| boundary_notes           | 可以缺省 | 是           | 边界判定备注                                              | `用户仅要求外围盘点，未授权深度应用分析`  |

## 4.边界参数设计

### 4.1 边界参数总表

    边界参数用于明确总控agent“可以做什么”“不能做什么”“不确定时是否必须停下来反问”。以下参数必须由边界鉴定层统一给出，不能分散在多个模块中各自解释。

| 边界参数     | 是否必须 | 是否允许模糊 | 作用说明                                     | 模糊处理规则                                                         | 示例                                     |
| ------------ | -------- | ------------ | -------------------------------------------- | -------------------------------------------------------------------- | ---------------------------------------- |
| 主体边界     | 必须     | 条件允许     | 明确本次任务围绕谁展开                       | 若企业名不明，但主域明确，可先以主域为边界；若二者都不明，则必须反问 | `华星科技有限公司` 或 `hxtech.com`   |
| 目标边界     | 必须     | 否           | 明确哪些域名、地址、URL、主体允许进入执行    | 不允许模糊；没有明确目标边界不得进入执行                             | `hxtech.com`、`*.hxtech.com`         |
| 任务目标边界 | 必须     | 否           | 明确做企业情报、资产发现、应用分析中的哪一类 | 不允许“都做一点”式空泛描述                                         | `企业情报 + 外围资产盘点`              |
| 收集边界     | 必须     | 否           | 明确允许收集到什么层级                       | 若未说明是否允许深度应用分析，应默认不允许                           | `仅限公开信息与外围资产`               |
| 输出边界     | 必须     | 是           | 明确输出快照、图谱、报告中的哪些内容         | 若未说明详细程度，可默认简版                                         | `任务快照 + 简要报告`                  |
| 模块启用边界 | 必须     | 是           | 明确哪些模块直接允许，哪些禁止，哪些条件触发 | 未明确授权的高成本模块默认关闭                                       | `禁止流量审查，脚本解析按条件触发`     |
| 时间边界     | 可以缺省 | 是           | 是否限定时间范围或时效要求                   | 若未说明，可不作为阻塞项                                             | `近一年企业变更`                       |
| 成本边界     | 可以缺省 | 是           | token、并发、时长等预算控制                  | 若未说明，采用系统默认画像                                           | `低成本模式`                           |
| 证据边界     | 可以缺省 | 是           | 允许保留哪些证据对象                         | 若未说明，采用标准保留策略                                           | `保留页面与报告引用，不保留大体积附件` |

### 4.2 必须明确与允许模糊的判定标准

    必须明确的参数，是指“缺失后会直接影响执行边界、可能导致越权、错误路由或错误报告”的参数；允许模糊的参数，是指“缺失后不会改变任务边界，只影响执行风格、资源消耗或呈现颗粒度”的参数。

    必须明确参数如下：
    1. 主体边界或主目标至少一项明确；
    2. 目标边界明确；
    3. 任务目标边界明确；
    4. 收集边界明确；
    5. 至少有一种可接受输出形式明确。

    可以模糊参数如下：
    1. 报告详细度；
    2. 并发等级；
    3. 是否输出图谱文件；
    4. 价值评估结果的展示颗粒度；
    5. 非关键模块是否预热。

## 5.边界注释规则

### 5.1 注释一：边界参数示例

    注释一：当用户输入“帮我看看华星科技和官网，先整理企业背景和外围资产，给我一个简要报告”时，可判定如下：
    1. 主体边界为“华星科技有限公司”，若企业全称未完全确认但官网域名明确，可暂以`hxtech.com` 作为主目标补足；
    2. 目标边界为 `hxtech.com` 及其子域，不自动扩展到第三方外联域名；
    3. 任务目标边界为“企业情报 + 外围资产盘点”，不等同于“深度应用分析”；
    4. 收集边界为“公开可访问的信息与外围资产”，因此脚本深度解析、流量审查、文件提取均不默认开启；
    5. 输出边界为“任务快照 + 简要报告”，报告详细程度可以模糊默认，但“是否输出报告”本身不能模糊。

### 5.2 反问触发规则

    下列情况必须触发反问，不得直接进入任务契约生成：
    1. 用户只说“帮我看看这个”，没有主体、域名、地址中的任何一个；
    2. 用户输入同时出现多个无明显关系的主体，无法判断本次任务主对象；
    3. 用户要求“分析应用”“看看接口”，但没有给出任何网站、域名、URL、地址或企业主体；
    4. 用户要求“全面分析”，但未说明是否允许深度应用层分析，且当前系统无法安全默认。

    下列情况不需要反问，可走默认值：
    1. 未说明报告详细版还是简版；
    2. 未说明是否导出图谱文件；
    3. 未说明并发等级或时长预算；
    4. 未说明是否展示中间过程日志。

## 6.落地约束

    为保证输入表转化稳定，工程上应固定以下约束：
    1. 所有下游模块只能读取`normalized_input` 与 `boundary_input`，不得直接消费 `raw_input`；
    2. 边界鉴定必须在任务契约生成前完成；
    3. 任一“必须明确参数”缺失时，状态机只能停留在澄清阶段，不得推进到执行阶段；
    4. 任一“允许模糊参数”缺失时，必须写入默认值来源，不能留空不管；
    5. 每一次边界判定都必须留痕，便于后续审计“为什么启动了某个模块，为什么跳过了某个模块”。

# 六、任务契约生成要求与格式设计

## 1.设计目标

    任务契约是总控agent在完成“输入标准化”和“边界鉴定”之后生成的统一执行依据。它的作用不是简单保存几个参数，而是把“这次任务允许做什么、禁止做什么、优先做什么、如何结束、如何复核”固定下来，作为整个任务生命周期的唯一约束锚点。

    任务契约生成后，应同时满足以下要求：
    1. 下游所有agent都只能在契约范围内执行；
    2. 调度器只能依据契约决定启用、跳过、补跑和加派；
    3. 状态机只能在契约允许的阶段和边界内推进；
    4. 审计与回放时，能够明确说明某个模块为什么被启动或禁止；
    5. 前端、后端、运行时三方看到的是同一份任务约束定义。

## 2.生成原则

### 2.1 契约生成时机

    任务契约必须在以下条件全部满足后生成：
    1.`normalized_input（标准化输入）` 已生成；
    2. `boundary_input（边界鉴定输入）` 已完成；
    3. 所有“必须明确参数”已明确；
    4. 不存在未解决的必须反问项。

    若仍存在必须反问项，则不允许进入任务契约生成阶段，只能停留在澄清阶段。

### 2.2 契约生成原则

    任务契约生成建议遵循以下原则：
    1. 最小必要原则：只启用完成当前目标所需的模块，不默认全开；
    2. 边界优先原则：契约必须先体现范围和限制，再体现执行路径；
    3. 可审计原则：所有启用、禁用、默认、降级策略都必须可解释；
    4. 可执行原则：字段要能直接被调度器、状态机、快照构建器消费；
    5. 可扩展原则：新增agent或新增阶段时，不应破坏已有契约结构。

## 3.任务契约总体结构

    建议总控agent输出统一结构的`task_contract（任务契约）`，至少包含以下九个对象：
    1. `contract_meta（契约元信息）`
    2. `task_scope（任务范围）`
    3. `task_objective（任务目标）`
    4. `execution_policy（执行策略）`
    5. `module_policy（模块策略）`
    6. `state_policy（状态策略）`
    7. `evidence_policy（证据策略）`
    8. `review_policy（复核策略）`
    9. `output_policy（输出策略）`

## 4.参数设计

### 4.1 `contract_meta（契约元信息）`

| 参数名           | 是否必须 | 参数注释                                             | 示例                                      |
| ---------------- | -------- | ---------------------------------------------------- | ----------------------------------------- |
| contract_id      | 必须     | 任务契约唯一标识，用于和任务、状态流转、审计记录关联 | `contract_20260408_0001`                |
| task_id          | 必须     | 所属任务唯一标识                                     | `task_20260408_0001`                    |
| contract_version | 必须     | 契约版本号，每次人工修订或重大策略变更递增           | `1`                                     |
| contract_status  | 必须     | 当前契约状态，如有效、待确认、已替换                 | `有效`                                  |
| generated_by     | 必须     | 契约生成来源，一般为总控agent                        | `orchestrator_agent`                    |
| generated_at     | 必须     | 契约生成时间                                         | `2026-04-08T10:22:18+08:00`             |
| based_on_inputs  | 必须     | 契约基于哪些输入对象生成                             | `["normalized_input","boundary_input"]` |

### 4.2 `task_scope（任务范围）`

| 参数名              | 是否必须 | 参数注释                                         | 示例                                                        |
| ------------------- | -------- | ------------------------------------------------ | ----------------------------------------------------------- |
| scope_subject       | 必须     | 本次任务围绕的主体，可以是企业主体，也可以是主域 | `华星科技有限公司`                                        |
| primary_target      | 必须     | 主目标，作为后续调度和展示的核心对象             | `hxtech.com`                                              |
| allowed_targets     | 必须     | 允许进入执行流程的目标白名单                     | `["hxtech.com","*.hxtech.com"]`                           |
| denied_targets      | 可以缺省 | 明确禁止扩展的目标范围                           | `["第三方供应商域名","外部登录回调域名"]`                 |
| target_types        | 必须     | 本次任务覆盖的目标类型集合                       | `["企业","域名"]`                                         |
| collection_boundary | 必须     | 允许采集的信息边界说明                           | `仅收集公开可访问的企业与外围资产信息`                    |
| out_of_scope_notes  | 可以缺省 | 明确写出不在本次任务内的范围                     | `["不进入深度应用层接口语义分析","不处理第三方托管资产"]` |

### 4.3 `task_objective（任务目标）`

| 参数名           | 是否必须 | 参数注释                       | 示例                                          |
| ---------------- | -------- | ------------------------------ | --------------------------------------------- |
| core_goal        | 必须     | 本次任务的核心目标摘要         | `企业情报梳理 + 外围资产盘点`               |
| intent_tags      | 必须     | 标准化后的任务意图标签         | `["企业情报","资产发现"]`                   |
| expected_outputs | 必须     | 期望输出类型                   | `["任务快照","分析层结果"]`                 |
| priority_focus   | 可以缺省 | 本次任务优先关注的重点方向     | `["主体关联域名","高活跃接口子域"]`         |
| non_goals        | 可以缺省 | 明确声明不是本次任务目标的内容 | `["不生成深度推断链路","不做流量内容分析"]` |

### 4.4 `execution_policy（执行策略）`

| 参数名              | 是否必须 | 参数注释                                               | 示例                                             |
| ------------------- | -------- | ------------------------------------------------------ | ------------------------------------------------ |
| execution_profile   | 必须     | 执行画像，决定整体策略，如轻量扫描、表层分析、图谱优先 | `surface_scan`                                 |
| max_concurrency     | 必须     | 当前任务允许的最大并发数                               | `4`                                            |
| token_budget_level  | 必须     | 令牌预算等级，如低、中、高                             | `中`                                           |
| max_runtime_minutes | 可以缺省 | 任务最长运行时间，超时后进入告警或降级                 | `60`                                           |
| fallback_strategy   | 必须     | 模块失败后的降级策略摘要                               | `关键链路失败则重试，非关键链路失败则部分完成` |
| scheduling_mode     | 必须     | 调度模式，如串行优先、并行优先、条件触发               | `条件并行`                                     |

### 4.5 `module_policy（模块策略）`

    该对象用于定义“哪些模块直接启用、哪些禁止、哪些条件触发”，建议拆成三个数组。

| 参数名              | 是否必须 | 参数注释                               | 示例                                               |
| ------------------- | -------- | -------------------------------------- | -------------------------------------------------- |

| disabled_modules    | 必须     | 明确禁止启动的模块列表                 | `["流量审查模块","文件情报模块"]`                |
| conditional_modules | 可以缺省 | 只有满足特定事件条件时才允许补跑的模块 | `["脚本解析模块","扩展词典模块"]`                |
| module_notes        | 可以缺省 | 模块启停的补充说明                     | `["脚本解析仅在发现大量JS资源时启动"]`           |

| 子参数名          | 是否必须 | 参数注释             | 示例                                    |

| note              | 可以缺省 | 补充说明             | `仅对主域及其确认关联域执行`          |

| 参数名                   | 是否必须 | 参数注释                 | 示例                                               |
| ------------------------ | -------- | ------------------------ | -------------------------------------------------- |

| retryable_states         | 必须     | 允许重试的状态集合       | `["DISCOVERING","ANALYZING","RETRYING"]`         |

## 6.校验规则

### 6.1 必须校验项

    任务契约生成后，建议总控agent至少执行以下校验：
    1.`task_id` 与当前任务一致；
    2. `allowed_targets` 不为空；
    3. `enabled_modules` 与 `disabled_modules` 不得重复；
    4. `core_goal` 与 `intent_tags` 不得冲突；
    5. `allowed_stage_flow` 必须与系统状态机兼容；
    6. `expected_outputs` 与 `output_policy` 不得冲突；
    7. `review_required`、`review_trigger_rules` 与任务风险等级逻辑一致。

### 6.2 生成失败处理

    若任务契约生成失败，建议分三类处理：
    1. 缺失必须输入：回退到澄清阶段；
    2. 目标与边界冲突：回退到边界鉴定阶段；
    3. 模块策略或状态策略不合法：标记为系统生成错误，写入审计日志并停止推进。

## 7.落地约束

    为保证任务契约长期稳定，工程上应固定以下约束：
    1. 下游agent不得自行修改任务契约；
    2. 人工变更任务契约时必须提升`contract_version`；
    3. 任务契约变更后必须记录变更原因与操作来源；
    4. 任务执行过程中，只允许在明确规则下补充条件模块，不允许静默扩大任务范围；
    5. 任意阶段的启用、跳过、重试、暂停、复核，都必须能回溯到任务契约中的对应字段。

# 七、返回输出设计

## 1.实时输出层

### 1.1 设计目标

    实时输出层的目标，不是把底层事件原样推给前端，而是由总控agent输出“可直接展示、可持续刷新、可稳定追踪”的任务运行视图。该层主要回答以下问题：
    1. 当前任务进行到哪一步；
    2. 当前正在运行哪些agent；
    3. 最近发生了什么关键事件；
    4. 是否出现阻塞、失败、重试、暂停或等待；
    5. 前端是否需要刷新分析层数据。

    实时输出层只描述“过程状态”和“运行动态”，不承担发现详情聚合，也不直接暴露底层原始事件表。

### 1.2 输出原则

    实时输出层建议遵循以下原则：
    1. 轻量优先：适合高频轮询或后续消息推送；
    2. 稳定字段：字段名与结构尽量固定，避免前端频繁改协议；
    3. 聚合展示：同类底层事件应先由总控agent聚合后再输出；
    4. 状态优先：先输出状态、阶段、进度、阻塞信息，再输出次级细节；
    5. 可追踪：关键事件必须保留事件标识与来源引用，便于排障。

### 1.3 输出结构

    实时输出层建议统一输出为`realtime_snapshot（实时快照）`，结构上至少包含以下六个对象：
    1. `task_meta（任务元信息）`
    2. `task_runtime_state（任务运行状态）`
    3. `active_agents（活跃agent列表）`
    4. `recent_events（最近事件列表）`
    5. `runtime_alerts（运行告警列表）`
    6. `refresh_hints（刷新提示）`

### 1.4 `task_meta（任务元信息）` 参数设计

| 参数名     | 是否必须 | 参数解释                                 | 示例                          |
| ---------- | -------- | ---------------------------------------- | ----------------------------- |
| task_id    | 必须     | 任务唯一标识                             | `task_20260408_0001`        |
| task_name  | 可以缺省 | 任务展示名称，供前端列表和详情页展示     | `华星科技外围资产梳理`      |
| task_mode  | 必须     | 当前任务执行模式，如标准模式、低成本模式 | `标准模式`                  |
| created_at | 必须     | 任务创建时间                             | `2026-04-08T10:20:00+08:00` |
| updated_at | 必须     | 本次实时快照更新时间                     | `2026-04-08T10:31:10+08:00` |
| requester  | 可以缺省 | 提交者标识或名称                         | `user_admin`                |

| current_state          | 必须     | 当前任务全局状态                                     | `DISCOVERING（发现中）`      |

| affects_state  | 必须     | 是否影响状态流转             | `true`                                        |

### 1.8 `runtime_alerts（运行告警列表）` 参数设计

    运行告警用于把“值得用户注意但未必是错误”的情况独立展示，避免用户只能从事件流里读异常。

| 参数名           | 是否必须 | 参数解释                             | 示例                                         |

| related_agent    | 可以缺省 | 关联agent                            | `JS解析agent`                              |
| related_event_id | 可以缺省 | 关联事件标识                         | `evt_001041`                               |
| suggested_action | 可以缺省 | 建议动作                             | `继续观察，无需人工介入`                   |

### 1.9 `refresh_hints（刷新提示）` 参数设计

    刷新提示用于指导前端如何更新界面，而不是每次都整页重拉。

| 参数名                  | 是否必须 | 参数解释                   | 示例                                   |

| refresh_reason          | 可以缺省 | 触发刷新建议的原因         | `发现层新增有效资产，建议刷新分析层` |

## 2.分析层

### 2.1 设计目标

    分析层用于向前端提供“已经形成结构化结论的当前阶段成果”，其核心不是实时过程，而是当前可读、可比较、可筛选、可追踪的分析结果集合。

    分析层主要回答以下问题：
    1. 当前发现了哪些目标、实体、资产或线索；
    2. 每条发现的置信度、来源和关联对象是什么；
    3. 当前有哪些需要重点关注的高价值发现；
    4. 发现之间存在怎样的关联关系；
    5. 哪些结论已经确认，哪些仍然待复核。

### 2.2 输出原则

    分析层建议遵循以下原则：
    1. 结构化优先：前端不应自己再从文本中抽字段；
    2. 证据可追溯：每条分析结论必须能指向事件或证据引用；
    3. 置信度明确：不允许把推断和确认混在一起；
    4. 去重聚合：同类发现先归并，再返回前端；
    5. 阶段可见：要让前端知道结果来自哪个阶段、哪个agent。

### 2.3 输出结构

    分析层建议统一输出为`analysis_snapshot（分析快照）`，结构上至少包含以下七个对象：
    1. `summary_metrics（汇总指标）`
    2. `findings（发现列表）`
    3. `key_entities（关键实体列表）`
    4. `asset_overview（资产概览）`
    5. `relation_hints（关联提示）`
    6. `pending_reviews（待复核项）`
    7. `source_refs（来源引用）`

# 八、任务能力路由分解设计

## 1.设计目标

    任务能力路由分解，是总控agent把“用户任务”转化为“可执行多agent计划”的核心步骤。它解决的不是简单的模块映射问题，而是以下四个问题：
    1. 用户当前任务应该拆成哪些阶段和子任务；
    2. 每个子任务应该由哪个agent处理；
    3. 哪些agent需要立即启动，哪些应条件触发，哪些必须跳过；
    4. 当结果返回后，是否要补跑、加派、降级或升级复核。

    因此，第八部分的核心目标是建立一套“可解释、可配置、可追踪、可落地”的任务能力路由机制，使总控agent不依赖临场自由发挥，而是依据输入、边界、契约、事件和规则来完成路由拆解。

## 2.路由分解总体逻辑

### 2.1 总体流程

    总控agent的路由分解建议固定为六步：
    1. 输入识别：读取`normalized_input（标准化输入）` 与 `boundary_input（边界鉴定输入）`；
    2. 意图拆解：把任务目标拆成若干能力需求；
    3. 阶段映射：将能力需求挂到状态机阶段；
    4. agent路由：为每项能力需求匹配最合适的agent；
    5. 依赖排序：根据前置依赖生成执行顺序与条件触发关系；
    6. 路由输出：生成可执行的路由计划，供调度器直接消费。

### 2.2 路由判断优先级

    为避免多个规则同时命中导致混乱，建议按以下优先级进行路由判断：
    1. 边界限制优先：超出边界的模块直接禁止；
    2. 任务目标优先：先满足核心目标，再考虑增强模块；
    3. 明确输入优先：用户明确指定的目标与意图优先于推断；
    4. 低成本优先：在多个可选路径都成立时，优先选择成本更低的路径；
    5. 证据增益优先：只有能显著增加信息增益的补跑或加派才执行。

### 2.3 路由结果类型

    路由结果建议拆成四类：
    1. 直接路由：任务创建后立即生成并调度；
    2. 条件路由：满足某事件或阈值后才调度；
    3. 禁止路由：受边界、契约或策略限制而不允许调度；
    4. 升级路由：当出现高风险、低置信或冲突结论时，升级到复核链路。

## 3.任务分解逻辑

### 3.1 一级分解：按任务目标分解

    第一级分解要先回答“用户到底要做哪类事”。建议总控agent固定把任务拆到以下能力域：
    1. 企业情报；
    2. 资产发现；
    3. 应用分析；
    4. 图谱聚合；
    5. 价值评估；
    6. 复核与质量控制。

    例如：
    1. 若任务目标是“企业背景整理”，优先落到企业情报能力域；
    2. 若任务目标是“外围资产盘点”，优先落到资产发现能力域；
    3. 若任务目标是“接口、参数、隐藏路径分析”，优先落到应用分析能力域；
    4. 若任务目标是“关系梳理与关联理解”，优先落到图谱聚合能力域；
    5. 若任务目标是“高价值资产优先排序”，优先落到价值评估能力域。

### 3.2 二级分解：按阶段分解

    当一级能力域确定后，再将其映射到任务状态机阶段，建议固定映射如下：

| 阶段名称   | 阶段作用                         | 典型输入               | 典型输出         |
| ---------- | -------------------------------- | ---------------------- | ---------------- |
| 界定范围   | 确认主体、目标、边界、禁用项     | 标准化输入、边界输入   | 范围确认结果     |
| 发现       | 发现域名、地址、路径、候选资产   | 主目标、词典、种子目标 | 资产发现结果     |
| 分析       | 识别指纹、脚本线索、文件线索     | 页面、响应、脚本、文件 | 结构化线索       |
| 扩展       | 基于新线索进行补充发现或字典扩展 | 发现结果、分析结果     | 扩展资产与新词典 |
| 关联       | 将实体、资产、线索进行归并与关联 | 多源结果               | 关系摘要         |
| 优先级判定 | 计算优先级与关注顺序             | 资产、关系、风险线索   | 优先级列表       |
| 评审       | 处理低置信、冲突、高风险结论     | 争议发现、冲突线索     | 复核结论         |

### 3.3 三级分解：按agent能力分解

    在阶段内部，再把能力域细分到具体agent。建议映射如下：
    1. 企业主体、股东、备案、历史变更，分配给企业情报agent；
    2. 子域、IP、端口、基础路径，分配给基础发现agent；
    3. 页面技术栈、中间件、CMS 指纹，分配给指纹识别agent；
    4. JS 接口、参数、隐藏路径、外联域名，分配给 JS 解析agent；
    5. 新词生成、路径扩展、增量扫描词典，分配给扩展词典agent；
    6. 实体归并、资产关联、拓扑关系，分配给图谱聚合agent；
    7. 高价值资产排序、优先级标注，分配给价值评估agent；
    8. 高风险、低置信、冲突结论，分配给复核agent；
    9. 文件下载、文本抽取、敏感内容线索，分配给文件情报agent；
    10. 流量样本中的风险线索，分配给流量审查agent；
    11. 推断性链路提示，分配给链路提示agent。

## 5.路由拆解的具体实现逻辑

### 5.1 规则匹配流程

    路由规则匹配建议按以下顺序执行：
    1. 读取任务契约中的允许模块和禁止模块；
    2. 根据`intent_tags（意图标签）` 选出候选能力域；
    3. 根据 `target_types（目标类型）` 与 `primary_target（主目标）` 收窄规则集合；
    4. 对规则按 `priority（优先级）` 排序；
    5. 逐条校验阻断条件；
    6. 命中后生成路由项；
    7. 对重复能力项去重，保留最高优先级路由。

### 5.2 阶段内依赖处理

    路由器不能只决定“给谁”，还要决定“先后顺序”。建议固定以下依赖规则：
    1. 企业情报agent可与基础发现agent并行；
    2. 指纹识别agent依赖基础发现产生的页面或服务结果；
    3. JS 解析agent依赖页面抓取或脚本资源结果；
    4. 扩展词典agent依赖企业情报、指纹识别或 JS 解析的新增线索；
    5. 图谱聚合agent依赖多个阶段结果沉淀后再执行；
    6. 价值评估agent依赖资产与关系结果；
    7. 复核agent依赖争议结果，不参与初始发现链路。

### 5.3 条件触发逻辑

    对高成本模块，不建议在任务开始时统一下发，而应采用条件触发：
    1. 当发现阶段输出“JS 资源数超过阈值”时，生成 JS 解析路由；
    2. 当 JS 解析输出“新域名 / 新路径 / 文件线索”时，生成扩展词典或文件情报路由；
    3. 当分析阶段发现“高价值低置信结论”时，生成复核路由；
    4. 当图谱聚合阶段发现“关系缺口较大”时，可生成链路提示路由。

### 5.4 禁止与跳过逻辑

    路由器必须明确记录为什么没有调度某个agent，建议分为三种情况：
    1.`blocked（被阻止）`：被边界或契约明确禁止；
    2. `skipped（被跳过）`：当前任务目标不需要；
    3. `deferred（延后）`：当前阶段尚未满足依赖。

    这样可以避免后续排查时只看到“没执行”，却不知道是“不能做”还是“暂时没做”。

## 7.落地约束

    为保证任务能力路由长期稳定，建议固定以下约束：
    1. 路由器只能读取标准化输入、边界输入、任务契约和事件结果，不直接读取前端原始表单；
    2. 路由规则必须显式配置，不能只依赖大模型临场判断；
    3. 所有被阻止、跳过、延后、升级的路由都必须留痕；
    4. 同一能力项若被多个规则命中，应只保留一个最终路由结果；
    5. 条件模块的触发必须依赖事件或阈值，不能无限补跑；
    6. 任意新增agent时，都必须补充其能力项、输入要求、阶段位置和依赖关系，才能纳入正式路由。

# 九、状态存储管理设计

## 1.设计目标

    状态存储管理的目标，不是简单记录“当前任务进行到哪里”，而是要同时解决以下问题：
    1. 总控agent当前处于什么状态；
    2. 为什么会进入这个状态；
    3. 是由哪个事件、哪个agent、哪个操作触发的；
    4. 出现异常时，能否回放、重建、审计；
    5. 前端查询时，能否快速得到稳定一致的任务快照。

    因此本部分采用“事件驱动（Event-Driven，事件驱动）+ 状态快照（Snapshot，快照）+ 状态流转日志（State Transition Log，状态流转日志）”的组合设计。核心原则是：
    1. 状态推进以事件为准，不以页面显示结果为准；
    2. 当前状态可以缓存，但缓存不是唯一真相；
    3. 每一次状态变化都必须可追溯到具体触发原因；
    4. 任一任务都必须支持回放、重建、审计和故障恢复。

## 2.状态管理总体模型

### 2.1 三层状态视图

    为避免“写库一套、前端一套、调度器一套”的状态割裂，建议总控agent统一维护三层状态视图：
    1. 任务真实状态层：由事件流和状态流转日志共同定义，是系统唯一事实来源；
    2. 当前查询状态层：由任务主表中的当前状态字段和快照表提供，服务于前端和接口查询；
    3. 运行时控制状态层：由调度器、队列和agent执行器维护，服务于调度与故障控制。

    三层关系如下：
    1. 真实状态层决定“状态是什么”；
    2. 查询状态层决定“状态怎么高效读出来”；
    3. 运行时控制状态层决定“状态下一步怎么推进”。

### 2.2 全局任务状态集合

    总控agent的全局任务状态固定为以下集合，并作为数据库枚举或常量表统一管理：
    1.`CREATED（已创建）`
    2. `SCOPED（已界定范围）`
    3. `DISCOVERING（发现中）`
    4. `ANALYZING（分析中）`
    5. `EXPANDING（扩展中）`
    6. `CORRELATING（关联中）`
    7. `PRIORITIZING（优先级判定中）`
    8. `REPORTING（报告生成中）`
    9. `REVIEWING（复核中）`
    10. `DONE（已完成）`
    11. `RETRYING（重试中）`
    12. `PAUSED（已暂停）`
    13. `PARTIAL_DONE（部分完成）`
    14. `FAILED（失败）`
    15. `QUARANTINED（隔离中）`

    其中：
    1. 前十项属于主流程状态；
    2. 后五项属于旁路控制状态；
    3. 状态机只允许按预定义规则跳转，不允许任意跨阶段写入。

### 2.3 单个agent运行状态集合

    每个agent执行实例建议单独维护运行状态，用于区分“任务卡住”与“某个agent还在跑”：
    1.`PENDING（待领取）`
    2. `CLAIMED（已领取）`
    3. `RUNNING（执行中）`
    4. `WAIT_TOOL（等待工具）`
    5. `WAIT_EVENT（等待事件）`
    6. `SELF_CHECK（自检中）`
    7. `DONE（完成）`
    8. `RETRY（待重试）`
    9. `FAILED（失败）`
    10. `ESCALATED（已升级复核）`
    11. `CANCELLED（已取消）`

## 3.状态存储对象设计

### 3.1 核心存储对象

    状态管理建议至少拆分以下七类核心对象，分别承担不同职责：

| 对象名                                   | 作用                                    | 存储建议                   | 说明                           |
| ---------------------------------------- | --------------------------------------- | -------------------------- | ------------------------------ |
| `tasks（任务主表）`                    | 保存任务当前态、当前阶段、主视图字段    | PostgreSQL（关系型数据库） | 面向查询的当前任务入口         |
| `task_state_transitions（状态流转表）` | 记录每次状态变化                        | PostgreSQL（关系型数据库） | 面向审计与重建                 |
| `observation_events（观察事件表）`     | 记录agent、工具、人工动作产生的标准事件 | PostgreSQL（关系型数据库） | 是状态推进的原始依据           |
| `agent_runs（agent执行表）`            | 记录每次agent执行实例                   | PostgreSQL（关系型数据库） | 用于定位执行耗时、失败原因     |
| `task_snapshots（任务快照表）`         | 保存前端直接查询的聚合结果              | PostgreSQL（关系型数据库） | 避免每次现算                   |
| `context_capsules（上下文胶囊表）`     | 保存阶段摘要和可复用上下文              | PostgreSQL（关系型数据库） | 用于恢复和阶段切换             |
| `state_runtime_cache（状态运行缓存）`  | 保存短期锁、去重键、心跳、租约          | Redis（内存键值库）        | 面向高频调度，不作为最终事实源 |

### 3.2 任务主表 `tasks（任务主表）` 建议字段

| task_contract_ref | 必须     | 任务契约引用               | `contract_20260408_0001`    |
| latest_event_id   | 可以缺省 | 最近一次成功消费的事件标识 | `evt_001028`                |
| snapshot_id       | 可以缺省 | 当前快照引用               | `snap_000117`               |
| progress_percent  | 可以缺省 | 任务进度百分比             | `35`                        |

### 3.3 状态流转表 `task_state_transitions（状态流转表）` 建议字段

| 字段名            | 是否必须 | 说明                                     | 示例                             |

## 4.状态管理实现方式

### 4.1 状态推进采用“事件驱动更新 + 快照查询”

    状态推进不建议由前端轮询后自行猜测，也不建议由任意agent直接改写任务当前状态。正确实现方式如下：
    1. agent、工具适配器、人工动作先产出标准事件；
    2. 事件统一写入`observation_events（观察事件表）`；
    3. 总控agent中的状态处理器读取事件并校验是否合法；
    4. 若满足状态转移规则，则写入 `task_state_transitions（状态流转表）`；
    5. 同时更新 `tasks（任务主表）` 的 `current_state`、`current_stage`、`state_version`；
    6. 异步或同步刷新 `task_snapshots（任务快照表）`，供前端查询。

    也就是说：
    1. 事件是“为什么变”；
    2. 流转表是“怎么变”；
    3. 主表是“现在是什么”；
    4. 快照是“对前端怎么展示”。

### 4.2 状态处理器实现建议

    建议在总控agent内部实现独立的状态处理器`StateReducer（状态归并器）`，职责如下：
    1. 接收事件；
    2. 判断该事件是否重复；
    3. 判断该事件是否允许触发当前状态跳转；
    4. 生成新的状态流转记录；
    5. 更新任务当前状态；
    6. 触发快照重建与下游调度。

    `StateReducer（状态归并器）` 不应由各个agent各自实现，而应由总控agent统一维护。这样可以避免：
    1. 不同agent对同一状态含义理解不一致；
    2. 出现非法跨阶段流转；
    3. 出现“事件已落库但状态没更新”或“状态更新了但没有证据”的不一致问题。

### 4.3 状态转移规则实现建议

    建议维护一份显式状态转移表`State Transition Matrix（状态转移矩阵）`，以配置或常量形式定义允许的流转关系。例如：
    1. `CREATED（已创建） -> SCOPED（已界定范围）`
    2. `SCOPED（已界定范围） -> DISCOVERING（发现中）`
    3. `DISCOVERING（发现中） -> ANALYZING（分析中）`
    4. 任意主流程状态 -> `PAUSED（已暂停）`
    5. 任意主流程状态 -> `FAILED（失败）`
    6. 高风险结论阶段 -> `REVIEWING（复核中）`

    每次流转前必须校验：
    1. 当前状态是否合法；
    2. 目标状态是否合法；
    3. 触发事件是否匹配；
    4. 是否满足前置依赖；
    5. 是否存在更高优先级的阻断状态，如`QUARANTINED（隔离中）`。

### 4.4 一致性实现建议

    为降低“事件写入成功但状态没改”或“状态改了但快照没更新”的问题，建议将以下操作放入同一数据库事务`Transaction（事务）` 中执行：
    1. 写入状态流转记录；
    2. 更新任务主表当前状态；
    3. 记录最近消费事件标识；
    4. 写入必要的审计字段。

    `task_snapshots（任务快照表）` 的刷新可根据性能选择：
    1. 同步更新：适合低并发，查询一致性更强；
    2. 异步更新：适合高并发，通过事件补偿保证最终一致。

    但无论采用哪种模式，主表状态与流转表必须先一致，快照只能延迟，不能与事实层冲突。

## 5.状态追踪设计

### 5.1 追踪目标

    状态追踪要解决三个问题：
    1. 我们如何知道任务现在卡在哪里；
    2. 我们如何知道是谁让任务进入这个状态；
    3. 我们如何知道从开始到现在完整发生了什么。

    因此，状态追踪至少覆盖以下链路：
    1. 任务追踪；
    2. 阶段追踪；
    3. agent追踪；
    4. 事件追踪；
    5. 人工操作追踪；
    6. 错误与重试追踪。

### 5.2 任务级追踪

    任务级追踪面向“一个任务从创建到结束的生命周期”：
    1. 记录任务创建时间、最后活跃时间、当前状态、当前阶段；
    2. 记录最近一次流转原因；
    3. 记录当前是否被阻塞、暂停、失败或隔离；
    4. 记录累计执行耗时、累计重试次数、累计失败次数；
    5. 记录当前活跃agent数量与待执行agent数量。

### 5.3 事件级追踪

    事件级追踪面向“任务为什么会变化”：
    1. 每个事件必须带`event_id（事件标识）`；
    2. 每个事件必须带 `task_id（任务标识）`；
    3. 每个事件必须带 `agent_id（agent标识）` 或人工来源；
    4. 每个事件必须带 `dedupe_key（去重键）`，防止重复消费；
    5. 每个事件必须带 `payload_ref（负载引用）` 或 `evidence_ref（证据引用）`。

    通过这五个字段，可以从一个任务状态追溯到：
    1. 哪个agent产生了事件；
    2. 该事件引用了什么证据；
    3. 该事件是否被重复消费；
    4. 该事件是否真正触发了状态流转。

### 5.4 状态时间线设计

    建议对外提供统一的状态时间线`Timeline（时间线）` 视图，供前端和排障使用。每条时间线记录建议包含：
    1. 时间；
    2. 所处状态；
    3. 触发来源；
    4. 触发说明；
    5. 关联agent；
    6. 关联事件；
    7. 是否异常；
    8. 是否需要人工关注。

    这样前端就不需要自己拼接底层事件，只需读取任务快照和时间线即可稳定展示任务过程。

### 5.5 审计与回放设计

    状态追踪必须支持回放`Replay（回放）`。建议实现方式如下：
    1. 读取某任务的全部 `observation_events（观察事件表）`；
    2. 按事件时间和消费顺序重放；
    3. 根据 `task_state_transitions（状态流转表）` 校验每一步是否合法；
    4. 如主表状态与回放结果不一致，以流转表和事件表为准进行修复；
    5. 将修复过程写入审计日志 `Audit Log（审计日志）`。

    这意味着：
    1. 主表状态可以修；
    2. 快照可以重建；
    3. 事件和流转日志不能被静默篡改；
    4. 任何人工修正也必须留下日志。

## 6.故障恢复与排障设计

### 6.1 异常场景

    状态管理至少要覆盖以下异常场景：
    1. 事件重复写入；
    2. 状态重复流转；
    3. agent执行超时但任务未回收；
    4. 主表状态与快照不一致；
    5. 主表状态与流转表不一致；
    6. 某阶段已经完成但前端仍显示处理中。

### 6.2 恢复机制

    建议固定以下恢复动作：
    1. 事件重复：根据`dedupe_key（去重键）` 拦截并记录重复次数；
    2. 状态异常：根据 `task_state_transitions（状态流转表）` 重新计算当前状态；
    3. 快照损坏：根据主表状态、最近事件和聚合结果重新生成快照；
    4. agent失联：根据 Redis（内存键值库）中的租约和心跳判断是否回收任务；
    5. 长时间无进展：标记为阻塞并生成告警事件。

### 6.3 排障建议

    排查状态问题时，建议按以下顺序进行：
    1. 先看`tasks（任务主表）` 当前状态；
    2. 再看 `task_state_transitions（状态流转表）` 最后一次流转；
    3. 再看 `observation_events（观察事件表）` 最近事件是否被消费；
    4. 再看 `agent_runs（agent执行表）` 是否有失败、超时、重试；
    5. 最后看 `task_snapshots（任务快照表）` 是否只是展示层未刷新。

## 7.落地约束

    为保证状态管理长期可用，工程上应固定以下约束：
    1. 不允许任意agent直接改写`tasks（任务主表）` 的当前状态；
    2. 不允许以前端轮询结果作为状态推进依据；
    3. 不允许只写当前状态、不写状态流转日志；
    4. 不允许只记录成功状态，不记录失败、重试、暂停和人工干预；
    5. 不允许快照成为唯一数据源，快照只能服务查询；
    6. 所有英文状态名、字段名和机制名在文档与接口层都应附中文说明，避免多人协作时理解偏差。

# 十、多agent的异步调用与并发设计

## 1.设计目标

    多agent的异步调用与并发设计，不是单纯为了“更快执行”，而是为了让总控agent在多模块协同时具备可扩展、可恢复、可隔离、可控成本、可追踪的运行能力。该部分重点解决以下问题：
    1. 总控agent如何把同步串行调用改造成统一的异步任务流；
    2. 多个agent如何在不彼此直连的前提下协同执行；
    3. 高成本模块如何受控并发，避免拖垮整个运行时；
    4. 重复任务、重复事件、循环扩展如何被拦截；
    5. 执行器异常退出、超时、失败后如何回收与恢复。

    一期默认采用“Flask 同进程 API + 后台轻量执行器 + Redis（内存键值库）短期调度缓存 + PostgreSQL（关系型数据库）持久化状态”的方案，不额外引入新的重型消息中间件。异步执行只改变后端内部调度方式，不改变外部 API 形态。

## 2.异步调用总体模型

### 2.1 总体调用链路

    总控agent的异步调用链路建议固定如下：
    1.`Orchestrator（总控调度器）` 根据任务契约和路由计划生成 `AgentTask（agent任务）`；
    2. `AgentTask（agent任务）` 落入待调度队列，并生成 `TaskDispatchTicket（调度票据）`；
    3. `AgentExecutor（agent执行器）` 领取调度票据并申请 `Lease（租约）`；
    4. 执行器加载 `input_ref（输入引用）` 与 `context_ref（上下文引用）` 后执行对应agent；
    5. agent输出统一转换为 `ObservationEvent（观察事件）`；
    6. `StateReducer（状态归并器）` 消费事件并推进状态机；
    7. 若事件触发条件路由、补跑或复核，则由总控agent再次生成新的 `AgentTask（agent任务）`。

    这意味着：
    1. agent之间不直接调用彼此；
    2. 所有下游执行都先任务化，再执行；
    3. 所有执行结果都先事件化，再推进状态；
    4. 总控agent始终是唯一的调度与补跑决策者。

### 2.2 调度层分工

    异步调用建议拆成四层职责：
    1.`Orchestrator（总控调度器）`：决定发什么任务、发给谁、什么时候补跑；
    2. `QueueManager（队列管理器）`：维护任务入队、出队、优先级和死信隔离；
    3. `AgentExecutor（agent执行器）`：领取并执行单个 `AgentTask（agent任务）`；
    4. `StateReducer（状态归并器）`：将执行结果转化为状态推进依据。

## 4.并发控制设计

### 4.1 三层并发模型

    并发控制建议至少分三层：
    1. 任务级并发：限制单个`task_id（任务标识）` 在同一时刻可同时运行多少个子任务；
    2. agent级并发：限制某类高成本agent在整个系统中的并发上限；
    3. 目标级并发：限制同一 `target_key（目标归一标识）` 在同一时刻被多少个任务或agent并行处理。

    三层控制的作用如下：
    1. 任务级并发防止单个任务吞掉全部系统资源；
    2. agent级并发防止高成本模块成为系统瓶颈；
    3. 目标级并发防止同一域名、地址或证据对象被重复轰炸。

### 4.2 并发冲突优先级

    当同一子任务同时命中多层并发限制时，建议按以下优先级处理：
    1. 目标级并发优先；
    2. agent级并发其次；
    3. 任务级并发最后。

    原因是：
    1. 目标级冲突最容易导致重复抓取和热点拥塞；
    2. agent级冲突最容易造成高成本模块堆积；
    3. 任务级冲突影响的是单任务吞吐，不是全局风险最高项。

### 4.3 默认并发策略

    一期默认策略建议写入 runtime（运行时）配置，而不是写死在 prompt（提示词）中。默认推荐如下：
    1. 企业情报模块：低并发；
    2. 基础发现模块：中并发；
    3. 指纹识别模块：中并发；
    4. JS 解析模块：低并发；
    5. 文件情报模块：低并发；
    6. 图谱聚合模块：单任务单实例；
    7. 价值评估模块：单任务单实例；
    8. 复核模块：低并发。

### 4.4 并发配额实现建议

    并发配额建议通过 Redis（内存键值库）中的配额计数器与短期锁实现：
    1. 任务级配额键：`concurrency:task:{task_id}`；
    2. agent级配额键：`concurrency:agent:{agent_type}`；
    3. 目标级配额键：`concurrency:target:{target_key}`。

    领取任务前，执行器必须先检查三层配额：
    1. 任一配额超限，则不得领取；
    2. 可以进入等待队列或延迟重试；
    3. 释放任务或任务结束后，必须回收配额。

## 5.去重、幂等与防循环设计

### 5.1 去重键生成规则

    每个`ObservationEvent（观察事件）` 与高风险重复调度都建议生成 `Dedupe Key（去重键）`。推荐构成方式如下：

    `task_id + agent_type + normalized_target + event_type + content_hash`

    例如：`task_20260408_0001:asset_discovery:domain:hxtech.com:DISCOVERY_BATCH_DONE:sha256_abcd1234`

### 5.2 重复事件处理逻辑

    同一去重窗口内若再次出现相同`Dedupe Key（去重键）`，建议按以下规则处理：
    1. 不重复触发下游补跑；
    2. 不重复推进状态机；
    3. 只更新重复计数；
    4. 必要时刷新最近发生时间；
    5. 写入审计说明，标记为重复消费已拦截。

### 5.3 幂等控制逻辑

    异步运行下，必须假设以下情况可能发生：
    1. 执行器超时后又成功回写结果；
    2. 同一任务被多个执行器重复领取；
    3. 事件被重复投递；
    4. 网络抖动导致“执行成功但确认失败”。

    因此建议：
    1.`AgentTask（agent任务）` 状态变更必须带版本号或原子条件；
    2. `Lease（租约）` 校验失败的执行器不得回写最终结果；
    3. 状态推进只能基于“未消费事件”；
    4. 已完成或已消费对象再次写入时必须走幂等校验。

### 5.4 防循环设计

    对于“发现 -> 扩展 -> 再发现 -> 再解析”的链路，建议固定三类防循环阈值：
    1. 最大展开深度：例如最多展开 3 轮；
    2. 最小增益阈值：新增信息增益低于阈值则停止扩展；
    3. 阶段循环上限：同一阶段对同一目标的重复进入次数有限制。

    例如：
    1. JS 解析发现新路径后触发扩展词典；
    2. 扩展词典反过来触发基础发现；
    3. 若两轮后新增有效目标数趋近于 0，则停止链路扩展，进入下一阶段。

## 6.异常控制与恢复设计

### 6.1 `pause（暂停）`

    `Pause（暂停）` 作用于任务级，而不是强行终止所有执行器。建议语义如下：
    1. 不再下发新的 `AgentTask（agent任务）`；
    2. 已领取任务允许自然结束；
    3. 当前运行结果仍然落库并保留；
    4. 任务状态进入 `PAUSED（已暂停）`。

### 6.2 `resume（恢复）`

    `Resume（恢复）` 建议从最近稳定状态继续，而不是整任务重跑：
    1. 读取最后稳定的状态与未完成子任务；
    2. 恢复调度队列；
    3. 重新发放未完成且未失效的子任务；
    4. 状态从 `PAUSED（已暂停）` 回到可执行状态。

### 6.3 `retry（重试）`

    重试必须区分三层：
    1. agent级重试：单个`AgentTask（agent任务）` 的局部失败重试；
    2. 阶段级重试：当一个阶段整体关键结果缺失时，对该阶段重新补算；
    3. 任务级重试：只在明确人工触发时使用，不作为默认自动策略。

    默认建议：
    1. 工具超时、临时网络错误，允许 agent级重试；
    2. 格式异常、解析失败，先自检修复一次，再进入复核或失败；
    3. 非法状态流转禁止盲目重试，必须进入隔离检查。

### 6.4 `timeout（超时）`

    超时控制建议分两层：
    1. 执行超时：单个agent执行超过`deadline（截止时间）`；
    2. 租约超时：`Lease（租约）` 超过 `lease_expire_at（到期时间）` 未续租。

    超时处理建议如下：
    1. 记录超时事件；
    2. 若仍有有效租约，则等待一次心跳恢复；
    3. 若租约失效，则回收任务；
    4. 根据重试策略决定重新入队还是转入隔离。

### 6.5 `quarantine（隔离）`

    当出现异常循环、重复失败、结果冲突严重或疑似数据污染时，任务或子任务应进入`Quarantine（隔离）`。建议通过 `Dead Letter Queue（死信队列）` 或“隔离任务队列”承接此类任务。

    进入隔离的典型条件：
    1. 同一子任务连续失败超过阈值；
    2. 同一目标重复扩展超过上限；
    3. 多个agent结论冲突且无法自动归并；
    4. 状态流转与事件事实不一致；
    5. 出现疑似脏数据回放。

# 十一、上下文裁剪设计

## 1.设计目标

    上下文裁剪设计的核心目标，不是单纯“把内容缩短”，而是在不超过模型 token（令牌）上限的前提下，保证上下文仍然连续、可理解、可追溯、可继续推理。该部分主要解决以下问题：
    1. 如何在不同模型窗口下自动检测可用上下文长度；
    2. 如何在发送模型之前判断是否超限；
    3. 如何在超限时按规则裁剪，而不是粗暴截断；
    4. 如何在裁剪后仍保持阶段语义连续、证据链不断裂；
    5. 如何在最终发送前做二次校验，确保绝不超过模型上限。

    因此，第十一部分采用“预算先行、分层装配、评分裁剪、胶囊保连续、发送前复检”的设计原则。总控agent必须做到：
    1. 先估算，再组装，再裁剪，最后复检；
    2. 不允许在已完成组装后直接截断最后 N 个 token；
    3. 不允许破坏 JSON、代码块、请求对、函数边界和阶段语义；
    4. 若仍超限，必须拆任务、降级或换更大窗口模型，而不是强塞。

## 2.上下文裁剪总体模型

### 2.1 总体处理链路

    建议总控agent的上下文裁剪固定采用以下链路：

    `ContextAssembler（上下文装配器） -> TokenEstimator（长度估算器） -> ContextScorer（上下文评分器） -> ContextPruner（上下文裁剪器） -> ContextCapsuleBuilder（上下文胶囊构建器） -> FinalPromptPayload（最终提示负载）`

    该链路的职责如下：
    1. `ContextAssembler（上下文装配器）`：负责按层装配候选上下文；
    2. `TokenEstimator（长度估算器）`：负责计算或估算候选内容 token 数；
    3. `ContextScorer（上下文评分器）`：负责为候选内容打分，确定保留优先级；
    4. `ContextPruner（上下文裁剪器）`：负责按预算和分层规则删减、压缩、替换；
    5. `ContextCapsuleBuilder（上下文胶囊构建器）`：负责生成跨阶段连续性锚点；
    6. `FinalPromptPayload（最终提示负载）`：负责形成最终可发送模型的结构，并在发送前二次校验。

### 2.2 裁剪顺序约束

    上下文处理必须严格遵守以下顺序：
    1. 先读取模型窗口信息；
    2. 再生成 `context_budget_plan（上下文预算计划）`；
    3. 再装配候选上下文；
    4. 再做长度估算与评分；
    5. 再按规则裁剪；
    6. 再生成最终上下文负载；
    7. 最后做一次发送前复检。

    禁止的做法包括：
    1. 先把全部历史拼进去，再靠字符串截断；
    2. 不做预算计算，直接把长文本发给模型；
    3. 裁剪时切断代码结构、HTML 结构、请求链或 JSON 结构；
    4. 超限后继续强发，期待模型“自己适应”。

## 3.自动长度检测设计

### 3.1 `model_registry（模型注册表）`

    `model_registry（模型注册表）` 是上下文预算的唯一来源，用于记录不同模型的窗口能力和默认预留比例，不允许写死在 prompt（提示词）中。

| 参数名 | 是否必须 | 参数注释 | 示例 |
| --- | --- | --- | --- |
| model_name | 必须 | 模型名称 | `gpt-5.2` |
| max_context_window | 必须 | 模型最大上下文窗口 | `128000` |
| recommended_context_ratio | 必须 | 推荐上下文占比，用于正常装配目标 | `0.70` |
| hard_context_ratio_limit | 必须 | 硬上限占比，超过必须裁剪或拆包 | `0.75` |
| default_reserved_output_ratio | 必须 | 默认输出预留比例 | `0.15` |
| default_reserved_system_ratio | 必须 | 默认系统提示预留比例 | `0.10` |
| default_reserved_safety_ratio | 必须 | 默认安全缓冲预留比例 | `0.10` |

### 3.2 `context_budget_plan（上下文预算计划）`

    上下文预算必须先形成结构化计划，再进入装配环节。预算公式固定为：

    `budget_context = max_context_window - reserved_output - reserved_system - reserved_safety`

| 参数名 | 是否必须 | 参数注释 | 示例 |
| --- | --- | --- | --- |
| task_id | 必须 | 当前任务标识 | `task_20260408_0001` |
| agent_type | 必须 | 当前即将执行的 agent 类型 | `js_semantic` |
| model_name | 必须 | 当前使用模型名称 | `gpt-5.2` |
| max_context_window | 必须 | 模型最大上下文窗口 | `128000` |
| reserved_output | 必须 | 输出预留 token 数 | `19200` |
| reserved_system | 必须 | 系统提示预留 token 数 | `12800` |
| reserved_safety | 必须 | 安全缓冲 token 数 | `12800` |
| budget_context | 必须 | 实际可用于上下文装配的 token 预算 | `83200` |
| budget_status | 必须 | 当前预算状态，如充足、紧张、超限 | `充足` |

### 3.3 `token_estimation_result（长度估算结果）`

    上下文候选项必须逐项估算长度，而不是只估算总量。这样才能知道应该删什么、压什么、保什么。

| 参数名 | 是否必须 | 参数注释 | 示例 |
| --- | --- | --- | --- |
| context_item_id | 必须 | 上下文项唯一标识 | `ctx_html_0012` |
| content_type | 必须 | 内容类型，如 HTML、JS、文本、事件、胶囊 | `HTML` |
| estimated_tokens | 必须 | 当前项估算 token 数 | `620` |
| layer | 必须 | 所属层级，范围 1~5 | `4` |
| score | 必须 | 综合保留评分 | `0.81` |
| must_keep | 必须 | 是否属于必须保留项 | `false` |
| compression_allowed | 必须 | 是否允许压缩 | `true` |
| trim_priority | 必须 | 裁剪优先级，数值越高越优先被裁剪 | `70` |

### 3.4 `final_context_check（最终上下文校验）`

    发送模型前必须做一次最终校验，保证 `actual_context_tokens（实际上下文 token）` 不超过 `budget_context（上下文预算）`。

| 参数名 | 是否必须 | 参数注释 | 示例 |
| --- | --- | --- | --- |
| model_name | 必须 | 当前模型名称 | `gpt-5.2` |
| budget_context | 必须 | 预算允许的最大上下文 token | `83200` |
| actual_context_tokens | 必须 | 当前最终负载实际 token 数 | `80120` |
| within_limit | 必须 | 是否在预算内 | `true` |
| overflow_tokens | 必须 | 若超限则记录超出的 token 数，未超限为 0 | `0` |
| overflow_action | 必须 | 超限时采取的动作 | `none` |
| checked_at | 必须 | 最终校验时间 | `2026-04-08T11:05:18+08:00` |

### 3.5 长度检测规则

    长度检测必须固定为“两次检测”：
    1. 第一次检测：在候选上下文装配后进行预算估算；
    2. 第二次检测：在最终负载生成后进行发送前复检。

    其中：
    1. 第一次检测决定“要不要裁剪”；
    2. 第二次检测决定“能不能发送”；
    3. 若第二次检测 `within_limit=false`，则必须阻止发送并记录失败原因。

## 4.上下文分层与裁剪顺序

### 4.1 五层上下文模型

    第十一部分沿用前文已确定的五层模型，但将其明确为“可执行裁剪顺序”：
    1. 第 1 层不可裁剪：`task_contract（任务契约）`、边界、当前阶段目标、禁用项；
    2. 第 2 层高优先保留：确认事实、关键资产、已验证结论、Review（复核）结论；
    3. 第 3 层工作上下文：当前待解问题、最近上游摘要、直接依赖事件；
    4. 第 4 层证据片段：HTML、JS、文件、流量、图谱片段；
    5. 第 5 层冗余历史：旧日志、低置信、重复片段、细粒度执行日志。

### 4.2 裁剪顺序

    裁剪顺序必须从低价值层开始，固定如下：
    1. 先裁第 5 层冗余历史；
    2. 再压缩第 4 层长证据片段；
    3. 再压缩第 3 层工作上下文；
    4. 第 2 层仅在极端情况下做摘要压缩；
    5. 第 1 层禁止删除，只允许最小化结构表达。

### 4.3 不允许删除的内容

    以下内容即使预算紧张，也不得直接删除：
    1. 当前任务契约与边界约束；
    2. 当前阶段目标；
    3. 明确禁用项；
    4. 已确认核心事实；
    5. 与当前推理链直接相关的必要依赖锚点。

### 4.4 超过高优先层仍超限时的处理

    若第 1~3 层压缩后仍然超限，则不允许继续强裁，必须执行以下动作之一：
    1. 降低任务粒度；
    2. 将任务拆分为更小的 `AgentTask（agent任务）`；
    3. 切换更大窗口模型；
    4. 阻止发送并返回结构化错误。

## 5.连续性保障设计

### 5.1 `chunk_policy（分块策略）`

    为保证裁剪后的内容仍然连续，所有长内容都必须先分块，再评分、再裁剪。

| 参数名 | 是否必须 | 参数注释 | 示例 |
| --- | --- | --- | --- |
| content_type | 必须 | 内容类型 | `JS` |
| chunk_size_min | 必须 | 最小分块 token 大小 | `300` |
| chunk_size_max | 必须 | 最大分块 token 大小 | `600` |
| overlap_size | 必须 | 相邻块之间的重叠 token 大小 | `80` |
| split_rule | 必须 | 分块规则 | `按函数与对象边界分块` |
| continuity_rule | 必须 | 连续性保障规则 | `禁止切断函数体和调用链` |

### 5.2 `overlap_policy（重叠策略）`

    `Overlap（重叠）` 不是冗余浪费，而是连续性保障手段。它的作用是：
    1. 保证相邻块之间语义不断裂；
    2. 避免函数、请求链、页面区域在块边界被切开后无法理解；
    3. 提高模型对上下游依赖的恢复能力。

    不同内容类型的建议如下：
    1. HTML：重叠 80~120 token；
    2. JS：重叠 50~100 token；
    3. 文本：按段落自然过渡，必要时保留上下标题；
    4. 流量：请求与响应必须成对保留，不按单条切片；
    5. 事件历史：按阶段聚合，不按单条日志切断。

### 5.3 `context_capsule（上下文胶囊）`

    `Context Capsule（上下文胶囊）` 是跨阶段连续性的主锚点，不允许完全依赖原始长历史替代它。

| 参数名 | 是否必须 | 参数注释 | 示例 |
| --- | --- | --- | --- |
| capsule_id | 必须 | 胶囊唯一标识 | `capsule_discovery_0003` |
| stage | 必须 | 所属阶段 | `发现` |
| summary | 必须 | 阶段摘要，建议 200~500 token 内 | `已完成主域及 12 个子域发现，其中 3 个高价值候选目标待后续分析` |
| confirmed_facts | 必须 | 当前阶段已确认事实列表 | `["主体已确认","主域可访问","发现 3 个活跃接口子域"]` |
| open_questions | 可以缺省 | 待验证问题列表 | `["api-admin.hxtech.com 是否为后台管理入口"]` |
| key_entity_refs | 必须 | 关键实体引用列表 | `["ent_0012","asset_0088"]` |
| key_evidence_refs | 必须 | 关键证据引用列表 | `["evidence://http/1001","evidence://dns/2208"]` |
| next_actions | 必须 | 下阶段建议动作列表 | `["继续指纹识别","必要时触发脚本解析"]` |
| excluded_paths | 可以缺省 | 已排除路线，避免重复工作 | `["已排除第三方回调域名扩展"]` |

### 5.4 `dependency_anchor（依赖锚点）`

    除胶囊外，还应为当前任务保留最小依赖锚点，用于说明：
    1. 当前agent依赖哪些上游结果；
    2. 哪些证据是当前推理链的必要前置；
    3. 哪些内容虽然短，但必须原样保留。

    典型依赖锚点包括：
    1. 当前阶段目标；
    2. 当前问题陈述；
    3. 关键实体 ID；
    4. 关键证据引用；
    5. 最近一次决定性的事件。

### 5.5 `source_refs（来源引用）`

    裁剪后仍必须保留来源引用，保证“被压缩”和“被省略”不会造成不可追溯。建议每个关键结论至少带一种引用：
    1. `event_ref（事件引用）`
    2. `evidence_ref（证据引用）`
    3. `capsule_ref（胶囊引用）`
    4. `entity_ref（实体引用）`

## 6.超预算降级与拆包设计

### 6.1 降级顺序

    当 `actual_context_tokens > budget_context` 时，必须按固定顺序降级：
    1. 删除低分冗余历史；
    2. 用阶段摘要替代原始执行日志；
    3. 用“摘要 + 关键片段 + 对象引用”替代长证据；
    4. 压缩第 3 层工作上下文；
    5. 降低任务粒度，拆成更小 `AgentTask（agent任务）`；
    6. 必要时切换更大窗口模型；
    7. 若仍超限，则中止发送并返回结构化错误。

### 6.2 长证据降级规则

    长证据不得直接整块删除，建议统一替换为三段式：
    1. 证据摘要；
    2. 最关键片段；
    3. 原始对象引用。

    例如：
    1. HTML 页面不直接保留整页，而保留关键 DOM 区域 + 页面引用；
    2. JS 文件不直接保留整文件，而保留关键函数片段 + 文件引用；
    3. 流量日志不保留整会话，而保留关键请求对 + 会话引用。

### 6.3 拆包条件

    以下情况建议直接拆包，而不是继续压缩：
    1. 当前任务问题本身已经包含多个独立分析目标；
    2. 第 1~3 层压缩后仍接近模型硬上限；
    3. 关键证据必须保留大量原始结构；
    4. 当前agent实际承担了多个可拆分子任务。

### 6.4 超限错误输出

    若最终仍超限，必须输出结构化错误，不允许硬发送。建议至少包含：
    1. 当前模型；
    2. 预算值；
    3. 实际估算值；
    4. 已执行的降级动作；
    5. 下一步建议动作，如拆任务或换模型。

# 十二、错误处理设计

## 1.设计目标

    总控agent的错误处理设计，不是简单补一个try/catch，而是要把“错误被发现、被分类、被记录、被决策、被恢复、被审计”的完整链路固化下来。错误处理模块至少要满足以下目标：
    1. 不丢错误：任何agent、工具、状态机、存储层异常都必须结构化落库；
    2. 不盲重试：是否重试必须由错误分类和重试策略决定，而不是统一反复执行；
    3. 不整任务陪葬：非核心分支失败时，应优先进入降级或部分完成，而不是直接整任务失败；
    4. 不让污染扩散：遇到状态错乱、证据冲突、范围越界、重复循环时，必须支持隔离；
    5. 不可黑盒：前端、审计和运维都要能看到“错在哪里、做了什么、当前还能否继续”；
    6. 不可一次性：错误恢复必须支持pause / resume / retry / replay，而不是只能从头重跑。

## 2.错误处理总体模型

### 2.1 总体处理链路

    建议总控agent将错误处理固定为以下链路：
    `ErrorCapture（错误捕获） -> ErrorClassifier（错误分类） -> SeverityEvaluator（严重度评估） -> RetryPlanner（重试决策） -> CompensationDispatcher（补偿派发） -> DegradationPlanner（降级决策） -> StateGuard（状态保护） -> ErrorSnapshotBuilder（错误快照构建）`

    各组件职责建议如下：
    1. `ErrorCapture（错误捕获）`：统一接收agent异常、工具异常、状态异常、写库异常和人工阻断信号；
    2. `ErrorClassifier（错误分类）`：根据来源、错误码、上下文阶段、影响对象对错误归类；
    3. `SeverityEvaluator（严重度评估）`：判断错误是否可忽略、可重试、可降级、必须隔离；
    4. `RetryPlanner（重试决策）`：根据错误类型与次数生成重试计划；
    5. `CompensationDispatcher（补偿派发）`：在主写入失败或中断时生成补偿动作；
    6. `DegradationPlanner（降级决策）`：决定跳过模块、缩小范围、返回部分结果还是转人工复核；
    7. `StateGuard（状态保护）`：阻断非法状态跳转、循环重放、重复消费和脏快照写入；
    8. `ErrorSnapshotBuilder（错误快照构建）`：生成前端可读、运维可用、审计可回放的统一错误视图。

### 2.2 处理顺序约束

    错误处理顺序必须固定，不能乱序执行。建议严格遵守以下顺序：
    1. 先捕获原始错误并生成原始异常记录；
    2. 再做错误分类与严重度判定；
    3. 再判断是否允许自动重试；
    4. 再判断是否需要补偿写入或回放；
    5. 再判断是否需要降级、隔离或升级Review；
    6. 最后更新任务状态、任务快照和前端展示层。

    禁止的做法包括：
    1. 未落错误记录就直接重试；
    2. 未判断错误类别就统一指数退避；
    3. 写库失败后只记日志不做补偿；
    4. 状态机异常时仍继续派发下游agent；
    5. 前端只看到“失败”，却看不到失败阶段、失败原因和处理动作。

## 3.错误分类与分级设计

### 3.1 `error_record（错误记录）`

    所有错误都必须统一沉淀为 `error_record（错误记录）`，作为后续重试、补偿、隔离和审计的唯一依据。

| 参数名 | 是否必填 | 参数注释 | 示例 |
| --- | --- | --- | --- |
| error_id | 必填 | 错误唯一标识 | `err_20260408_0001` |
| task_id | 必填 | 所属任务ID | `task_20260408_0001` |
| stage | 必填 | 当前所处阶段 | `ANALYZING` |
| agent_type | 可以缺省 | 触发错误的agent类型 | `js_semantic` |
| error_domain | 必填 | 错误域，如input、tool、agent、state、data、security | `tool` |
| error_code | 必填 | 结构化错误码 | `HTTP_FETCH_TIMEOUT` |
| severity | 必填 | 严重度级别 | `high` |
| retryable | 必填 | 是否允许自动重试 | `true` |
| degradable | 必填 | 是否允许降级继续 | `true` |
| quarantine_required | 必填 | 是否需要进入隔离 | `false` |
| root_cause | 必填 | 根因摘要 | `页面抓取超时，未拿到HTML响应` |
| affected_scope | 必填 | 受影响范围，如单agent、单阶段、全任务 | `单agent` |
| cause_event_ref | 可以缺省 | 引发错误的上游事件引用 | `event://obs/8891` |
| evidence_ref | 可以缺省 | 相关证据引用 | `evidence://http/4002` |
| retry_count | 必填 | 当前已重试次数 | `1` |
| handling_action | 必填 | 当前已执行的处理动作 | `retry_scheduled` |
| created_at | 必填 | 错误记录创建时间 | `2026-04-08T13:20:05+08:00` |

### 3.2 错误域分类

    建议将总控agent中的错误固定分为以下六类：
    1. `input_error（输入错误）`：目标为空、域名/IP格式非法、范围不清、任务契约字段缺失；
    2. `tool_error（工具错误）`：HTTP请求超时、外部API失败、扫描器崩溃、AST服务不可用；
    3. `agent_error（agent错误）`：输出结构不合法、上下文不足、推理中止、自检失败；
    4. `state_error（状态错误）`：非法状态跳转、重复消费、事件顺序错乱、循环触发；
    5. `data_error（数据错误）`：写库失败、对象存储失败、快照构建失败、引用断裂；
    6. `security_error（安全边界错误）`：任务越权、范围漂移、禁用模块被误触发、异常对象超限。

    分类目的不是为了“写得好看”，而是为了绑定不同的恢复动作：
    1. 输入错误优先终止或返回澄清；
    2. 工具错误优先重试与补偿；
    3. agent错误优先自检修复与Review；
    4. 状态错误优先隔离与阻断扩散；
    5. 数据错误优先保证事件不丢并进行补偿回放；
    6. 安全边界错误优先停止派发并进入隔离或人工确认。

### 3.3 严重度分级

    建议将错误严重度统一分为四级：
    1. `low`：不影响主链路，可记录后继续执行，例如单条低价值证据解析失败；
    2. `medium`：影响单agent或单子链路，需要重试或降级，例如某一HTML拉取失败；
    3. `high`：影响当前阶段完整性，需要暂停派发、补偿或升级Review，例如图谱写入失败；
    4. `critical`：可能导致范围越界、状态污染或任务不可恢复，必须隔离，例如非法状态跳转、重复循环失控。

    严重度不是由异常类型单独决定，而应综合以下因素：
    1. 是否影响主链路；
    2. 是否影响证据可追溯性；
    3. 是否可通过补跑恢复；
    4. 是否存在数据污染风险；
    5. 是否可能违反任务边界或安全约束。

## 4.重试、补偿与降级设计

### 4.1 `retry_policy（重试策略）`

    自动重试必须结构化，不能写死在各agent内部。建议统一维护 `retry_policy（重试策略）`。

| 参数名 | 是否必填 | 参数注释 | 示例 |
| --- | --- | --- | --- |
| error_domain | 必填 | 适用错误域 | `tool` |
| error_code | 可以缺省 | 适用错误码 | `HTTP_FETCH_TIMEOUT` |
| max_retry | 必填 | 最大重试次数 | `3` |
| backoff_strategy | 必填 | 退避策略 | `exponential_with_jitter` |
| retry_window_seconds | 必填 | 允许重试的时间窗口 | `300` |
| self_repair_first | 必填 | 是否先做一次自修复再重试 | `false` |
| escalate_after_retry | 必填 | 重试耗尽后的动作 | `degrade_or_review` |
| idempotency_required | 必填 | 是否要求幂等校验 | `true` |

    推荐规则如下：
    1. `input_error` 默认不自动重试，直接返回澄清或失败；
    2. `tool_error` 可做指数退避重试，但必须限制次数和窗口；
    3. `agent_error` 优先执行一次 `self_check -> self_repair`，再决定是否重试；
    4. `state_error` 禁止盲重试，必须先冻结状态并检查事件序；
    5. `data_error` 可对幂等写入做补偿重放，但禁止直接重复下游副作用调用；
    6. `security_error` 默认不自动重试，优先隔离。

### 4.2 `compensation_action（补偿动作）`

    当错误发生在“事件已生成但状态未更新”或“状态已推进但快照未刷新”等半成功场景时，必须使用补偿动作而不是简单回滚。

| 参数名 | 是否必填 | 参数注释 | 示例 |
| --- | --- | --- | --- |
| compensation_id | 必填 | 补偿动作ID | `comp_20260408_0031` |
| task_id | 必填 | 所属任务ID | `task_20260408_0001` |
| source_error_id | 必填 | 来源错误ID | `err_20260408_0001` |
| compensation_type | 必填 | 补偿类型，如snapshot_rebuild、event_replay、write_backfill | `snapshot_rebuild` |
| target_ref | 必填 | 补偿目标对象引用 | `snapshot://task/20260408_0001` |
| idempotency_key | 必填 | 幂等键 | `comp:task_20260408_0001:snapshot_rebuild:v1` |
| status | 必填 | 当前状态 | `pending` |
| scheduled_at | 必填 | 调度时间 | `2026-04-08T13:21:10+08:00` |

    建议优先支持以下补偿类型：
    1. `event_replay`：事件落库成功但状态未推进时重放事件；
    2. `snapshot_rebuild`：快照构建失败时按事件流重建快照；
    3. `write_backfill`：对象存储或关系库写入局部失败时补写；
    4. `link_repair`：证据引用丢失时补建引用关系。

### 4.3 `degradation_plan（降级计划）`

    非核心模块失败时，总控agent必须优先尝试降级执行，而不是直接终止全任务。建议统一使用 `degradation_plan（降级计划）`。

| 参数名 | 是否必填 | 参数注释 | 示例 |
| --- | --- | --- | --- |
| degrade_id | 必填 | 降级计划ID | `deg_20260408_0012` |
| task_id | 必填 | 所属任务ID | `task_20260408_0001` |
| disabled_agent_types | 必填 | 被临时禁用的agent列表 | `["js_semantic"]` |
| capability_loss | 必填 | 缺失能力说明 | `应用层接口解析缺失` |
| fallback_path | 必填 | 替代执行路径 | `继续执行fingerprint -> graph -> report` |
| output_annotation | 必填 | 需要暴露给前端和报告的标记 | `报告中标记“JS解析未完成”` |
| review_required | 必填 | 是否要求进入复核 | `false` |

    典型降级策略建议如下：
    1. `JS Semantic Agent` 不可用时，继续基础发现、指纹识别、图谱与基础报告；
    2. `Value Assessment Agent` 不可用时，返回未排序发现列表并标注待补评估；
    3. `Graph & Topology Agent` 局部失败时，先返回列表视图和证据视图，暂停图谱块；
    4. `Report Agent` 失败时，先返回可审计快照与结构化结论，允许后续补生成报告。

### 4.4 `PARTIAL_DONE（部分完成）` 与 `QUARANTINED（隔离）`

    建议总控agent将两类结果严格区分：
    1. `PARTIAL_DONE`：主链路可交付，部分能力缺失但结论仍可审计；
    2. `QUARANTINED`：存在状态污染、越界风险、严重冲突或循环失控，必须停止自动扩散。

    判定建议如下：
    1. 非关键agent失败且已有足够证据支持当前输出，可进入 `PARTIAL_DONE`；
    2. 关键证据丢失、结论互相冲突且无法自动合并时，进入 `QUARANTINED`；
    3. 任何 `security_error` 默认优先考虑 `QUARANTINED`；
    4. 进入 `QUARANTINED` 后，禁止继续自动派发新的扩展型agent任务。

## 5.状态恢复与回放设计

### 5.1 错误相关状态流转

    总控agent的错误处理必须和状态机强绑定。建议补充以下错误相关状态流转规则：
    1. `RUNNING -> RETRYING`：命中可重试错误，且未超过重试策略上限；
    2. `RUNNING -> PARTIAL_DONE`：非核心链路失败，但主链路已闭环；
    3. `RUNNING -> QUARANTINED`：检测到高风险冲突、越界或状态污染；
    4. `RUNNING -> FAILED`：关键链路失败且无可用补偿、降级或人工接管路径；
    5. `QUARANTINED -> REVIEWING`：复核模块或人工开始介入；
    6. `REVIEWING -> RETRYING`：复核后允许补跑；
    7. `RETRYING -> RUNNING`：重试任务被重新领取并开始执行；
    8. `FAILED -> RETRYING`：仅允许在人工确认或明确恢复计划存在时发生。

### 5.2 `recovery_plan（恢复计划）`

    为避免“知道失败了，但不知道怎么恢复”，建议在错误进入 `high` 或 `critical` 时生成 `recovery_plan（恢复计划）`。

| 参数名 | 是否必填 | 参数注释 | 示例 |
| --- | --- | --- | --- |
| recovery_id | 必填 | 恢复计划ID | `rec_20260408_0004` |
| task_id | 必填 | 所属任务ID | `task_20260408_0001` |
| source_error_id | 必填 | 来源错误ID | `err_20260408_0008` |
| resume_from_state | 必填 | 恢复起点状态 | `DISCOVERING` |
| required_actions | 必填 | 恢复前置动作列表 | `["event_replay","snapshot_rebuild"]` |
| blocked_actions | 必填 | 恢复前禁止动作 | `["new_agent_dispatch"]` |
| approval_required | 必填 | 是否需要Review或人工批准 | `true` |
| recovery_status | 必填 | 当前恢复状态 | `waiting_review` |

### 5.3 回放与幂等约束

    错误恢复过程中，最容易出现的问题不是“恢复不了”，而是“恢复时重复执行”。因此必须固定以下约束：
    1. 所有重试和补偿动作都必须带 `idempotency_key`；
    2. `ObservationEvent` 重放前必须先校验 `dedupe_key`；
    3. 不允许对已经产生外部副作用的高风险工具调用做无保护重复执行；
    4. 快照重建优先使用事件流回放，而不是重新推测当前状态；
    5. 恢复后必须重新生成 `TaskSnapshot` 和错误处置日志，确保前端看到的是新状态。

## 6.前端输出、审计与运维设计

### 6.1 `task_error_snapshot（任务错误快照）`

    前端不应直接消费底层异常堆栈，而应读取总控agent聚合后的 `task_error_snapshot（任务错误快照）`。

| 参数名 | 是否必填 | 参数注释 | 示例 |
| --- | --- | --- | --- |
| task_id | 必填 | 任务ID | `task_20260408_0001` |
| current_state | 必填 | 当前任务状态 | `PARTIAL_DONE` |
| current_stage | 必填 | 当前阶段 | `REPORTING` |
| latest_error_code | 可以缺省 | 最近一次关键错误码 | `JS_AST_SERVICE_UNAVAILABLE` |
| latest_error_summary | 可以缺省 | 最近一次关键错误摘要 | `JS语义解析服务不可用，已降级继续执行` |
| latest_handling_action | 必填 | 最近一次处置动作 | `degraded_continue` |
| retry_status | 必填 | 重试状态 | `exhausted` |
| degraded_modules | 可以缺省 | 已降级模块列表 | `["js_semantic"]` |
| review_required | 必填 | 是否待复核 | `false` |
| user_visible_impact | 必填 | 用户可见影响描述 | `报告不包含应用层API提取结果` |
| updated_at | 必填 | 更新时间 | `2026-04-08T13:24:30+08:00` |

### 6.2 面向前端的展示规则

    前端展示错误时，建议固定以下原则：
    1. 展示“失败阶段 + 错误摘要 + 当前动作 + 是否还能继续”；
    2. 区分“正在自动恢复”“已降级继续”“等待复核”“已失败终止”；
    3. 对 `PARTIAL_DONE` 任务明确展示缺失能力，而不是伪装成完整完成；
    4. 对 `QUARANTINED` 任务明确展示“已隔离，暂停自动扩展”；
    5. 仅调试视图展示详细错误码、事件引用和补偿动作，普通视图不暴露底层堆栈。

### 6.3 审计与运维指标

    错误处理模块至少要沉淀以下可观测指标：
    1. 按错误域统计的错误次数；
    2. 各agent平均重试次数与重试成功率；
    3. 降级任务占比与部分完成占比；
    4. 隔离任务数量与解除隔离耗时；
    5. 快照补偿成功率、事件重放成功率、写入补偿耗时；
    6. 因状态错乱、重复消费、越界命中而触发阻断的次数。

### 7.5 落地约束

    为保证错误处理长期稳定，工程上应固定以下约束：
    1. 任何自动重试前必须先落 `error_record（错误记录）`；
    2. 禁止所有错误共用同一重试逻辑；
    3. 禁止 `state_error` 和 `security_error` 直接自动续跑；
    4. 禁止在未做幂等校验的情况下重复派发补偿或重放事件；
    5. 禁止把 `PARTIAL_DONE` 伪装成 `DONE`；
    6. 禁止 `QUARANTINED` 状态下继续自动扩展目标范围；
    7. 任何降级都必须同步写入任务快照、报告注释和审计日志；
    8. 快照、状态、事件任一写入失败时，必须有补偿动作而不能只打日志；
    9. 恢复执行必须基于 `recovery_plan（恢复计划）`，不能从头盲跑；
    10. 最终对外输出前，若仍存在未处理的 `critical` 错误，则必须阻止任务进入 `DONE`。
