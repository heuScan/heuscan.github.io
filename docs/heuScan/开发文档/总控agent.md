# 一、agrnt任务描述

    作为项目的核心总控agent，需要承担对接用户输入输出，拆解任务，处理错误等核心功能，并且具体调用那一部分agent，也需要该总控agent控制；因此该agent处于绝对的核心。

    使用PLAN范式设计总控agent：需要在开始任务之前，明确用户目标，以及信息收集的边界；同时明确用户信息收集的目的（例如，需求仅为收集企业信息等，则不需要启动诸如JS解析等agent，节省Token）；
    使用框架：
    引入状态机：在plan确定后，需要生成状态管理（分布任务），用于管理整个项目的生命周期。

# 二、具体功能拆解

## 1.用户输入与任务建模

### 1.1 输入接收与标准化

    总控agent首先只接收统一输入，不直接把前端原始表单透传给下游agent。这里要先完成输入标准化，生成统一的TaskCommand。标准化内容至少包括：企业名称标准化、域名去协议与去路径、IP/CIDR格式校验、URL拆分、批量目标去重、别名归并、用户意图标签化（企业情报/资产发现/应用分析/图谱聚合/价值评估/报告输出）。

    工程上建议把输入拆成三层：原始输入raw_input、标准化输入normalized_input、任务契约task_contract。raw_input只做留痕；normalized_input作为后续状态机和路由的唯一输入；task_contract则补充范围、约束、优先级、预算、风险等级，成为整个任务生命周期的固定锚点。

### 1.2 澄清机制与默认策略

    总控agent不是对所有不确定项都追问，而是只对会影响执行边界的问题做澄清。建议将字段分成两类：
    1. 必须澄清项：目标主体不明确、扫描对象类型不明确、收集边界不明确、输出目标不明确；
    2. 可默认项：并发等级、输出风格、非关键模块开关、报告详细程度。

    如果命中必须澄清项，总控agent应中断plan生成，返回结构化澄清问题；如果仅缺少可默认项，则写入默认策略继续执行。默认策略建议固化为profile，如enterprise_only、surface_scan、deep_app_scan、graph_first、report_first，避免每次由LLM自由发挥。

### 1.3 任务契约生成

    总控agent需要在任务开始前生成可持久化任务契约，至少包含：task_id、目标集合、目标类型、执行目的、启用agent列表、禁用agent列表、最大并发、最大token预算、证据保留策略、允许的状态跳转、人工复核阈值、最终输出格式。后续所有agent都只能在该契约范围内执行，不能自行扩大范围。

## 2.返回输出与结果聚合

### 2.1 输出不是直接返回报告文本

    总控agent返回前端的内容不能只是报告生成agent的原始文本，而应统一经Result Aggregator组装成TaskSnapshot。TaskSnapshot至少包含：全局状态、当前阶段、阶段摘要、关键发现、图谱摘要、价值排序、待复核项、最终报告引用、证据引用、错误与告警列表。

### 2.2 面向前端的输出分层

    返回前端建议分三层：
    1. 实时层：任务进度、当前运行agent、最近事件、失败重试信息；
    2. 分析层：发现列表、置信度、证据片段、关联实体、图谱快照；
    3. 产物层：报告正文、报告块结构、导出链接、审计回放入口。

    前端不直接消费底层ObservationEvent，而是消费总控agent输出的聚合视图，避免前端自己拼状态导致不一致。

### 2.3 渲染安全与可回放

    返回输出时要同时保存render_payload和source_refs。render_payload用于前端渲染，source_refs用于回放和审计；任意前端展示块都必须能追溯到证据引用或上游事件。这样既能保证报告展示稳定，也能支持后续“某结论为何出现”的排查。

## 3.任务分析拆解与能力路由

### 3.1 规划阶段职责

    总控agent在plan阶段要把任务转换成可执行的阶段图，而不是简单列出agent名单。建议固定映射到项目文档中的状态机阶段：SCOPED -> DISCOVERING -> ANALYZING -> EXPANDING -> CORRELATING -> PRIORITIZING -> REPORTING -> REVIEWING。

### 3.2 能力路由规则

    总控agent应维护一份显式的能力路由表，而不是靠prompt临场判断。例如：
    1. 企业主体、股东、备案、历史变更类需求，路由到Enterprise Intel Agent；
    2. 子域、IP、端口、路径发现类需求，路由到Asset Discovery Agent；
    3. 页面技术栈与中间件识别，路由到Fingerprint Agent；
    4. JS接口、参数、隐藏路径、外联域名提取，路由到JS Semantic Agent；
    5. 图关系拼装与拓扑构建，路由到Graph & Topology Agent；
    6. 资产优先级、价值排序，路由到Value Assessment Agent；
    7. 最终对外可读产物，路由到Report Agent；
    8. 高风险、低置信或冲突结论，升级到Review Agent。

### 3.3 跳过、补跑与动态加派

    总控agent必须支持三类调度决策：
    1. 跳过：如任务仅为企业情报收集，则不启动JS Semantic Agent和Passive Traffic Audit Agent；
    2. 补跑：如资产发现阶段发现大量JS资源，再补跑JS Semantic Agent；
    3. 加派：如JS解析发现新域名、接口或文件线索，则触发Dynamic Dictionary Agent、File Intel Agent或Chain Hint Agent。

    这部分要基于显式规则和事件触发条件实现，不能完全依赖LLM自由决定，否则很容易造成链路不可控和token浪费。

### 3.4 依赖图与执行顺序

    总控agent内部建议维护DAG而不是单一串行列表。典型依赖关系如下：Enterprise Intel与Asset Discovery可并行；Fingerprint依赖基础HTTP结果；JS Semantic依赖页面/JS采集结果；Graph & Topology依赖实体与发现沉淀；Value Assessment依赖图谱和发现；Report依赖前述摘要；Review插在高风险分支和最终产出前。这样既能提高吞吐，也便于失败隔离。

## 4.状态机与状态存储

### 4.1 全局状态存储模型

    总控agent不能只把“当前状态”写进一张表，而要同时保存当前态和状态流转历史。建议至少拆分以下对象：
    1. tasks：任务主表，保存task_contract、当前状态、当前阶段；
    2. task_state_transitions：状态跳转流水，记录from_state、to_state、trigger_event、operator；
    3. agent_runs：每个agent执行实例，记录开始时间、结束时间、输入摘要、输出摘要、token消耗、失败原因；
    4. observation_events：标准事件表，作为状态推进和回放依据；
    5. task_snapshots：前端查询快照，避免每次现算；
    6. context_capsules：阶段摘要与上下文压缩结果；
    7. review_tickets：复核工单；
    8. evidence_refs：证据对象索引。

### 4.2 存储职责划分

    PostgreSQL适合保存任务、事件、状态流转、报告元数据；Redis适合保存队列、短期锁、去重键、心跳、短时上下文缓存；MinIO适合保存HTML、JS、流量包、文件、报告原文；Neo4j适合保存实体图与拓扑图。总控agent只通过统一repository写入，不允许各agent自己决定往哪存、怎么存。

### 4.3 状态一致性策略

    当前状态建议采用“事件驱动更新 + 快照查询”模式：状态流转以append-only事件为准，tasks.current_state和task_snapshots仅作为查询加速层。发生异常时，可以根据task_state_transitions和observation_events重建最新状态，避免因为缓存或单次写入失败导致任务不可恢复。

## 5.多agent异步调用、调度与并发控制

### 5.1 异步调度基本模型

    总控agent不直接同步串行调用所有agent，而是生成AgentTask并交给AgentExecutor。每个AgentTask至少包含：task_id、stage、agent_type、input_ref、context_ref、priority、retry_policy、deadline、parent_event_id。执行结果统一回写ObservationEvent，再由总控agent消费事件推进状态机。

### 5.2 任务领取与租约机制

    为防止重复执行，建议对每个AgentTask使用claim lease机制：执行器领取任务时写入lease_owner、lease_expire_at，并定期心跳续租；超时未续租则回到待执行队列。这样即使worker异常退出，也不会出现任务永久卡死。

### 5.3 并发控制

    并发至少分三层控制：
    1. 任务级并发：单个task允许同时运行多少个agent；
    2. agent级并发：某类agent的全局并发上限，避免高成本模块拖垮系统；
    3. 目标级并发：同一域名、同一IP或同一证据对象的并发限制，防止重复抓取和热点拥塞。

    默认建议为：企业情报模块低并发、基础发现中并发、JS解析与文件抽取低并发、图谱与报告单任务单实例。具体数值不写死在prompt，而写在runtime配置中。

### 5.4 去重、幂等与防循环

    多agent协同时最容易出现事件风暴和重复调用，所以总控agent必须内置幂等控制。建议每个ObservationEvent都生成dedupe_key，构成方式可为task_id + agent_type + normalized_target + event_type + content_hash。若同一个dedupe_key在有效窗口内已被消费，则只更新计数，不重复触发下游。

    对于容易形成闭环的链路，如“JS解析发现新路径 -> 扩展词典生成新词 -> 再次基础发现 -> 再次触发JS解析”，必须增加最大展开深度、最小增益阈值和阶段循环次数上限。若新增信息增益低于阈值，应停止扩展并进入下一阶段。

### 5.5 暂停、恢复与人工介入

    总控agent要支持任务级pause/resume/retry。pause时不再下发新任务，但保留已运行任务结果；resume时从最近一个稳定状态重新调度，不从头执行；retry分为agent级retry和阶段级retry，避免因为单个子任务失败导致整任务完全重跑。

## 6.上下文装配与上下文剪裁

### 6.1 上下文不是“把所有历史拼进去”

    总控agent的上下文装配必须先分层再裁剪。建议固定为五层：
    1. 不可裁剪层：task_contract、范围约束、当前阶段目标、明确禁用项；
    2. 核心事实层：已确认实体、关键资产、已验证结论、Review结论；
    3. 当前工作层：本阶段待解决问题、最近上游摘要、直接依赖事件；
    4. 证据片段层：与当前agent最相关的HTML/JS/文件/流量/图谱片段；
    5. 冗余历史层：旧日志、低置信发现、重复片段、细粒度执行日志。

    裁剪顺序必须从第5层往上删，前3层原则上只允许压缩，不允许直接删除。

### 6.2 token预算计算标准

    上下文裁剪必须以模型窗口为硬约束。建议总控agent维护model_registry，记录不同模型的最大上下文窗口W。每次组装上下文时按以下方式预留预算：
    1. 输出预算reserved_output = max(预估输出token, W * 15%)；
    2. 系统提示与工具协议预算reserved_system = W * 10%；
    3. 安全缓冲reserved_safety = W * 10%；
    4. 可用上下文budget_context = W - reserved_output - reserved_system - reserved_safety。

    若budget_context为负，则必须降低任务粒度、拆分任务或切换更大窗口模型，不能强塞。工程上建议把实际注入上下文控制在W的65%~70%，75%作为硬上限，超过即触发剪裁或拆包。

### 6.3 裁剪评分标准

    对证据与历史事件不应按时间粗暴截断，而应按综合分排序后保留。建议评分维度如下：
    score = 相关性 * 0.35 + 置信度 * 0.25 + 新鲜度 * 0.15 + 依赖度 * 0.15 + 多样性收益 * 0.10。

    其中：
    1. 相关性：与当前agent目标是否直接相关；
    2. 置信度：上游agent或规则打分是否可靠；
    3. 新鲜度：是否来自最近阶段、是否已被后续结论覆盖；
    4. 依赖度：是否是当前推理链的必要前置；
    5. 多样性收益：是否提供新的实体、接口、路径或异常模式。

    重复内容要先做归并再打分，避免同类HTML片段、相似JS函数或重复事件占满token。

### 6.4 不同内容类型的分块标准

    为保证上下文连续性，裁剪之前要先按内容类型做结构化分块：
    1. HTML：按DOM区块、script块、form块或接口相关区域分块，每块建议400~800 token，重叠80~120 token；
    2. JS：按文件模块、函数、对象或AST节点簇分块，每块建议300~600 token，重叠50~100 token；
    3. 文本文件：按标题/段落分块，每块建议500~1000 token；
    4. 流量日志：按会话或请求链分块，保持request/response成对；
    5. 事件历史：按阶段和实体聚合，不按单条日志逐条注入。

    这样可以避免“从中间截断一个函数”或“把同一请求链拆开”导致语义断裂。

### 6.5 保证上下文连续性的机制

    为了在裁剪后仍保证连续性，建议总控agent在每个阶段结束后生成Context Capsule，作为下一阶段的稳定输入锚点。一个Context Capsule至少包含：
    1. 本阶段200~500 token摘要；
    2. 已确认事实列表；
    3. 待验证假设列表；
    4. 关键实体ID与证据ID；
    5. 下阶段建议动作；
    6. 已排除路线与禁止重复工作项。

    下游agent优先读取capsule，再按需附加少量原始证据片段。这样即便原始历史非常长，也能保持语义连续，而不是完全依赖长上下文记忆。

### 6.6 超预算时的降级策略

    当上下文仍然超预算时，建议按以下顺序降级：
    1. 删除低分冗余事件；
    2. 用阶段摘要替代原始执行日志；
    3. 将长证据替换为“摘要 + 关键片段 + 对象引用”；
    4. 将任务拆分成更小的AgentTask；
    5. 必要时切换到更大窗口模型。

    禁止的做法是：直接截断最后N个token后发送给模型，因为这会造成证据断裂、JSON结构损坏和上下游推理不一致。

## 7.复核、质量控制与人工介入

### 7.1 自动复核触发条件

    总控agent应自动将以下情况升级到Review Agent：高价值资产判定但证据不足、多个agent对同一实体给出冲突结论、链路提示带有较强推断性质、报告中出现高风险表述、同一发现置信度波动过大。

### 7.2 输出前质量闸门

    在进入REPORTING和REVIEWING前，总控agent应做统一质量检查：字段完整性校验、证据可追溯校验、置信度下限校验、重复发现归并校验、图谱节点孤立度检查、报告块渲染校验。任何一项失败都不能直接进入最终输出，而应回退到对应阶段补算或发起复核。

### 7.3 人工介入点设计

    即使一期不做复杂人工工作台，也要预留人工介入点，包括：手动确认任务范围、手动忽略误报、手动确认图谱合并、手动触发补跑、手动批准最终报告。这些介入动作也要写入状态流转和审计日志。

## 8.错误处理与降级恢复

### 8.1 错误分类

    总控agent建议把错误分为五类：
    1. 输入错误：参数不合法、目标不可解析、范围不清；
    2. 工具错误：扫描器超时、外部API失败、解析器异常；
    3. agent错误：输出格式不合法、上下文不满足、推理失败；
    4. 状态错误：非法状态跳转、重复消费、事件顺序错乱；
    5. 数据错误：写库失败、对象丢失、快照构建失败。

### 8.2 重试与补偿策略

    不同错误不能统一重试。建议：工具类错误采用指数退避重试；agent格式错误先进入自检修复一次，再失败则升级Review或标记FAILED；状态错误禁止盲目重试，应进入隔离队列；数据写入错误要优先保证事件不丢，可先落本地补偿日志或补偿表，随后异步回放。

### 8.3 降级策略

    当某一agent不可用时，总控agent应支持降级而不是整任务直接失败。例如：JS Semantic Agent不可用时，仍可继续资产发现、指纹识别、图谱构建和基础报告，但在TaskSnapshot中标记“应用层解析缺失”；Value Assessment不可用时，可先返回未排序发现列表并标记待补评估。

### 8.4 部分完成与隔离

    如果非核心分支失败且不影响主链路，可进入PARTIAL_DONE；如果发现异常循环、严重冲突、可疑数据污染，则进入QUARANTINED，停止自动扩散，由Review或人工确认后再恢复。这样可以保证系统在复杂任务下仍具备可控失败能力。

## 9.观测性、安全边界与工程治理补充

### 9.1 可观测性

    总控agent至少要输出四类观测数据：状态流转日志、agent执行指标、工具调用记录、token与成本统计。核心指标包括：任务总耗时、各阶段耗时、各agent成功率、重试次数、上下文大小、平均token消耗、快照构建延迟、队列积压量。

### 9.2 审计与回放

    审计不是最终报告的附属功能，而是总控agent的基础职责。每个最终结论都要能回溯到ObservationEvent、EvidenceRef、AgentRun和StateTransition；每次重试、跳过、补跑、人工干预都必须留痕。这样后续做问题追查、质量评估和模型调优时才有数据基础。

### 9.3 安全边界

    总控agent要严格受task_contract约束，只进行授权范围内的信息收集与分析。工程上应显式校验目标范围、禁用越权扩展、限制高风险工具调用，并对外部对象下载、文件抽取和流量样本处理设置大小上限与类型白名单，防止任务被异常输入拖垮。
