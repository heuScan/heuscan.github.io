# Graph & Topology Agent 

## 1. 模块定位与核心职责

### 1.1 模块定位

Graph & Topology Agent 是 heuScan 聚合后端 **Agent Runtime Layer** 中的一个核心 Specialist Agent。它在系统中处于 **统一事实底座** 的关键位置，负责将上游 Agent 产生的离散、异构的数据（实体、资产、线索、证据）转化为结构化、可关联、可查询的统一图谱。

### 1.2 核心职责

| 职责 | 说明 |
| :--- | :--- |
| **统一图谱构建** | 接收标准化输入，构建包含 `Asset`、`Entity`、`Vulnerability`、`Evidence` 等节点的属性图。 |
| **拓扑关系建立** | 基于线索和证据建立 `BELONGS_TO`、`RESOLVES_TO`、`CONNECTS`、`REFERENCES` 等关系边。 |
| **数据聚合去重** | 确保相同资产或实体在图中唯一，采用幂等写入策略。 |
| **事实底座维护** | 作为 Neo4j 图数据库的唯一写入入口（遵循 Tool Adapter 层规范），保证图谱数据的一致性和可追溯性。 |

### 1.3 技术范式与栈

| 项目 | 内容 |
| :--- | :--- |
| **范式** | PLAN (Process-Logic-Action-Node) |
| **技术栈** | Python 3.10+ + Neo4j 5.x (通过官方驱动 `neo4j` 交互) |
| **依赖层** | Tool Adapter Layer (图谱写入适配器) |

## 3. 输入与输出契约 (PLAN 模型)

### 3.1 输入 (Inputs)
Graph & Topology Agent 不主动扫描，仅消费 **ObservationEvent**。输入数据结构定义为 `GraphConstructionContext`。

| 输入类型 | 字段示例 | 来源 Agent | 目标节点类型 |
| :--- | :--- | :--- | :--- |
| **实体 (Entity)** | `domain`, `ip`, `cert_serial`, `org_name` | Enterprise Intel, Fingerprint | `Domain`, `IPAddress`, `Certificate`, `Organization` |
| **资产 (Asset)** | `url`, `host`, `port`, `service`, `title` | Asset Discovery, Fingerprint | `WebAsset`, `Host`, `Port`, `Service` |
| **线索 (Clue)** | `ip_to_domain_resolve`, `link_href` | JS Semantic, Passive Traffic | `RESOLVES_TO`, `LINKS_TO` |
| **证据 (Evidence)** | `screenshot_path`, `js_snippet`, `header_hash` | Asset Discovery, File Intel | `Evidence` (节点) 关联到具体资产 |

### 3.2 输出 (Outputs)
- **图谱节点 (Nodes)**：符合 Neo4j 标签规范的实体与资产。
- **拓扑边 (Edges)**：带有 `confidence` (置信度) 和 `evidence_ref` (证据引用) 属性的关系。

### 3.3 PLAN 处理流程
1. **Process (预处理)**：校验输入字段，提取唯一标识符 (如 `ip` 去重，`domain` 标准化)。
2. **Logic (逻辑)**：根据 `Clue` 类型推断关系方向，进行节点去重合并逻辑。
3. **Action (动作)**：调用 `GraphWriteAdapter` 执行 Cypher `MERGE` 语句。
4. **Node (节点管理)**：返回生成的图节点 ID 供下游 Agent (如 Value Assessment) 引用。

## 4. 详细设计

### 4.1 类与模块结构
```python
# 路径: backend/agents/specialist/graph_topology_agent.py

class GraphTopologyAgent(BaseAgent):
    """聚合实体、资产、线索、证据，构建统一图谱"""
    
    def __init__(self, graph_adapter: GraphWriteAdapter):
        self.adapter = graph_adapter
        self.agent_name = "GraphTopologyAgent"
        
    def execute(self, context: AgentContext) -> AgentResult:
        # 1. 从上下文提取构造指令
        construction_plan = self._build_plan(context.events)
        
        # 2. 执行图谱写入事务
        with self.adapter.session() as session:
            for node in construction_plan.nodes:
                session.merge_node(node)
            for rel in construction_plan.relationships:
                session.merge_relationship(rel)
                
        # 3. 产生完成事件
        return AgentResult(
            events=[ObservationEvent(type="GRAPH_UPDATED", data=construction_plan.summary())]
        )
```

### 4.2 核心数据模型 (Neo4j Schema 设计)

#### 节点标签定义
| 标签 | 关键属性 | 说明 |
| :--- | :--- | :--- |
| `:Asset` | `asset_id`, `type`, `value`, `first_seen`, `last_seen` | 资产超类 |
| `:IPAddress` | `address` | 继承 Asset |
| `:Domain` | `name`, `registered_domain` | 继承 Asset |
| `:Port` | `number`, `protocol` | 端口节点 |
| `:Service` | `name`, `version`, `banner` | 服务节点 |
| `:Evidence` | `evidence_id`, `type`, `storage_path`, `hash` | 证据文件引用 |
| `:Organization` | `name`, `country` | 企业/组织实体 |

#### 关系类型定义
| 关系类型 | 方向 | 属性 | 语义 |
| :--- | :--- | :--- | :--- |
| `[:HAS_PORT]` | Asset -> Port | `discovery_method` | 资产开放端口 |
| `[:RUNS_SERVICE]` | Port -> Service | `confidence` | 端口运行服务 |
| `[:RESOLVES_TO]` | Domain -> IPAddress | `last_resolved`, `record_type` | DNS 解析 |
| `[:REFERENCES]` | Asset/Entity -> Evidence | `evidence_type` | 关联证据文件 |
| `[:BELONGS_TO]` | Asset -> Organization | `source` | 资产归属组织 |
| `[:CONNECTS_TO]` | Asset -> Asset | `protocol`, `port` | 资产间网络连接 |

### 4.3 关键算法：幂等去重与合并
由于不同 Agent 可能上报同一 IP 的不同端口信息，Agent 必须执行智能合并：

1. **IP 节点唯一性**：基于 `address` 属性 MERGE。
2. **端口冲突解决**：若 `Asset Discovery` 发现 80 端口为 HTTP，而 `Fingerprint` 识别为 Nginx，则更新 `Service` 节点的 `version` 并追加 `banner`，而非创建新节点。
3. **证据链绑定**：所有创建的关系必须附带 `evidence_ref` 字段，指向具体的 `ObservationEvent.id`，确保前端可追溯结论来源。

## 5. 与 Orchestrator 的协作流程

### 5.1 触发时机
Orchestrator 在收到以下任一 **状态信号** 时，将触发 Graph & Topology Agent 进行增量/全量构建：
- `AssetDiscoveryAgent` 完成第一阶段端口扫描。
- `FingerprintAgent` 完成服务识别。
- `EnterpriseIntelAgent` 完成 WHOIS/ICP 查询。
- **条件**：Orchestrator 收集到至少 10 条未入库的 `ObservationEvent` 或任务进入 `:ANALYZING` 阶段。

### 5.2 上下文限制策略
为防止单次构建上下文过大，Agent 采用 **批次构建**：
- **输入**：`AgentContext` 仅包含当前批次的 Event IDs。
- **处理**：Agent 从 PostgreSQL 事件表读取具体数据，处理后标记事件为 `:PROCESSED_FOR_GRAPH`。

## 6. 统一事实底座的查询接口

虽由 Graph & Topology Agent 写入，但读取由 `GraphQueryService` 提供，前端常用查询场景如下：

| 前端场景 | Cypher 查询逻辑 | 对应 API |
| :--- | :--- | :--- |
| **资产拓扑图** | `MATCH (a:Asset {asset_id:$id})-[r]-(b) RETURN *` | `/api/graph/asset/expansion` |
| **解析链追溯** | `MATCH (d:Domain)-[:RESOLVES_TO]->(ip)-[:HAS_PORT]->(p) ...` | `/api/graph/path` |
| **证据清单** | `MATCH (a)-[:REFERENCES]->(e:Evidence) WHERE a.id=$id RETURN e` | `/api/evidence/list` |
