# 一、模块定位与任务描述

企业情报模块（Enterprise Intel Agent）的目标：围绕“目标域名/企业主体”，通过多数据源（ICP、企业信息平台等）获取备案主体、企业画像、关键人员、股权与投资链路、历史变更等信息，并以**可审计证据链**形式写入图谱与结构化结果，供总控agent聚合为任务快照与最终报告。

- 范式：PLAN + REFLECT
- 技术栈：Python

## 1.1 输入目标与典型场景

- 输入线索：公司名、备案线索、外部情报响应
- 输出目标：
  - 备案主体识别（domain → ICP → 主体公司）
  - 企业画像（基础信息/状态/地址/法人/股东/对外投资/主要人员/历史变更等，视数据源能力）
  - 关系拓扑（公司↔公司、公司↔个人、个人↔个人的关系图谱）

## 1.2 边界、约束与默认策略

- 仅用于授权场景下的信息收集、分析与图谱构建，不涉及自动化攻击执行设计。
- 必须受控扩展：默认 `max_depth=2`，并受 `max_nodes/max_edges/time_budget_s` 任一约束；触发上限必须停止扩展并产出“部分结果 + 告警”。
- 人名同名属于高歧义：除非数据源提供稳定标识（如内部 id、证件号脱敏标识等），不得强行跨公司合并为同一自然人；必要时触发复核。
- 不得把推测写成事实：不确定结论输出为“疑似/可能”，并附证据引用与置信度。

# 二、职责拆解（对齐总控文档的要素）

## 2.1 核心职责

1. 主体解析：域名 → ICP → 备案主体公司（或主体类型判定）
2. 企业画像采集：公司 → 基础信息 / 法人 / 股东 / 对外投资 / 任职 / 历史变更（视数据源能力）
3. 关系扩展：围绕“控制/投资/任职”关系递归扩展相关公司与人员（可配置策略）
4. 实体消歧与去重：公司优先按 `credit_code`，人物默认不跨公司合并
5. 图谱写入：Neo4j 写入节点与关系，保证幂等与去重
6. 证据与审计：保留来源、查询时间、原始回包引用（source_refs），支持回放
7. 结构化输出：输出 `companies/persons/relations + evidence + warnings + stats` 供总控聚合

## 2.2 与其他 Agent/模块的职责边界

结合 `docs/heuScan/项目介绍与功能设计/1.项目介绍与解决痛点.md` 与 `docs/heuScan/项目介绍与功能设计/2.第一阶段多Agent功能拆解与协同架构设计.md` 的一期目标（模块化单体 + 编排优先 + 统一数据面），企业情报模块的边界必须明确如下：

- 与“基础发现/资产发现”能力边界：
  - 本模块**不**负责子域名枚举、IP/端口扫描、路径探测、CDN 解析等资产发现工作。
  - 本模块产出的是企业主体与关系；域名/IP/端口等资产实体与其关联关系应由资产发现模块产出，并由图谱聚合层统一关联。
- 与“指纹识别”能力边界：
  - 本模块**不**识别前后端技术栈、中间件、CMS、组件等指纹信息；该类信息由指纹识别模块负责。
- 与“动态扫描字典生成（扩展词典）”能力边界：
  - 本模块**不**直接生成目录/路径/参数字典；它最多提供下游可用的企业别名线索（如规范化公司名、简称、曾用名），由扩展词典模块生成字典并回流到发现/扫描阶段。
- 与“JS 语义解析”能力边界：
  - 本模块**不**解析 JS，不提取接口/参数/外联域名；该类能力由 JS 语义解析模块负责。
- 与“被动流量审查/漏洞线索标记”能力边界：
  - 本模块**不**分析 HTTP 流量、不输出未授权/CORS/漏洞线索；该类能力由流量审查模块负责。
- 与“文件情报”能力边界：
  - 本模块**不**下载文件/抽取文档内容；该类能力由文件情报模块负责。
- 与“网络拓扑结构生成/图谱聚合”能力边界：
  - 本模块写入/输出的图谱仅包含 **Company/Person** 及其工商关系（`LEGAL_PERSON/SHAREHOLDER/INVEST/EMPLOY`）。
  - “网络拓扑结构”（域名→IP→端口→路径/服务）与跨域关系拼装（企业↔域名↔资产）由图谱聚合模块负责统一建模与展示。
- 与“价值评估/攻击建议/报告输出”能力边界：
  - 本模块**不**做资产价值排序、不产出攻击链路建议、不生成最终报告；这些由价值评估模块与报告输出模块负责。
- 与“复核模块”协作边界：
  - 本模块只负责产出 `review_required + warnings + evidence_refs`，**不**做人工复核决策闭环；复核流程由复核模块与总控统一推进。

# 三、与总控Agent的对接契约（模块级输入输出）

> 总控的 `task_contract` 是全局约束；企业情报模块必须在契约范围内执行。

## 3.1 模块输入（建议内部 `EnterpriseIntelCommand`）

```json
{
  "task_id": "uuid",
  "targets": [
    { "type": "domain", "value": "example.com" }
  ],
  "max_depth": 2,
  "limits": {
    "max_nodes": 1000,
    "max_edges": 3000,
    "time_budget_s": 180
  },
  "source_policy": {
    "preferred_sources": ["icp", "aiqicha", "qichacha", "zhanzhang"],
    "allow_paid_sources": true,
    "allow_fallback": true
  },
  "evidence_policy": {
    "store_raw_payload": true,
    "store_source_refs": true
  }
}
```

## 3.2 模块输出（建议内部 `EnterpriseIntelResult`）

```json
{
  "task_id": "uuid",
  "status": "succeeded",
  "companies": [],
  "persons": [],
  "relations": [],
  "graph_delta": { "nodes": [], "edges": [] },
  "evidence": [
    {
      "source": "icp",
      "query": "example.com",
      "fetched_at": "2026-04-08T12:00:00Z",
      "source_ref": "minio://.../raw/icp/xxx.json"
    }
  ],
  "warnings": [],
  "review_required": false,
  "stats": { "depth_reached": 2, "nodes": 120, "edges": 260 }
}
```

## 3.3 字段解释与校验规则

### 3.3.1 `EnterpriseIntelCommand` 字段说明

| 字段 | 类型 | 必填 | 默认值 | 说明 | 校验/约束 |
|---|---:|:---:|---:|---|---|
| `task_id` | string | 是 | - | 任务唯一标识，贯穿全链路（状态、证据、图谱、结构化结果） | 非空；建议 UUID |
| `targets` | array | 是 | - | 输入线索集合（域名/企业/统一社会信用代码） | 至少 1 项；同类去重 |
| `targets[].type` | enum | 是 | - | 线索类型 | `domain` / `company_name` / `credit_code` |
| `targets[].value` | string | 是 | - | 线索值 | `domain`：去协议去路径；`credit_code`：18 位校验；`company_name`：trim |
| `max_depth` | int | 否 | 2 | 递归扩展深度 | 取值建议 `0~3`；超过上限按上限截断并 warning |
| `limits.max_nodes` | int | 否 | 1000 | 图谱节点上限（单 task） | `>=1`；触发即停止扩展并 warning |
| `limits.max_edges` | int | 否 | 3000 | 图谱边上限（单 task） | `>=1`；触发即停止扩展并 warning |
| `limits.time_budget_s` | int | 否 | 180 | 模块级时间预算（秒） | `>=1`；触发即停止扩展并 warning |
| `source_policy.preferred_sources` | array | 否 | `["icp"]` | 数据源优先级顺序 | 仅允许白名单；未知 source 直接 400 |
| `source_policy.allow_paid_sources` | bool | 否 | true | 是否允许使用付费源（若已配置密钥） | 若 false，遇到仅付费可得字段应降级并 warning |
| `source_policy.allow_fallback` | bool | 否 | true | 上游失败时是否允许其他源兜底 | false 时，上游失败应直接进入 review 或 fail（按场景） |
| `evidence_policy.store_raw_payload` | bool | 否 | true | 是否保存原始回包 | false 时也必须保留 `source_ref` 的最小元数据 |
| `evidence_policy.store_source_refs` | bool | 否 | true | 是否保存证据引用（回放索引） | 建议一期恒为 true |

### 3.3.2 `EnterpriseIntelResult` 字段说明

| 字段 | 类型 | 说明 |
|---|---:|---|
| `status` | enum | `succeeded` / `failed` / `partial`（达到限制或降级导致不完整） |
| `companies/persons/relations` | array | 结构化实体与关系（供聚合层/前端列表展示） |
| `graph_delta` | object | 本次任务写入/新增的图谱增量（用于前端增量刷新；也可返回全量） |
| `evidence` | array | 证据索引（见第六章）；任一关键结论必须能回溯到 evidence |
| `warnings` | array | 可解释的风险/降级/冲突信息（见 3.5） |
| `review_required` | bool | 是否必须进入复核模块（高风险/冲突/歧义/越界） |
| `stats` | object | `depth_reached/nodes/edges` 等统计，便于前端与聚合层展示 |

### 3.3.3 `status`/`stage` 枚举（建议统一）

- `status`：`pending` / `running` / `succeeded` / `failed` / `partial`
- `stage`：`resolve` / `fetch` / `expand` / `write` / `aggregate`

### 3.3.4 结构化实体与关系模型（`companies/persons/relations`）

> 目标：前端/聚合层可直接展示、可排序、可复核；任意关键关系必须能回溯到 `evidence_refs`。

`companies[]`（建议最小字段集）：

| 字段 | 类型 | 必填 | 说明 |
|---|---:|:---:|---|
| `id` | string | 是 | 内部唯一 id（建议：`company:{credit_code}` 或 `company:hash(name)`） |
| `name` | string | 是 | 公司名称（标准化后的展示名） |
| `credit_code` | string \| null | 否 | 统一社会信用代码（若可得） |
| `status` | string \| null | 否 | 企业状态（存续/注销/吊销等，来源决定） |
| `source_tags` | string[] | 否 | 命中的数据源列表（如 `["icp","qichacha"]`） |
| `confidence` | number \| null | 否 | 主体置信度（0~1；可选） |

`persons[]`（建议最小字段集）：

| 字段 | 类型 | 必填 | 说明 |
|---|---:|:---:|---|
| `id` | string | 是 | 内部唯一 id（优先使用来源稳定 id，否则 task 内临时 id） |
| `name` | string | 是 | 姓名 |
| `source_tags` | string[] | 否 | 命中的数据源列表 |
| `ambiguity` | boolean \| null | 否 | 是否存在同名歧义风险 |

`relations[]`（建议最小字段集）：

| 字段 | 类型 | 必填 | 说明 |
|---|---:|:---:|---|
| `type` | enum | 是 | `LEGAL_PERSON` / `SHAREHOLDER` / `INVEST` / `EMPLOY` |
| `source_id` | string | 是 | 起点实体 id（对应 company/person 的 id） |
| `target_id` | string | 是 | 终点实体 id |
| `attrs` | object | 否 | 关系属性：`ratio`、`role_title`、`confidence` 等 |
| `evidence_refs` | string[] | 否 | 与该关系直接相关的证据引用（`source_ref` 或 `evidence_id`） |

### 3.3.5 图谱数据结构（`graph_delta` / `/intel/graph`）

`nodes[]`（GraphNode）建议字段：

| 字段 | 类型 | 必填 | 说明 |
|---|---:|:---:|---|
| `id` | string | 是 | 与 `companies[].id / persons[].id` 对齐 |
| `type` | enum | 是 | `company` / `person` |
| `name` | string | 是 | 展示名 |

`edges[]`（GraphEdge）建议字段：

| 字段 | 类型 | 必填 | 说明 |
|---|---:|:---:|---|
| `source` | string | 是 | 起点节点 id |
| `target` | string | 是 | 终点节点 id |
| `type` | enum | 是 | 与 `relations[].type` 对齐（如 `LEGAL_PERSON`） |
| `attrs` | object | 否 | `ratio/role_title/confidence/source_ref` 等 |

## 3.4 复核触发（对齐总控的“复核模块”路由）

满足任一条件应输出 `review_required=true` 并给出 `warnings`（供总控路由到复核模块）：

- 同一公司出现冲突 `credit_code` 或主体关键信息（如名称/状态）在不同来源明显不一致
- 人物仅凭姓名产生跨公司合并（应禁止或降级为“疑似同名”并触发复核）
- 股权比例/控制关系来自不同来源且冲突明显
- 扩展过程中出现异常膨胀（接近/达到规模上限）

## 3.5 `warnings` 工程化格式（建议）

建议 warnings 统一结构，便于总控聚合、前端展示与复核路由：

```json
{
  "code": "SOURCE_RATE_LIMIT",
  "severity": "warn",
  "message": "qichacha 触发限流，已降级使用 aiqicha 补齐",
  "source": "qichacha",
  "retryable": true,
  "hint": "降低并发或延长重试间隔",
  "evidence_refs": ["minio://.../raw/qichacha/error-xxx.json"]
}
```

建议 code 白名单（一期至少覆盖）：

- `SUBJECT_UNRESOLVED`：主体无法确认
- `SOURCE_RATE_LIMIT`：上游限流/风控
- `SOURCE_TIMEOUT`：上游超时/网络异常
- `DATA_CONFLICT`：多源关键字段冲突
- `IDENTITY_AMBIGUOUS`：同名歧义无法消解
- `LIMIT_REACHED`：触发深度/规模/时间预算上限

## 3.6 模块标准输出模板(待议)

当企业情报模块作为 LLM Agent 运行时，建议约束其**只输出**以下结构化 JSON（便于总控/状态机直接消费），并确保字段含义与本章一致：

```json
{
  "task_id": "",
  "status": "succeeded",
  "stage": "aggregate",
  "companies": [],
  "persons": [],
  "relations": [],
  "graph_delta": { "nodes": [], "edges": [] },

  "warnings": [],
  "review_required": false,
  "stats": {
    "depth_reached": 0,
    "nodes": 0,
    "edges": 0,
    "time_used_s": 0
  }
}
```

# 四、对外 HTTP API 设计（面向前端/调用方）

> 若一期最终由总控统一对外 API，本节可作为“企业情报子资源接口”；但字段仍应与模块契约一致。

## 4.1 任务创建接口

POST `/intel/task/create`

请求（兼容最小请求：仅 `target/depth`；其余为可选增强字段）：

```json
{
  "target": "example.com",
  "depth": 2,
  "preferred_sources": ["icp", "aiqicha", "qichacha"],
  "max_nodes": 1000,
  "max_edges": 3000,
  "time_budget_s": 180,
  "store_raw_payload": true
}
```

字段解释（与 `EnterpriseIntelCommand` 的映射关系）：

| 字段 | 类型 | 必填 | 默认值 | 映射 | 说明 |
|---|---:|:---:|---:|---|---|
| `target` | string | 是 | - | `targets[0]` | 目标线索；若含协议/路径需规范化为 domain |
| `depth` | int | 否 | 2 | `max_depth` | 扩展深度（建议 0~3） |
| `preferred_sources` | string[] | 否 | `["icp"]` | `source_policy.preferred_sources` | 数据源优先级列表 |
| `max_nodes` | int | 否 | 1000 | `limits.max_nodes` | 节点上限 |
| `max_edges` | int | 否 | 3000 | `limits.max_edges` | 边上限 |
| `time_budget_s` | int | 否 | 180 | `limits.time_budget_s` | 时间预算（秒） |
| `store_raw_payload` | bool | 否 | true | `evidence_policy.store_raw_payload` | 是否保存原始回包 |

返回：

```json
{
  "task_id": "uuid-xxx",
  "status": "pending"
}
```

## 4.5 错误码与错误场景（建议）

- `400`：参数非法（depth 超限、target 为空、sources 不支持）
- `401/403`：鉴权/无权限（若有多租户）
- `429`：触发源站限流/配额（返回可重试建议，并记录 warnings）
- `502/504`：上游数据源失败/超时（降级并产出 warnings）
- `500`：内部异常（建议返回 `trace_id` 便于回放）

## 4.6 错误响应体（建议统一）

为便于前端与总控统一处理，建议错误响应结构固定为：

```json
{
  "code": "INVALID_ARGUMENT",
  "message": "depth must be <= 3",
  "trace_id": "trace-xxx",
  "retryable": false,
  "details": {}
}
```

## 4.7 幂等与限流（建议）

- 任务创建建议支持幂等键（如 `Idempotency-Key`）避免重复创建
- 每个数据源 Adapter 独立并发与 QPS 控制（避免风控/封禁）

# 五、数据源适配层（一期：可插拔多数据源）

## 5.1 Adapter 抽象（建议）

将爱企查/企查查/ICP/站长之家等实现为多个 Adapter，统一为同一接口：

- `resolve_subject(target)`：域名/公司名 → 主体（公司名/credit_code）
- `fetch_company_profile(company)`：公司画像（基础信息、状态、地址等）
- `fetch_company_relations(company)`：法人/股东/对外投资/任职等
- `fetch_change_history(company)`：历史变更（若可用）

Adapter 统一返回结构建议包含：`source`、`fetched_at`、`raw_ref`（可回放引用）、`confidence`（可选）。

## 5.2 优先级与降级策略（一期默认）

1. ICP（域名备案）优先用于“主体确认”
2. 企业画像来源按 `preferred_sources` 顺序尝试
3. 任一来源不可用：不中断任务，记录 warning，并继续用其他来源补齐
4. 若主体无法确认：停止扩展并 `review_required=true`

## 5.3 配额/限流/风控处理（建议）

- 每个 source 单独的并发与 QPS（如 `qps=1~3`）配置
- 触发 `429/验证码/风控`：快速降级到其他源，并记录失败原因（warnings）与失败证据引用（若可保存）

## 5.4 配置项与密钥管理（工程落地必须）

建议以环境变量或配置文件集中管理各数据源的密钥与限流参数（不得硬编码在代码与文档示例里）：

- `INTEL_SOURCE_ICP_ENABLED=true/false`
- `INTEL_SOURCE_AIQICHA_API_KEY=...`
- `INTEL_SOURCE_QICHACHA_API_KEY=...`
- `INTEL_SOURCE_ZHANZHANG_API_KEY=...`
- `INTEL_SOURCE_<SOURCE>_QPS=1`
- `INTEL_SOURCE_<SOURCE>_CONCURRENCY=2`
- `INTEL_EVIDENCE_STORE=minio`（或 `postgres`）
- `INTEL_EVIDENCE_RETENTION_DAYS=90`

# 六、数据面：图谱 + 证据 + 审计回放

## 6.1 证据（Evidence）落库建议

为支持“可回放、可审计”，建议每次拉取外部数据源时生成 evidence：

- `source`：来源（icp/aiqicha/qichacha/...）
- `query`：查询参数（域名/公司名/credit_code）
- `fetched_at`：拉取时间
- `source_ref`：原始回包的引用（如 MinIO 路径或数据库主键）
- `digest`：原始回包摘要（可选，便于去重）

> 证据不一定要全部写入 Neo4j；更建议写入独立存储（如 PostgreSQL/MinIO），Neo4j 节点/边仅保留 `source_ref` 关联。

建议 evidence 最小字段集（AI 生成代码时可直接作为数据结构）：

```json
{
  "evidence_id": "ev-uuid",
  "task_id": "uuid",
  "source": "icp",
  "query": "example.com",
  "fetched_at": "2026-04-08T12:00:00Z",
  "source_ref": "minio://.../raw/icp/xxx.json",
  "digest": "sha256:...",
  "http_status": 200,
  "cost_ms": 1200
}
```

MinIO 路径规范建议（便于检索与回放）：

- `raw/{task_id}/{source}/{yyyy-mm-dd}/{evidence_id}.json`

## 6.2 实体去重与消歧（必须说明）

- Company 去重：
  - 优先按 `credit_code`；将 `company.id` 设为 `company:{credit_code}`，确保全局唯一
  - 无 `credit_code` 时：将 `company.id` 设为 `company:hash(normalized_name)`（低置信），并在 `warnings` 标注主体不稳定
- Person 去重：
  - 默认不跨公司合并；将 `person.id` 设为 `person:{source}:{source_person_id}`（若来源提供）
  - 若无稳定 id：使用 task 内部临时 id（不做全局合并），并 `ambiguity=true`

# 七、图数据库表设计（Neo4j）

## 7.1 图模型设计

### 7.1.1 节点设计（Node）

系统中定义两类核心节点：

（1）Company（企业节点）

|字段|类型|说明|
|---|---|---|
|id|string|内部唯一标识（建议：优先 credit_code，否则 hash(name)）|
|name|string|公司名称|
|credit_code|string|统一社会信用代码（可空）|
|status|string|企业状态|
|source_tags|list|数据来源标记（可选）|
|confidence|number|主体置信度（可选）|

（2）Person（个人节点）

|字段|类型|说明|
|---|---|---|
|id|string|内部唯一标识（优先来源提供的稳定标识；否则 uuid）|
|name|string|姓名|
|source_tags|list|数据来源标记（可选）|
|ambiguity|boolean|是否存在同名歧义风险（可选）|

### 7.1.2 关系设计（Relation）

系统中定义以下关系类型（建议支持“公司股东”）：

|关系类型|起点|终点|说明|
|---|---|---|---|
|LEGAL_PERSON|Person|Company|法人|
|SHAREHOLDER|Person/Company|Company|股东|
|INVEST|Company|Company|投资关系|
|EMPLOY|Person|Company|任职关系|

关系属性建议：`source`、`source_ref`、`fetched_at`、`confidence`，以及可选字段 `ratio`（股权比例）、`role_title`（任职）。

### 7.1.3 图结构示例（Graph）

```text
张三 ──(法人)──> A公司
张三 ──(股东)──> B公司
A公司 ──(投资)──> C公司
```

## 7.2 查询模型设计

1 查询企业关联人物

```cypher
MATCH (p:Person)-[:LEGAL_PERSON|SHAREHOLDER|EMPLOY]->(c:Company {name: "目标公司"})
RETURN p
```

2 查询企业对外投资

```cypher
MATCH (c1:Company {name: "目标公司"})-[:INVEST]->(c2:Company)
RETURN c2
```

3 查询关系拓扑（核心）

```cypher
MATCH (n)-[r]-(m)
WHERE n.name = "目标公司"
RETURN n, r, m
LIMIT 100
```

4 多跳关系查询（Agent 核心能力）

```cypher
MATCH path = (c:Company {name: "目标公司"})-[*1..2]-(n)
RETURN path
```

## 7.3 数据写入设计（幂等入库策略）

1 Company 节点去重（建议统一以 `id` 作为主键）

```cypher
MERGE (c:Company {id: $id})
SET c.name = $name, c.credit_code = $credit_code
```

无 `credit_code` 的兜底（仍使用 `id`，但 `credit_code` 为空且 confidence 更低）：

```cypher
MERGE (c:Company {id: $id})
SET c.name = $name
```

2 关系创建（附带来源与证据引用）

```cypher
MATCH (p:Person {id: $person_id})
MATCH (c:Company {id: $company_id})
MERGE (p)-[r:LEGAL_PERSON]->(c)
SET r.source = $source, r.source_ref = $source_ref, r.fetched_at = $fetched_at
```

3 更新属性（示例）

```cypher
SET c.status = $status
```

## 7.4 图规模控制（必须与停止条件一致）

- 最大扩展深度：默认 2（可配置 2~3）
- 最大节点数限制：默认 1000（可配置）
- 最大边数限制：默认 3000（可配置）
- 过滤低价值节点：无新增关系的叶子节点不继续扩展
- 循环去重：同一公司在同一 task 内只扩展一次

# 八、核心处理流程（Agent 逻辑，阶段化）

### Stage 1：主体解析（resolve）

域名 → ICP → 备案主体公司：

1. 调用 ICP Adapter 获取备案信息与主体线索
2. 若无法确认主体：输出 warning，`review_required=true`，结束或降级为“仅备案信息结果”

### Stage 2：企业画像采集（fetch）

1. 按 `preferred_sources` 顺序拉取企业画像与关系数据
2. 每次拉取都生成 evidence（`source_ref/raw_ref`）
3. 对关键字段（credit_code、法人、主要股东、对外投资）做最小一致性校验

### Stage 3：递归扩展（expand，重点）

扩展入口建议（可配置）：

- 对外投资企业（INVEST）
- 关键股东/控制方关联公司（SHAREHOLDER）
- 关键人员任职关联公司（EMPLOY，谨慎）

停止条件：`depth==0` 或任一 `limits` 触发或命中异常膨胀/循环。

伪代码（示意）：

```python
def expand(company_id, depth):
    if depth == 0 or reached_limits():
        return
    if visited(company_id):
        return
    mark_visited(company_id)

    rels = fetch_relations(company_id)
    write_graph(company_id, rels)

    for next_company in pick_next_companies(rels):
        expand(next_company, depth - 1)
```

### Stage 4：图谱写入与一致性校验

1. 写入 Neo4j：节点/边/属性/证据引用
2. 冲突检测：明显冲突输出 warnings，并 `review_required=true`

### Stage 5：结构化输出与图谱输出（aggregate）

1. 输出结构化结果：`companies/persons/relations/evidence/warnings/stats`
2. 输出图谱数据：`nodes/edges` 或 `graph_delta`
3. 输出复核标记：`review_required`

# 十、参考实现骨架（用于 AI 直接生成 Agent 代码）

> 一期建议落地语言与框架：Python + Flask；数据结构使用 Pydantic（或 dataclass），与 `docs/heuScan/项目介绍与功能设计/2.第一阶段多Agent功能拆解与协同架构设计.md` 的后端形态保持一致。

## 10.1 模块入口（建议）

- 模块入口函数：`run_enterprise_intel(command: EnterpriseIntelCommand, ctx: RuntimeContext) -> EnterpriseIntelResult`
- 入口必须只依赖 `command + ctx`，不得直接读取 Flask request 对象（由总控/服务层装配输入）

## 10.2 Adapter 接口（建议直接照此生成代码）

```python
class IntelSourceAdapter(Protocol):
    source_name: str

    def resolve_subject(self, target: str, *, timeout_s: int) -> SubjectResolveResult: ...
    def fetch_company_profile(self, company_key: CompanyKey, *, timeout_s: int) -> CompanyProfileResult: ...
    def fetch_company_relations(self, company_key: CompanyKey, *, timeout_s: int) -> CompanyRelationsResult: ...
    def fetch_change_history(self, company_key: CompanyKey, *, timeout_s: int) -> ChangeHistoryResult: ...
```

## 10.3 重试/超时/降级（决策必须写死，避免 AI 自行发挥）

- 单次调用默认超时：`timeout_s=10`（可按 source 单独配置）
- 重试策略：最多 `max_attempts=2`；仅对 `timeout/5xx/网络错误` 重试；`429/风控` 不重试，直接降级
- 降级策略：按 `preferred_sources` 顺序切换；所有候选均失败则输出 `SUBJECT_UNRESOLVED` 或 `SOURCE_TIMEOUT` 等 warning，并在必要时置 `review_required=true`

## 10.4 规范化与 ID 生成（必须一致，否则图谱与结构化结果会漂移）

- `domain` 规范化：去协议、去路径、去端口、转小写（如 `https://Example.com/a` → `example.com`）
- `company_name` 规范化：trim、全角半角归一（若有）、连续空白压缩
- `credit_code` 校验：长度 18；若不合法不得作为强主键使用（只能作为弱字段，并 warning）
- `company.id`：
  - 有 `credit_code`：`company:{credit_code}`
  - 无 `credit_code`：`company:hash({normalized_name})`
- `person.id`：
  - 有来源稳定 id：`person:{source}:{source_person_id}`
  - 无稳定 id：`person:temp:{task_id}:{seq}`

# 十一、Neo4j 约束与索引（一期建议）

> 目的：保证幂等写入、提升查询性能，避免同一实体重复膨胀。

CREATE CONSTRAINT company_id_unique IF NOT EXISTS

CREATE CONSTRAINT person_id_unique IF NOT EXISTS
