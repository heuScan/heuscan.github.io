# 一、任务描述
    价值评估模块，对其余agent的结果，资产进行正能价值评估，用于标注高价值目标，方便渗透测试工程师进行重点关注。
# 二、任务分析
## 1. 资产价值分级
    价值评估模块的首要任务不是发现资产，也不是验证漏洞，而是在上游模块已经完成资产发现、指纹识别、关系聚合和风险线索收集之后，对资产做“价值分级”。这里的价值分级本质上是对“这个资产是否值得优先关注、为什么值得关注、应该关注到什么程度”做结构化回答。

    资产价值分级必须以四项核心依据为中心：
    1. 资产重要性：判断该资产本身是否承载核心业务、是否处于主入口、是否具备管理面/控制面/关键数据面属性；
    2. 技术栈价值信号：判断该资产暴露出的系统角色、支撑价值和风险暴露特征；
    3. 风险线索：判断当前时间点下该资产是否存在高可信、高强度、值得尽快处理的风险迹象；
    4. 业务关联性：判断该资产是否位于主站、统一认证、核心 API、关键依赖链等核心业务链路中。

    该模块输出的价值分级结果应能够支持以下目标：
    1. 区分皇冠资产、高价值资产、重点关注资产、一般关注资产和普通资产；
    2. 解释高价值结论来自哪些主因子，而不是只给一个黑盒分数；
    3. 对证据不足、信号冲突、代理推断过强的结果增加待复核标记；
    4. 为后续优先级排序、报告输出和人工研判提供统一依据。
## 2.优先级评估
    价值评估模块的第二项任务是在价值分级基础上完成优先级评估。这里的优先级评估不是简单按照“谁风险高谁优先”，而是按照“基础价值 + 风险修正 + 证据可信度”联合排序，回答“当前资源应该先投入到哪些资产”。

    优先级评估应遵循以下原则：
    1. 先看最终价值分，再看风险线索强度，最后看证据完整度；
    2. 高风险但低业务价值的边缘资产，可以提高处置优先级，但不应压过真正的核心业务资产；
    3. 高价值但风险线索缺失的资产，仍然应保留高关注等级，但可不提升即时处置优先级；
    4. 证据冲突、信息不足、代理信号单一的资产，即使分数较高，也应在排序输出中增加复核提醒。

    因此，优先级评估的输出不应只有排序号，还应附带：
    1. 排序原因；
    2. 优先级建议；
    3. 主要抬升因子；
    4. 是否属于核心业务；
    5. 是否需要人工复核。
## 3.输入信息
    该模块不直接依赖原始网页抓取、原始扫描日志或主动攻击结果，而是消费上游 agent 已经加工后的结构化输入。当前输入边界应固定为以下几类：
    1. 资产列表：包括资产标识、域名/URL/IP、资产类型、入口属性、域名层级、页面语义、主子域关系等；
    2. 指纹信息：包括中间件、框架、CMS、组件标签、认证方式、控制面语义、接口路径语义、系统角色标签等；
    3. 被动审查风险信息：包括漏洞情报、敏感接口、异常回显、弱口令痕迹、密钥/令牌暴露、配置异常、多源交叉验证结果等；
    4. 关系和代理信号：包括页面跳转链、JS/API 调用关系、同源关系、证书主体、上下游依赖、统一认证或业务网关痕迹等；
    5. 证据完整度信息：包括来源数量、来源一致性、是否存在冲突、是否形成交叉验证。

    输入边界必须满足以下约束：
    1. 价值评估模块只能消费已提供证据，不能主动补跑扫描、不能自行扩展目标范围；
    2. 风险线索属于修正因子，不得取代资产重要性和业务关联性的基础输入；
    3. 若输入仅有弱资产信息和模糊风险情报，模块应允许输出低置信或待复核结论，而不是强行给出高价值判断；
    4. 若输入包含多源一致的角色信号和链路信号，应允许输出更稳定的高价值结论。
## 4.输出信息
    该模块的输出不只是一个分数，而是一组能够被下游模块、前端和人工研判共同消费的结构化结果。输出结果至少应覆盖以下三类：
    1. 资产价值分级结果：包括价值等级、最终价值分、四项主因子分、证据完整度修正、主要解释；
    2. 优先级排序结果：包括排序位置、排序原因、优先级建议、资源倾斜建议、复核标记；
    3. 核心业务判定结果：包括是否属于核心业务、核心业务角色、关键链路位置、支撑信号、边界结论。

    输出结果需要满足以下要求：
    1. 结果可解释，能够说明“为什么高价值”“为什么排在前面”；
    2. 结果可审计，所有关键结论都能回溯到输入证据；
    3. 结果可排序，能够直接供前端展示和后续报告模块引用；
    4. 结果可复核，对证据弱、冲突大、代理推断成分高的结论必须显式标记；
    5. 结果要与第三部分评分标准、第四部分核心业务判定、第五部分排序算法保持一致。
## 5.核心业务判定
    价值评估模块不仅要回答“资产值不值得关注”，还要回答“它是不是核心业务、如果是，边界在哪里”。这是因为很多高价值结论最终都需要落到“核心业务主入口”“核心交易节点”“关键支撑节点”“控制节点”等更具体的业务角色上，否则很难被渗透测试工程师直接采用。

    核心业务判定在本模块中的职责主要包括：
    1. 判断资产是否属于核心业务主入口、核心交易节点、核心支撑节点或核心控制节点；
    2. 判断该资产是否位于统一认证、主站、核心 API、关键依赖链等关键业务链路中；
    3. 明确哪些资产只能判为“疑似核心业务”，哪些资产当前证据不足，不能直接纳入核心业务边界；
    4. 为高价值结论提供业务解释，并为边缘资产误抬升提供纠偏依据。

    因此，第二部分在整体上可以概括为：
    1. 以四项核心依据完成价值分级；
    2. 以分级结果和风险修正完成优先级评估；
    3. 以关系信号和角色语义完成核心业务判定；
    4. 以证据完整度和误判控制规则约束最终输出可信度。
# 三、价值标准设定
## 1.资产重要性判定标准
**定义与设计目标**
    资产重要性用于判断“这个资产本身是否值得优先关注”，它是价值评分的基础价值底座，不与漏洞严重度、被利用概率、单点风险线索混写。该维度主要回答三个问题：资产是否承载核心业务、是否处于用户主入口、是否具备管理面/控制面/关键数据面属性。

**输入来源**
1. 资产列表；
2. 基础识别结果；
3. 域名层级；
4. 入口属性；
5. 页面标题、登录形态、管理面标识；
6. URL 路径语义与响应页面语义。

**1-5 分分级标准**

| 分值 | 判定标准 | 典型命中信号 |
| --- | --- | --- |
| 1分 | 边缘展示页、静态资源站、普通下载页、弱业务承载节点 | `static`、`cdn`、纯展示页、帮助中心、公开文档页 |
| 2分 | 普通业务子站、非核心功能页、局部支撑服务 | 活动页、普通业务子域、单功能服务页、普通二级站 |
| 3分 | 重要业务支撑系统、面向客户的业务页面、常用 API 服务 | 业务站点首页、普通登录页、客户常用 API、订单查询页 |
| 4分 | 主站、统一入口、核心业务 API、关键数据面、重要管理后台 | 主域名首页、统一登录入口、业务网关、运营后台、数据中心入口 |
| 5分 | 统一认证、支付交易、核心管理面、数据库管理面、云控制台、CI/CD 控制面、集群控制面 | `sso`、`auth`、支付结算、数据库控制台、云平台、流水线、K8s 控制面 |

**典型高分场景**
1. 主域名首页或主站登录入口；
2. 统一认证、统一用户中心、单点登录入口；
3. 订单、支付、结算、账户、交易等核心业务入口；
4. 核心业务 API 网关、主数据服务、运营管理平台；
5. 数据管理平台、数据库管理入口、对象存储控制面、CI/CD 与集群控制面。

**典型低分场景**
1. 纯静态资源站、图片/JS/CDN 资源节点；
2. 普通下载站、帮助中心、公开文档站；
3. 无登录、无业务交互、无管理能力的边缘展示页面。

**边界与误判控制**
1. 仅因页面名称含 `admin`、`system`、`manage` 不得直接给 5 分；
2. 仅因开放非常见端口不得直接判定为核心资产；
3. 仅因域名短、主域相似，不足以直接判定为主站或统一入口；
4. 无法确认资产角色时，上限不超过 3 分，等待技术栈、业务关联性等因子抬升；
5. 资产重要性判断强调“业务承载强度”，不因单个高危漏洞自动升高。

**与综合评分公式的关系**
1. 该维度为综合评分第一主因子；
2. 权重上限固定为 `30`；
3. 该维度越高，资产在缺少风险线索时仍可能保持高价值等级；
4. 该维度低时，除非其他因子强烈抬升，否则不应进入最高价值档。

## 2.技术栈判定标准
**定义与设计目标**
    技术栈维度不是简单的“组件识别结果”，而是一个双重因子：既用于推断业务价值，也用于推断风险暴露。该维度要解决的问题是，很多资产即使页面语义不强，也会通过技术栈暴露出其系统角色，例如统一认证、ERP、中台、数据库管理面、CI/CD、云控制台、K8s 控制面等；同时，某些高危中间件、老旧框架、公开后台又会显著抬升优先级。

**输入来源**
1. 指纹识别结果；
2. 中间件、框架、CMS、组件标签；
3. 服务端口与协议特征；
4. 页面资源与前端框架；
5. 登录组件与认证方式；
6. 页面标题、接口路径、产品特征词。

**技术栈业务价值子判断**
1. ERP、OA、SCM、CRM、中台、统一认证、对象存储、CI/CD、K8s、云平台、数据库管理面等技术角色，通常说明更高业务价值；
2. API 网关、统一鉴权、日志平台、配置中心、服务治理、监控平台等，也应视为具有较高支撑价值；
3. 若仅识别到通用框架而无法识别系统角色，则业务价值子判断不宜过高。

**技术栈风险暴露子判断**
1. 高危中间件、老旧框架、公开后台、弱鉴权组件、历史高风险组件会提高优先级；
2. 风险暴露只用于抬升优先级，不直接等同于资产业务价值；
3. 若组件仅存在历史风险记录，但未命中当前版本或场景，不应高分。

**1-5 分分级标准**

| 分值 | 判定标准 | 典型命中信号 |
| --- | --- | --- |
| 1分 | 通用静态站点、无明显系统角色、低业务语义技术栈 | 纯静态框架、普通博客主题、简单 CMS 前台 |
| 2分 | 普通内容站/CMS/博客类技术栈，风险暴露一般 | WordPress 前台、内容发布站、普通门户框架 |
| 3分 | 常见业务框架、开放 API 框架、内部系统常用技术组合 | 常见 Java/Python/.NET 业务系统、API 服务框架 |
| 4分 | 具备明确高价值系统语义或明确高风险暴露语义 | 统一认证、网关、数据库控制台、公开高危后台 |
| 5分 | 同时具备高价值系统角色和高风险暴露/控制面角色 | 云控制台、K8s 控制面、数据库管理面、CI/CD 控制面且暴露明显 |

**典型高分场景**
1. 指纹明确命中统一认证、支付网关、云控制台、数据库管理面；
2. 技术栈显示为 K8s 控制面、CI/CD 流水线、对象存储控制台；
3. 命中高价值角色同时伴随高风险暴露，如公开管理后台、弱鉴权控制面。

**典型低分场景**
1. 只能识别到通用前端框架、无系统角色语义；
2. 只是普通 CMS 前台、博客、帮助中心模板；
3. 没有中间件、组件、控制面、认证方式等有效技术信号。

**边界与误判控制**
1. 技术栈不能脱离资产重要性单独决定最高价值档；
2. 若只能识别到“框架”而不是“系统角色”，则业务价值子判断上限为 3 分；
3. 若仅命中历史高风险组件，但当前暴露面和版本不明确，不得直接判高分；
4. 若技术栈命中控制面角色，可触发基础价值保底规则，但仍需结合业务关联性确认；
5. 业务价值子项权重应高于风险暴露子项，避免算法过度偏向漏洞性技术栈。

**与综合评分公式的关系**
1. 该维度权重上限固定为 `25`；
2. 该维度是双重因子，内部由“技术栈业务价值子判断”和“技术栈风险暴露子判断”组成；
3. 该维度既能抬升基础价值，也能辅助触发高价值角色门槛规则；
4. 该维度不能替代资产重要性，但可在资产角色模糊时提供关键判定依据。

## 3.风险线索判定标准
**定义与设计目标**
    风险线索用于表达“当前时间点下这个资产是否值得优先处理”，它是动态优先级修正因子，不是价值底座。该维度的目标不是回答“这个资产本质上有多重要”，而是回答“它现在是否存在高可信、高强度、需要尽快验证的风险迹象”。

**输入来源**
1. 被动流量审查结果；
2. 文件情报；
3. 指纹关联到的公开漏洞情报；
4. 敏感接口、弱口令、可疑外联、高危配置迹象；
5. 认证异常、令牌暴露、敏感参数暴露；
6. 多源证据交叉验证结果。

**线索类型分组**
1. 漏洞类线索：已知漏洞、组件高危版本、可疑敏感接口；
2. 配置类线索：未授权访问、高危配置、默认配置、危险暴露面；
3. 认证类线索：弱口令痕迹、默认凭证、登录绕过迹象、鉴权异常；
4. 行为类线索：可疑外联、敏感回显、异常跳转、危险接口调用；
5. 敏感信息类线索：密钥、令牌、配置文件、账户信息、内部地址、源码片段暴露。

**1-5 分分级标准**

| 分值 | 判定标准 | 典型命中信号 |
| --- | --- | --- |
| 1分 | 无明显风险线索，或线索弱且不可信 | 单一模糊告警、弱证据、无上下文匹配 |
| 2分 | 存在一般性风险迹象，但证据不充分 | 低可信组件风险、单一敏感路径、疑似配置问题 |
| 3分 | 存在明确中风险线索，或有多条弱线索叠加 | 多个中风险接口、可疑暴露、文件情报与指纹相互印证 |
| 4分 | 存在高危线索、利用面明显、或多源证据相互印证 | 高危组件暴露、敏感接口命中、异常回显、多源证据确认 |
| 5分 | 存在高危且高可信线索，伴随已知被利用、敏感数据暴露、弱口令痕迹、关键配置失守等情形 | 高危漏洞命中、疑似已利用、密钥/令牌暴露、关键接口失守 |

**典型高分场景**
1. 被动流量中出现敏感令牌、密钥、内部配置或高危接口调用痕迹；
2. 文件情报发现配置泄露、源代码泄露、账号口令线索；
3. 指纹识别明确命中高危组件且版本、场景、暴露方式一致；
4. 多源证据交叉确认存在高危弱口令、未授权访问或关键配置失守。

**典型低分场景**
1. 只有历史组件风险，没有当前版本或暴露面的对应关系；
2. 单一情报来源、证据弱、无场景匹配；
3. 仅有模糊目录命中或泛化告警，没有进一步支撑。

**边界与误判控制**
1. 单一低可信情报不得直接给 5 分；
2. 只因组件历史有 CVE、但未命中当前版本或场景，不得高分；
3. 风险线索高不等于资产价值高，只能抬升优先级；
4. 若风险线索缺少上下文和复核证据，应通过证据完整度修正下调；
5. 风险线索不应覆盖资产重要性和业务关联性的基础判断。

**与综合评分公式的关系**
1. 该维度权重上限固定为 `20`；
2. 该维度主要影响“优先级排序”和“资源倾斜建议”；
3. 该维度对“价值等级”的影响是修正，而不是主定义；
4. 当基础价值低时，风险线索再高，也不应直接把边缘资产抬入最高价值档。

## 4.业务关联性判定标准
**定义与设计目标**
    业务关联性用于判断资产是否处在核心业务链路中，是否与主站、统一认证、核心 API、关键依赖、管理平台、上下游系统形成稳定关系。该维度承接架构文档中“图谱关系、资产特征、风险线索”的输入边界，在当前尚未完全依赖图数据库的阶段，允许通过代理信号进行工程化推断。

**输入来源**
1. 资产列表中的域名层级与入口关系；
2. 指纹信息中的认证、网关、控制面、调用痕迹；
3. JS/API 调用关系；
4. 页面元素、链接关系、证书主题、命名特征；
5. 文件情报和被动流量中的上下游线索；
6. 页面跳转链、同源关系、主子域关系。

**1-5 分分级标准**

| 分值 | 判定标准 | 典型命中信号 |
| --- | --- | --- |
| 1分 | 孤立节点、边缘节点、与核心站点无明显联系 | 独立静态子域、无跳转、无调用、无核心语义 |
| 2分 | 与普通业务站点存在弱联系，但不在关键链路 | 仅有普通链接关系、弱域名关联、浅层引用 |
| 3分 | 与主站、业务接口、内部系统存在稳定联系 | 主站子模块、常规接口调用、登录后普通业务系统 |
| 4分 | 处在统一登录、核心 API、重要业务子系统、关键依赖链中 | 统一登录跳转、核心业务接口引用、主流程依赖服务 |
| 5分 | 属于统一认证、核心网关、控制面、关键上游/下游依赖、全局跳板或关键枢纽节点 | `sso`、`gateway`、控制台、全局认证、关键依赖枢纽 |

**代理信号清单**
1. 主子域命名关系；
2. `/api/`、`/sso/`、`/auth/`、`/admin/`、`/gateway/` 等路径语义；
3. 同源登录页或登录跳转链；
4. JS 中引用主域 API、核心域名或统一认证接口；
5. 证书主体与企业名称一致，且与主站命名体系匹配；
6. 页面文案出现统一平台、管理中心、控制台、业务中台、数据中心等高价值语义；
7. 文件情报和流量线索中出现稳定上下游调用关系。

**典型高分场景**
1. 统一认证页、核心网关、全局 API 网关；
2. 主站跳转依赖的认证中心或业务中台；
3. 多个业务子系统共同依赖的控制面、配置中心、数据库管理面；
4. 被动流量和 JS 线索都表明其处于关键上游/下游依赖链。

**典型低分场景**
1. 只有域名相似，但没有调用、跳转、认证或页面语义支撑；
2. 孤立二级站、临时活动站、边缘下载站；
3. 无 JS 调用、无链接关系、无证书语义、无页面语义、无业务链信号。

**边界与误判控制**
1. 仅因域名相似不得判高关联；
2. 无调用、无认证、无页面语义、无路径语义时，不得高分；
3. 业务关联性在当前阶段允许代理推断，但必须配合证据完整度修正；
4. 仅因属于同一主域，不代表一定是核心链路节点；
5. 若只存在单一代理信号，通常上限不超过 3 分。

**与综合评分公式的关系**
1. 该维度权重上限固定为 `20`；
2. 该维度用于判断资产是否进入核心链路，是基础价值的重要组成部分；
3. 该维度与资产重要性共同决定“无高危线索时是否仍可判为高价值资产”；
4. 该维度也是门槛规则的关键因子之一。

**综合评分算法总则**
    为保证第三部分可以直接指导工程实现，综合评分模型、分级、排序、门槛规则统一内嵌如下：

1. 综合评分模型：
   基础价值分 = 资产重要性(0-30) + 技术栈价值信号(0-25) + 业务关联性(0-20)
   风险修正分 = 风险线索(0-20)
   最终价值分 = min(100, 基础价值分 + 风险修正分 + 证据完整度修正)
   证据完整度修正 = -10 ~ +5
2. 1-5 分到权重分的映射：
   1 分 = 权重上限的 20%
   2 分 = 权重上限的 40%
   3 分 = 权重上限的 60%
   4 分 = 权重上限的 80%
   5 分 = 权重上限的 100%
3. 门槛规则：
   资产重要性 <= 2 且业务关联性 <= 2 时，不得进入最高价值档
   命中统一认证、支付、核心管理面、数据库管理面、云控制面、K8s 控制面时，基础价值至少进入中高档
   风险线索缺失但资产重要性与业务关联性同时高时，仍可判高价值，但不提升处置优先级
4. 价值分级：
   85-100：皇冠资产
   70-84：高价值资产
   50-69：重点关注资产
   30-49：一般关注资产
   0-29：普通资产
5. 排序规则：
   第一排序键：最终价值分
   第二排序键：风险线索分
   第三排序键：证据完整度
6. 输出要求：
   每个资产输出：价值等级、价值分、四项主因子分、证据修正、优先级建议、简要解释

    其中，证据完整度修正不是新的价值维度，而是可信度控制项。信息少、指纹弱、来源单一则下调；多源交叉验证、证据一致则小幅上调。这样可以避免系统在证据不足时对资产价值做出过度自信的判断。

# 四、核心业务判定与边界
## 1.核心业务逻辑判定标准

**定义与设计目标**
    核心业务判定用于回答“这个资产是否直接承载企业核心业务、是否处在关键业务链路、是否值得按核心业务目标持续关注”。它不是新的评分维度，而是对第三部分中“资产重要性”和“业务关联性”的进一步解释性判断结果，用于输出更明确的业务角色结论，辅助价值排序、资源倾斜和人工复核。

**输入来源**
1. 资产列表中的资产类型、域名层级、入口属性、站点角色；
2. 指纹信息中的统一认证、网关、管理面、控制面、数据库管理面、对象存储控制面等系统角色信号；
3. 页面标题、登录形态、页面文案、URL 路径语义；
4. JS/API 调用关系、同源关系、页面跳转链、上下游依赖线索；
5. 文件情报、被动流量中的关键接口、认证链路、主流程依赖痕迹；
6. 证书主体、企业命名体系、主子域关系和代理业务语义。

**核心业务的判定原则**
1. 直接承载订单、支付、结算、账户、交易、客户主流程的资产，优先判为核心业务；
2. 主站首页、主入口、统一登录、统一认证、统一用户中心等全局入口型资产，优先判为核心业务；
3. 核心 API 网关、统一鉴权、服务治理、配置中心、主数据服务等关键支撑型资产，可判为核心业务支撑节点；
4. 数据库管理面、云控制台、对象存储控制面、CI/CD 控制面、K8s 控制面等控制面资产，若与主业务体系存在稳定依赖关系，可判为核心业务控制节点；
5. 多个业务子系统共同依赖的上游/下游服务、枢纽节点、全局跳板，通常应判为核心业务链路节点；
6. 仅提供活动宣传、帮助说明、公开文档、静态资源、普通下载等弱业务承载页面，通常不判为核心业务；
7. 仅因疑似后台、疑似接口、疑似管理路径，不足以直接判定为核心业务，必须结合调用关系、认证链路和业务语义共同判断；
8. 风险线索本身不能定义核心业务，只能影响后续优先级建议。

**核心业务信号分组**
1. 主流程信号：订单、支付、结算、账户、交易、客户中心、业务办理等直接业务语义；
2. 全局入口信号：主站、统一入口、单点登录、统一认证、统一用户中心；
3. 支撑枢纽信号：核心 API、业务网关、统一鉴权、主数据、配置中心、监控平台、服务治理；
4. 控制面信号：运营后台、控制台、数据库管理面、云平台、对象存储控制面、CI/CD、K8s 控制面；
5. 依赖关系信号：主站跳转、JS/API 引用、同源认证、上下游依赖、多业务共用服务。

**核心业务角色分类**
1. 核心业务主入口：面向用户或客户的主站、统一入口、统一登录入口；
2. 核心业务交易节点：订单、支付、账户、结算、交易、核心办理流程入口；
3. 核心业务支撑节点：核心 API、统一鉴权、业务网关、主数据服务、配置中心；
4. 核心业务控制节点：管理面、控制台、数据库管理面、对象存储控制面、集群控制面；
5. 非核心或边缘节点：活动页、帮助中心、静态资源站、孤立下载站、弱业务子站。

**典型可判为核心业务的场景**
1. 主域名首页、主站登录入口、统一认证页、统一用户中心；
2. 订单系统、支付系统、账户系统、结算系统、交易处理系统；
3. 核心业务 API 网关、统一鉴权中心、业务网关、主数据服务；
4. 多业务子系统共同依赖的配置中心、控制台、数据库管理面、对象存储控制面；
5. 被动流量、JS 调用和页面跳转共同证明其位于主流程依赖链中。

**典型不可直接判为核心业务的场景**
1. 仅与主域名命名相似，但没有调用、跳转、认证或页面语义支撑的站点；
2. 孤立活动站、招聘站、帮助中心、公开文档站、纯静态资源站；
3. 只有后台路径命中、但缺少登录链路、系统角色、依赖关系证据的页面；
4. 只有单一弱指纹或单一页面标题，无法证明其业务承载强度的资产。

**判定输出要求**
1. 核心业务判定结果至少应输出：是否属于核心业务、核心业务角色、关键链路位置、主要支撑证据、待复核项；
2. 若结论为“是”，至少应有两类及以上支撑信号，不宜全部来自同一弱证据源；
3. 若结论为“否”或“待复核”，应明确指出缺失了哪些关键支撑；
4. 判定结果必须能反向解释第三部分中的高价值结论，而不能与评分体系相互冲突。

**与价值评估结果的关系**
1. 核心业务判定不是单独加分项，不直接进入综合评分公式；
2. 该判定结果用于解释高价值资产“为什么高价值”，并辅助人工研判；
3. 当资产重要性高、业务关联性高且核心业务判定明确时，可强化高价值结论的稳定性；
4. 当综合分高但核心业务结论弱时，应在输出中增加复核提醒。

## 2.核心业务逻辑边界判定标准

**定义与设计目标**
    核心业务逻辑边界判定用于回答“核心业务结论可以扩展到哪里、不能扩展到哪里、当前证据能支持到什么程度”。该部分的重点不是继续抬高价值，而是避免把普通业务、边缘资产、临时站点或单点高危资产误判为核心业务目标。

**边界判定原则**
1. 仅因与主站属于同一主域、同一企业或同一证书主体，不代表一定属于核心业务；
2. 仅因 URL 中出现 `admin`、`manage`、`system`、`api`、`gateway` 等关键词，不代表一定属于核心业务；
3. 仅因命中高危组件、历史漏洞或单个风险线索，不得越过业务边界直接判为核心业务；
4. 仅因页面为登录页，不代表一定是统一认证或主业务入口，必须结合认证跳转链、同源关系和调用关系确认；
5. 仅因属于控制面或管理面技术栈，不代表一定与当前企业核心业务存在稳定依赖；
6. 只有在“业务语义 + 关系信号 + 系统角色信号”至少形成双重支撑时，才适合稳定判为核心业务；
7. 当前阶段允许使用代理信号推断边界，但必须同时输出证据完整度和待补证据项。

**可判入核心业务边界的条件**
1. 与主站、统一认证、核心 API、支付结算或关键数据面存在稳定调用、跳转、认证或依赖关系；
2. 具备明确的核心业务语义，且与企业主流程、主入口或关键支撑链路一致；
3. 被多个业务子系统共同依赖，或在架构中承担枢纽、控制、统一入口等角色；
4. 多源证据交叉验证一致，如页面语义、指纹角色、调用关系、流量痕迹共同支持。

**不可直接判入核心业务边界的条件**
1. 只有域名相似、命名相似或页面标题相似；
2. 只有单一弱代理信号，且缺少调用、跳转、认证或依赖关系支撑；
3. 只有风险线索高，但资产重要性和业务关联性均弱；
4. 只有通用框架、通用 CMS、普通业务子站特征，缺少核心角色语义；
5. 只有临时活动、展示、下载、帮助、文档类语义。

**边界层级建议**
1. 明确核心业务边界：证据充分，可直接判定为核心业务节点；
2. 疑似核心业务边界：存在较强代理信号，但仍缺少关键关系证据，需要待复核；
3. 非核心业务边界：当前证据不足，或已明确属于普通业务、边缘业务或静态资源节点。

**待复核触发条件**
1. 高价值评分与核心业务结论不一致；
2. 资产重要性高，但业务关联性证据弱；
3. 命中控制面或管理面技术栈，但缺少与主业务体系的稳定依赖关系；
4. 仅存在单一强信号，缺少第二类独立证据交叉验证；
5. 支撑信号和反对信号同时存在，无法稳定收敛。

**典型误判控制规则**
1. 不得把所有登录页都视为统一认证；
2. 不得把所有后台路径都视为核心管理面；
3. 不得把所有数据库、中间件、控制台类指纹都视为当前企业核心业务控制节点；
4. 不得把高危边缘资产因风险高而抬入核心业务边界；
5. 不得把无业务链路证据的同主域资产自动扩展为核心业务子系统。

**输出边界要求**
1. 边界判定必须输出“支撑信号”和“反对信号”两类信息；
2. 若结论为“明确核心业务边界”，应给出当前边界覆盖到的角色范围，如主入口、交易节点、支撑节点、控制节点；
3. 若结论为“疑似核心业务边界”，必须给出待补证据，不得伪装成确定结论；
4. 若结论为“非核心业务边界”，应明确说明是普通业务、边缘业务、静态资源还是证据不足；
5. 核心业务边界判定结果必须与第三部分的误判控制规则保持一致，不得自相矛盾。

# 五、优先级标记（排序算法）

## 1.设计目标

    本部分用于把第三部分形成的价值评分结果，转换为可执行的优先级顺序和资源倾斜建议。它解决的问题不是“这个资产分数是多少”，而是“多个资产同时出现时，应该先关注谁、先验证谁、哪些结果需要优先人工复核”。

    优先级标记必须坚持以下目标：
    1. 将价值分级结果转化为稳定、可重复、可解释的排序顺序；
    2. 区分“价值高”和“风险急”两个维度，避免单纯用风险高低替代价值排序；
    3. 让真正处于核心业务链路中的高价值资产稳定排在高位；
    4. 为边缘但高风险的资产保留处置优先级抬升空间，但不允许其直接压过真正的核心业务目标；
    5. 将证据完整度和复核约束纳入排序决策，避免系统对弱证据高分资产过度自信。

    本部分与其他章节的职责关系固定如下：
    1. 第三部分负责定义评分依据、权重分配和分值映射；
    2. 第四部分负责解释核心业务角色、关键链路位置和业务边界；
    3. 第五部分负责把评分结果与业务结论落地为排序规则和优先级建议；
    4. 第六部分和第七部分分别负责输入标准格式与输出标准格式，不在本部分展开。

    因此，本部分明确不负责以下内容：
    1. 不新增新的评分维度；
    2. 不改写第三部分既定评分公式；
    3. 不定义第六部分的输入字段表；
    4. 不展开第七部分的输出 JSON 模板。

## 2.排序算法规则

    优先级排序的基础仍然来自第三部分的综合评分体系，不能在本部分重新发明模型。排序时必须先使用统一公式完成价值分计算，再使用固定排序键生成最终顺序。

**排序基础公式**
1. 基础价值分 = 资产重要性 + 技术栈价值信号 + 业务关联性；
2. 风险修正分 = 风险线索；
3. 最终价值分 = `min(100, 基础价值分 + 风险修正分 + 证据完整度修正)`；
4. 其中，证据完整度修正只用于可信度修正，不用于替代基础价值判断。

**排序键规则**
1. 第一排序键：最终价值分，从高到低；
2. 第二排序键：风险线索分，从高到低；
3. 第三排序键：证据完整度修正，从高到低。

**排序键解释**
1. 最终价值分用于决定“总体上谁更值得优先关注”；
2. 风险线索分用于决定“在总体价值接近时，谁更值得先验证”；
3. 证据完整度修正用于决定“在前两项接近时，谁的结论更稳定、更适合优先投入资源”。

**并列打散规则**
1. 若前三排序键相同，优先将“核心业务结论更明确”的资产排在前面；
2. 若核心业务结论明确性仍相同，则优先“支撑证据更完整、冲突更少”的资产；
3. 若仍然相同，则按稳定顺序或 `asset_id` 保持确定性，避免每次排序结果波动。

**排序约束**
1. 不得因为单个高危漏洞或单条高危情报，让边缘资产直接压过真正的核心业务资产；
2. 不得让风险修正分覆盖资产重要性和业务关联性的基础判断；
3. 不得因为证据不足而强行将高分资产排到最前，弱证据高分资产必须结合复核标记一起输出；
4. 不得脱离第四部分的核心业务边界结论进行自由抬升排序。

## 3.优先级标记与资源倾斜规则

    排序结果最终需要落到可执行的优先级建议上，供渗透测试工程师和后续模块直接消费。因此，价值等级与优先级建议应形成稳定映射关系。

**价值等级与优先级建议映射**
1. 皇冠资产：优先验证；
2. 高价值资产：重点关注；
3. 重点关注资产：常规跟进；
4. 一般关注资产：低优先排队；
5. 普通资产：低优先保留。

**资源倾斜原则**
1. 核心业务结论明确、基础价值高且证据稳定的资产，应优先倾斜人工验证资源；
2. 高价值但风险线索弱的资产，应保持高关注等级，但不一定提升即时处置优先级；
3. 边缘资产但风险线索高的资产，可以抬升处置优先级，但不得抬入最高价值档；
4. 高分但证据不足、冲突明显或代理推断过强的资产，必须增加复核标记，不得作为稳定结论直接推动资源集中；
5. 风险高且业务价值明显偏低的资产，应在输出中体现“处置优先”和“价值等级受限”的双重结论。

**复核优先级触发场景**
1. 高分低证据；
2. 核心业务结论待复核；
3. 支撑信号与反对信号冲突；
4. 高风险但业务价值明显偏低；
5. 高价值结论主要依赖单一代理信号；
6. 高排序结果与第四部分核心业务边界结论不一致。

**资源建议输出原则**
1. 排在前列的资产必须说明是“价值高优先”还是“风险急优先”；
2. 若排序靠前的原因主要来自风险修正，应明确提醒其基础价值并不一定最高；
3. 若排序靠前的原因主要来自核心业务属性，应明确指出其业务角色，如主入口、交易节点、支撑节点、控制节点；
4. 若排序结果需要人工确认，应在优先级建议中同步写出“待复核”。

## 4.输出与误判控制要求

    本部分只规定排序结果必须具备的行为级要求，不展开第七部分的字段格式细节，但要确保后续输出可以稳定落地。

**排序结果至少应能表达的内容**
1. 排序位置；
2. 价值等级；
3. 最终价值分；
4. 排序原因；
5. 优先级建议；
6. 是否核心业务；
7. 是否需要复核。

**排序解释要求**
1. 排序原因必须能够说明主要抬升因子，如资产重要性高、核心业务明确、风险线索强、证据稳定；
2. 若排序靠前主要依赖风险线索，应明确说明其属于“风险驱动的处置优先”；
3. 若排序靠前主要依赖核心业务与业务关联性，应明确说明其属于“价值驱动的重点目标”；
4. 若排序靠前但证据完整度较差，应同步附带不稳定性提醒。

**误判控制要求**
1. 不能仅因高危漏洞把边缘资产推到最前；
2. 不能仅因同主域、路径命中后台关键词或页面标题可疑，就把资产认定为高优先级核心目标；
3. 不能忽略证据冲突，证据冲突时应优先降置信、加复核，而不是强行抬排位；
4. 不能把所有高风险资产都解释成高价值资产；
5. 不能让“代理信号推断”覆盖“明确反对证据”。

**与后续章节的边界**
1. 第五部分只规定排序逻辑、优先级建议和误判控制规则；
2. 第六部分再定义价值评估模块的输入标准格式；
3. 第七部分再定义价值评估模块的输出标准格式；
4. 本部分不写字段表、不写 JSON 模板、不重写评分公式。

# 六、输入标准格式

    本部分用于把第二部分定义的输入边界，收敛为价值评估模块可直接消费的结构化输入对象。它解决的问题不是“允许输入哪些信息类型”，而是“这些信息进入价值评估模块时，必须以什么字段结构、什么粒度、什么约束被表达”，以保证后续单资产评分、核心业务判定和多资产排序都能稳定复用同一套输入语义。

## 1.设计目标

    第六部分的设计目标固定如下：
    1. 将资产列表、指纹信息、风险线索、业务代理信号和证据完整度信息统一封装为标准输入对象；
    2. 明确区分“单资产评分输入”“核心业务判定输入”“多资产排序输入”三类处理模式；
    3. 避免上游模块直接把原始网页、原始流量、原始扫描日志整包塞给价值评估模块；
    4. 保证每一个评分、判定和排序结果都可以回溯到对应的输入证据；
    5. 保证后续提示词、RAG 检索、前端展示和报告输出可以共享同一套输入字段约定。

## 2.输入总原则

    价值评估模块的标准输入必须遵守以下原则：
    1. 只允许消费上游已经加工后的结构化信息，不直接消费未经整理的原始 HTML、原始 JS、原始流量包或扫描器完整日志；
    2. 输入粒度以“资产证据包”为中心，而不是以零散事件拼接为中心；
    3. 风险线索、技术栈、业务代理信号、证据完整度必须分层表达，禁止混在同一个自由文本字段中；
    4. 所有关键判断字段都必须具备 `evidence_ref`、`source` 或等价引用能力，不能出现无法回溯来源的结论性描述；
    5. 不允许把推测写进事实字段，推测性内容只能进入“代理信号”“说明”“待补证据”等受约束字段；
    6. 单资产评分与核心业务判定可以直接消费资产证据包；多资产排序原则上消费已完成的评分结果与核心业务结果，而不是重新消费原始证据。

## 3.统一输入封装

    价值评估模块的顶层输入对象建议统一为 `value_assessment_input`，格式如下：

```json
{
  "task_id": "",
  "assessment_mode": "",
  "input_version": "v1",
  "scope_note": "",
  "assets": [],
  "single_asset_results": [],
  "core_business_results": [],
  "upstream_refs": []
}
```

**顶层字段说明**

| 字段 | 是否必填 | 说明 |
| --- | --- | --- |
| `task_id` | 是 | 当前任务的唯一标识，用于关联上下游结果与审计链路 |
| `assessment_mode` | 是 | 处理模式，只允许为 `single_asset_score`、`core_business_judgement`、`batch_rank`、`full_assessment` |
| `input_version` | 是 | 输入格式版本，建议固定为 `v1` |
| `scope_note` | 否 | 对本次评估范围、批次说明、过滤条件的补充说明 |
| `assets` | 条件必填 | 资产证据包列表；在 `single_asset_score`、`core_business_judgement`、`full_assessment` 模式下必填 |
| `single_asset_results` | 条件必填 | 单资产评分结果列表；在 `batch_rank` 模式下必填，在 `full_assessment` 模式下可为空 |
| `core_business_results` | 条件必填 | 核心业务判定结果列表；在 `batch_rank` 模式下必填，在 `full_assessment` 模式下可为空 |
| `upstream_refs` | 否 | 上游任务、快照、证据集合、图谱节点等引用列表 |

**处理模式约束**
1. `single_asset_score`：`assets` 只能包含 1 个资产证据包，且不要求已有评分结果；
2. `core_business_judgement`：`assets` 只能包含 1 个资产证据包，且重点消费关系信号与业务代理信号；
3. `batch_rank`：原则上不重新读取原始证据，必须直接消费 `single_asset_results` 与 `core_business_results`；
4. `full_assessment`：允许一次输入多个资产证据包，并在模块内依次完成单资产评分、核心业务判定和多资产排序。

```json
    }
```

```json
  "core_business_results": [

      "asset_id": "asset_001",
      "是否核心业务": "是",
```

**多资产排序输入约束**
1. `single_asset_results` 与 `core_business_results` 中的 `asset_id` 必须一一对应；
2. 若存在评分结果而不存在核心业务结果，应在进入排序前补齐，或明确标记为“待复核”；
3. 排序输入不应重新注入大段原始证据，只保留评分摘要、关键证据摘要和复核标记；
4. 对于高分低证据、证据冲突、核心业务待复核的资产，排序输入中必须保留对应提醒，供最终排序结果继承。

## 8.落地约束

    为了保证第六部分可以稳定落地，输入标准格式必须满足以下工程约束：
    1. 所有字段名应固定，不允许每次调用时自由改写同义字段；
    2. `asset_id`、`task_id`、`evidence_ref` 必须可追溯，禁止生成无法反查的临时描述；
    3. 不允许把“高价值”“核心业务”“皇冠资产”这类最终结论预写进输入事实字段；
    4. 不允许把未经归并的原始日志长文本直接塞入 `note` 或 `page_semantics` 中充当结构化输入；
    5. 输入为空、字段缺失严重或 `assessment_mode` 非法时，应直接拒绝执行，而不是让模型自行猜测补齐；
    6. 第六部分只定义价值评估模块的输入对象，不重写第三部分评分标准，也不替代第九部分提示词。

# 七、输出标准格式

    本部分用于把第四部分和第五部分要求的输出能力，收敛为可直接落地的标准结构化结果。它解决的问题不是“价值评估模块大概要输出什么”，而是“单资产评分、核心业务判定、多资产排序最终必须以什么 JSON 结构输出，哪些字段是固定的，哪些标记必须显式给出”。

## 1.设计目标

    第七部分的设计目标固定如下：
    1. 统一单资产评分、核心业务判定和多资产排序三类结果的输出格式；
    2. 保证输出结果可直接被前端、报告模块、人工复核和审计链路消费；
    3. 保证所有关键结论都具备解释字段、证据摘要和复核标记；
    4. 禁止价值评估模块输出散文化长文，而应只输出受约束的结构化 JSON；
    5. 保证输出字段与第六部分输入字段、第三部分评分标准、第四部分核心业务边界和第五部分排序规则保持一致。

## 2.输出总原则

    价值评估模块输出必须遵守以下原则：
    1. 输出对象必须与 `assessment_mode` 一一对应，不能在单资产评分模式下输出多资产排序结果；
    2. 输出字段必须保留 `asset_id` 或 `task_id`，以保证上下游结果可以稳定串联；
    3. 数值字段应尽量保持可计算，如等级分、权重分、最终价值分应为可解析数值；
    4. 解释字段必须尽量短而明确，优先写证据、角色、原因，不写长段散文；
    5. 复核相关字段不得省略，尤其是高分低证据、证据冲突、核心业务待复核场景；
    6. 若输入证据不足，允许输出低置信或待复核结论，不允许伪造确定性结论。

# 九、提示词设计

    本部分用于固化价值评估模块的提示词设计，保证模型在执行价值评分、核心业务判定和多资产排序时，严格沿用本模块已定义的输入边界、评分标准和误判控制规则，不把自由发挥带入核心判定链路。

    本部分所有提示词统一遵守以下约束：
    1. 仅允许使用上游模块已经提供的资产列表、指纹信息、被动审查风险信息、代理信号和证据摘要，不得补造证据；
    2. 禁止把猜测写成事实，禁止把单一弱信号写成确定结论；
    3. 禁止因高危漏洞、单一风险线索或单条情报直接定义高价值资产；
    4. 禁止忽略证据完整度修正，信息不足时必须降置信、降分或标记待复核；
    5. 输出必须为结构化结果，不得写成散文化报告；
    6. 当出现“高风险但低证据”“高价值但证据冲突”“核心业务结论支撑不足”时，必须显式输出复核标记。

## 1.单资产价值评分提示词

    本提示词用于价值评估模块对单个资产进行价值评分。其职责是基于已经收敛好的输入证据，对单资产输出四项主因子分、证据完整度修正、最终价值分、价值等级和优先级建议。它不是漏洞验证器、不是主动探测器，也不是报告生成器。

    建议提示词正文如下：

```text
你是“价值评估模块”的单资产价值评分单元。你的唯一职责是基于上游已经提供的资产信息、指纹信息、风险线索和代理信号，对单个资产进行结构化价值评分。你不能主动探测，不能补充未提供的证据，不能把风险严重度直接等同于资产价值，不能把模糊推测写成确定事实。

你的处理目标如下：
1. 对单个资产完成资产重要性、技术栈价值信号、风险线索、业务关联性的四因子评分；
2. 根据证据完整度，对评分结果进行可信度修正；
3. 输出最终价值分、价值等级、优先级建议和简要解释；
4. 明确指出当前结论依赖了哪些证据、哪些地方仍需复核。

你必须遵守以下规则：
1. 只允许使用输入中已经存在的证据，不得自行补造资产角色、业务属性、系统关系或漏洞事实；
2. 资产重要性、技术栈价值信号、风险线索、业务关联性必须分别判断，禁止混写；
3. 风险线索用于动态修正优先级，不能替代基础价值判断；
4. 技术栈价值信号是双重因子，但最终输出时必须归并为一个统一分值；
5. 证据完整度修正不是新的价值维度，而是可信度控制项；
6. 若只有单一弱信号、历史模糊情报或缺少上下文印证，必须降分或标记待复核；
7. 若资产重要性小于等于2且业务关联性小于等于2，即使风险线索较高，也不得进入最高价值档；
8. 命中统一认证、支付、核心管理面、数据库管理面、云控制面、K8s控制面时，可触发基础价值保底规则，但仍需结合业务关联性和证据完整度；
9. 不得因为单个高危漏洞、单个组件命中或单条弱情报直接给出“皇冠资产”结论；
10. 所有判断都必须可以回溯到输入证据。

你的输入边界固定如下：
1. 资产列表：资产标识、域名/URL/IP、入口属性、页面语义、域名层级、基础类型；
2. 指纹信息：中间件、框架、CMS、组件、认证方式、控制面语义、接口路径语义；
3. 被动审查风险信息：漏洞情报、敏感接口、异常回显、弱口令痕迹、密钥/令牌暴露、配置异常；
4. 业务代理信号：跳转关系、调用关系、证书主题、页面文案、主子域关系、上下游依赖迹象；
5. 证据完整度信息：来源数量、证据一致性、是否多源交叉验证、是否存在冲突。

你的评分规则固定如下：
1. 基础价值分 = 资产重要性(0-30) + 技术栈价值信号(0-25) + 业务关联性(0-20)
2. 风险修正分 = 风险线索(0-20)
3. 证据完整度修正 = -10 ~ +5
4. 最终价值分 = min(100, 基础价值分 + 风险修正分 + 证据完整度修正)
5. 1-5分映射到权重分的方式固定为：
   - 1分 = 权重上限的20%
   - 2分 = 权重上限的40%
   - 3分 = 权重上限的60%
   - 4分 = 权重上限的80%
   - 5分 = 权重上限的100%
6. 价值等级固定如下：
   - 85-100：皇冠资产
   - 70-84：高价值资产
   - 50-69：重点关注资产
   - 30-49：一般关注资产
   - 0-29：普通资产

你在评分时必须重点检查以下误判风险：
1. 仅因页面含admin、manage、system等词，不得直接判定高价值；
2. 仅因域名与主站相似，不得直接判定核心业务；
3. 仅因历史高危组件存在CVE记录，但当前版本、暴露面或场景不明确，不得高分；
4. 仅因开放非常见端口，不得直接判定为核心控制面；
5. 仅因风险线索很高，不得覆盖资产重要性与业务关联性的基础约束。

    }
```

## 2.核心业务判定与边界提示词

    本提示词用于价值评估模块判断某个资产是否属于核心业务、是否位于关键链路，以及其核心业务边界在哪里。该提示词不负责最终排序，也不直接代替单资产价值评分，而是输出“核心业务角色判断”和“边界说明”，供后续价值解释与排序引用。

    建议提示词正文如下：

```text
你是“价值评估模块”的核心业务判定与边界单元。你的职责是根据已有资产信息、指纹信息、关系信号和风险上下文，判断该资产是否属于核心业务、是否位于关键链路，以及它的业务边界在哪里。你不负责最终排序，不负责主动探测，不负责补造业务关系。

你的处理目标如下：
1. 判断资产是否属于核心业务节点；
2. 判断资产在关键链路中的位置，如主入口、统一认证、核心API、管理面、控制面、关键依赖等；
3. 输出支持该判断的证据、反对该判断的证据和边界结论；
4. 在证据不足时明确标记“待复核”，而不是给出过度自信结论。

你必须遵守以下规则：
1. 只允许使用已有输入中的代理信号和证据，不得补造业务拓扑；
2. 仅因域名相似、命名相似或页面标题可疑，不得直接判为核心业务；
3. 若仅存在单一弱代理信号，通常结论上限不超过中等；
4. 若存在认证跳转、主站依赖、统一网关、核心API调用、关键上下游关系等多源证据，可提高置信度；
5. 若支持信号与反对信号同时存在，必须保留冲突说明，不得强行下结论；
6. 若证据不足以证明其处在关键链路中，必须输出“待复核”或“非核心业务暂定”；
7. 所有判断必须可回溯到具体路径语义、调用关系、证书主题、页面语义、认证流程或上下游依赖证据。

你的主要判定依据如下：
1. 主站、统一入口、统一认证、统一用户中心、核心API、支付结算、账户交易；
2. 控制面、管理后台、数据库管理面、云控制台、CI/CD控制面、K8s控制面；
3. 关键数据面、对象存储控制面、配置中心、网关、服务治理、监控平台；
4. 主流程依赖服务、多个业务子系统共用的关键上游或下游依赖；
5. 业务中台、数据中心、统一平台、全局跳板、关键枢纽节点。

你的代理信号来源固定如下：
1. 域名层级与主子域关系；
2. URL路径语义，如/api/、/sso/、/auth/、/gateway/、/admin/；
3. 页面中的登录形态、统一认证跳转链、同源关系；
4. JS/API调用关系与主域接口引用；
5. 页面文案中的“统一平台”“管理中心”“控制台”“业务中台”“数据中心”等语义；
6. 证书主体与企业主体、主站命名体系的匹配关系；
7. 文件情报、被动流量和上下游调用中的稳定依赖线索。

你的边界判定规则如下：
1. 仅因属于同一主域，不代表一定是核心链路节点；
2. 仅因命中后台语义词，不代表一定是管理面或控制面；
3. 没有调用、跳转、认证、页面语义或上下游依赖支撑时，不得判高关联；
4. 核心业务结论必须结合支持信号和反对信号共同输出；
5. 当支撑不足、证据冲突或只有单一弱信号时，必须输出待补证据项。

当你完成处理后，只允许输出一个结构化JSON结果，格式如下：
{
  "asset_id": "",
  "是否核心业务": "",
  "核心业务角色": "",
  "关键链路位置": "",
  "边界结论": "",
  "支撑信号": [],
  "反对信号": [],
  "置信度": "",
  "待补证据": [],
  "复核建议": ""
}

补充要求如下：
1. “是否核心业务”只允许输出：是、否、待复核；
2. “关键链路位置”应尽量使用固定角色语义，如主入口、统一认证、核心API、管理面、控制面、关键依赖、边缘节点；
3. 若结论为“是”，支撑信号至少应有两类及以上，且不能全部来自同一弱证据源；
4. 若结论为“否”，也应说明为何当前证据不足以支持其进入核心业务链；
5. 不允许输出额外解释性散文。
```

# 十、关于价值判断的市场调研
## 1. 调研结论概览
    从开源工具、开放标准和公开方法的现状看，真正直接做“资产价值判断”的开源工具较少。主流做法通常不是单独计算“价值分”，而是将资产关键性、技术语义、风险上下文、业务/拓扑关系组合后再做排序。因此，市场上更成熟的不是“单一资产价值模型”，而是“多因子综合研判框架”。

    对 heuScan 而言，当前模块输入固定为：资产列表、指纹信息、被动审查的风险信息。若以这三类输入为边界，则它们与市场主流方法的对应关系可以收敛为：
    1. 资产列表：对应资产范围、暴露面、基础类型、入口属性；
    2. 指纹信息：对应技术栈、系统角色、业务语义、控制面属性；
    3. 被动审查的风险信息：对应风险线索、被利用上下文、动态优先级修正。

    在没有主动攻击数据、没有完整 CMDB、没有全量业务台账的前提下，价值判断不能等待“绝对准确的业务真值”，而必须采用业务代理信号。所谓业务代理信号，是指通过域名层级、URL 命名、页面元素、指纹组件、认证方式、API 结构、证书主题、JS 调用链等外部可观测事实，推断资产在业务体系中的角色与重要性。

    结合现有市场方法和 heuScan 的输入边界，本次调研更适合落地为以下判断骨架：
    1. 资产重要性是底座，用于回答“这个资产本身是否值得优先关注”；
    2. 技术栈是双重因子，既参与业务价值推断，也参与风险修正；
    3. 风险线索决定动态抬升，用于回答“现在是否应该优先处理”；
    4. 业务关联性决定是否进入核心链路，用于回答“它是不是关键链路中的节点”；
    5. 证据完整度只作为校正项，用于防止在信息不足时给出过高置信度结论。

    因此，本节市场调研的目标不是寻找一个现成开源资产价值算法直接照搬，而是从公开标准中抽取可复用的价值依据，再映射到 heuScan 当前可获取的输入证据。

## 2. 相关开源工具与开放方法对比
| 名称 | 类型 | 与 heuScan 输入的对应关系 | 可借鉴的价值依据 | 对评分算法的启发 | 局限 |
| --- | --- | --- | --- | --- | --- |
| [NIST FIPS 199](https://nvlpubs.nist.gov/nistpubs/FIPS/NIST.FIPS.199.pdf) + [NIST SP 800-60 Vol.1 Rev.1](https://csrc.nist.gov/pubs/sp/800/60/v1/r1/final) | 官方分类标准 | 可映射资产列表中的资产类型、功能角色、数据处理场景 | 提供资产重要性、数据/功能敏感度的底座分级思路 | 适合把“低/中/高影响”转成 1-5 离散评分，再映射到权重分 | 不直接处理互联网暴露面、技术栈和风险线索 |
| [OWASP Risk Rating Methodology](https://owasp.org/www-community/OWASP_Risk_Rating_Methodology) | 开放风险评估方法 | 可映射被动风险信息与部分指纹信息 | 强调 Likelihood 与 Impact 分离，技术影响与业务影响分离 | 适合作为“基础价值”和“风险修正”分层算法的依据 | 更适合风险项研判，不直接定义资产价值 |

| [BloodHound](https://github.com/SpecterOps/BloodHound) / [ImproHound](https://github.com/improsec/ImproHound) / [KubeHound](https://github.com/DataDog/KubeHound) / [Cartography](https://github.com/cartography-cncf/cartography) | 关系放大与控制面价值类工具 | 可映射指纹信息中的系统角色、控制面属性，以及资产列表中的层级关系 | 强调控制面权限、横向影响、依赖关系、图谱中心性会放大资产价值 | 适合作为“业务关联性”和“高价值角色门槛”的设计来源 | 对 heuScan 当前阶段而言，很多关系只能用代理信号而非完整图谱获得 |
| [CVSS v4.0](https://www.first.org/cvss/v4.0/specification-document) / [EPSS](https://www.first.org/epss/model) / [SSVC](https://certcc.github.io/SSVC/reference/decision_points/) | 风险修正辅助框架 | 主要映射被动审查风险信息，也可部分映射高危指纹 | 提供严重度、被利用概率、处置决策上下文 | 适合构造风险修正分，不适合直接定义资产价值 | 如果直接当价值模型使用，会把高危边缘资产抬得过高 |

    从市场调研结论看，资产重要性和风险严重度通常不是一个维度；技术栈在通用框架中常被隐含处理，但对 heuScan 这种指纹驱动系统必须提升为显式因子；图谱关系在当前阶段不必强依赖完整图库，可以先收敛为“业务关联性代理信号”。

## 3. 对价值依据与价值标准设计的可借鉴思路
    结合 heuScan 当前输入边界，更适合的方案不是继续平铺更多维度，而是以四项核心依据为中心，再补一个校正项，形成 5 维价值标准：
    1. 资产重要性；
    2. 技术栈价值信号；
    3. 风险线索；
    4. 业务关联性；
    5. 证据完整度修正。

    其中四项核心依据的落地方式如下：
    1. 资产重要性：根据资产类型、暴露位置、是否主站、是否统一入口、是否管理面、是否核心 API、是否关键数据面进行 1-5 分。其输入来源主要是资产列表及基础识别结果；
    2. 技术栈价值信号：定义为双重因子。第一层是业务价值推断，例如 ERP、OA、统一认证、支付、中台、对象存储、CI/CD、K8s、云控制台、数据库管理面通常意味着更高价值；第二层是风险暴露推断，例如高危中间件、老旧框架、公开后台、弱鉴权组件会提高优先级。其输入来源主要是指纹信息；
    3. 风险线索：根据被动流量审查、文件情报、已知漏洞、敏感接口、弱口令痕迹、可疑外联、高危配置痕迹等线索进行 1-5 分。这里的风险线索只做动态修正，不单独定义资产业务价值；
    4. 业务关联性：根据资产与核心域名、主站、统一认证、核心 API、重要子系统、敏感文件、上下游依赖的关系强度进行 1-5 分。在没有完整图谱时，允许使用 URL 命名、域名层级、页面元素、JS/API 调用、证书主题、企业命名等代理信号推断；
    5. 证据完整度修正：不是新的价值依据，而是评分可靠性控制项。信息少、指纹弱、来源单一则下调；多源交叉验证则小幅上调。

    这一设计与前文的四项核心依据是一致的：资产重要性、技术栈、风险线索、业务关联性仍然是主体，只是额外引入了“证据完整度修正”，用于抑制信息不完整时的过度自信，而不是另起一套新的价值体系。

| 维度 | 来源输入 | 判定标准 | 分值范围 | 说明 |
| --- | --- | --- | --- | --- |
| 资产重要性 | 资产列表、基础识别结果 | 主站、统一入口、管理面、核心 API、关键数据面得分更高 | 1-5 | 决定基础价值底座 |
| 技术栈价值信号 | 指纹信息 | 同时考虑业务价值语义和技术风险暴露 | 1-5 | 是双重因子，不是单纯风险因子 |
| 风险线索 | 被动审查风险信息、部分高危指纹 | 根据严重度、可信度、利用迹象、敏感行为线索评分 | 1-5 | 只做动态修正 |
| 业务关联性 | 资产列表、指纹信息、代理关系信号 | 与主站、认证、核心 API、上游下游关系越强，分数越高 | 1-5 | 决定是否处在核心链路 |
| 证据完整度修正 | 多源证据交叉情况 | 证据越多、来源越一致，修正越高 | -10~+5 | 只修正置信度，不重定义价值 |

## 4. 对价值评分生成算法的建议
    从市场主流方法抽象到 heuScan，可将算法设计为“输入映射 -> 因子计算 -> 总分公式 -> 分档规则 -> 排序策略”的五段式流程。

    第一，输入到评分特征的映射如下：
    1. 资产列表产出：资产类型、暴露层级、入口属性、域名层级、管理面特征；
    2. 指纹信息产出：技术栈标签、系统角色标签、控制面标签、技术风险标签；
    3. 被动审查风险信息产出：风险事件数、严重度、可信度、利用迹象、敏感行为线索。

    第二，因子计算统一采用 1-5 分离散评分，再映射到各维度权重上限：
    1. 1 分映射 20%；
    2. 2 分映射 40%；
    3. 3 分映射 60%；
    4. 4 分映射 80%；
    5. 5 分映射 100%。

    第三，建议采用如下评分公式：

    基础价值分 = 资产重要性(0-30) + 技术栈价值信号(0-25) + 业务关联性(0-20)

    风险修正分 = 风险线索(0-20)

    证据修正分 = 证据完整度(-10 ~ +5)

    最终价值分 = min(100, 基础价值分 + 风险修正分 + 证据修正分)

    其中，技术栈价值信号内部再拆成两个子项，但总上限仍然控制在 25 分以内：
    1. 技术栈业务价值子分：用于判断它是不是 ERP、OA、统一认证、支付、中台、CI/CD、云控制台、数据库管理面等高价值角色；
    2. 技术栈风险暴露子分：用于判断它是否对应高危中间件、老旧框架、公开后台、弱鉴权组件等高风险暴露。

    第四，门槛规则与分档规则建议如下：
    1. 如果资产重要性 <= 2 且业务关联性 <= 2，即使风险线索高，也不得直接进入最高价值档；
    2. 如果命中统一认证、支付、核心管理面、云控制面、K8s 控制面、数据库管理面等高价值角色，基础价值至少进入中高档；
    3. 如果风险线索缺失，但资产重要性和业务关联性同时较高，仍可判为高价值资产，只是不提升处置优先级；
    4. 85-100：皇冠资产；
    5. 70-84：高价值资产；
    6. 50-69：重点关注资产；
    7. 30-49：一般关注资产；
    8. 0-29：普通资产。

    第五，排序策略建议如下：
    1. 第一排序键：最终价值分；
    2. 第二排序键：风险线索分；
    3. 第三排序键：证据完整度；
    4. 输出时同时给出价值等级、价值分、优先级建议、证据说明。

    这种算法设计的核心理由是：
    1. 不直接照搬 CVSS，因为 CVSS 表达的是漏洞严重度，不是资产业务价值；
    2. 不只做线性漏洞加权，否则会让高危但低价值的边缘资产压过真正关键的业务资产；
    3. 将基础价值和风险修正分层，可以同时满足“价值分级”和“优先级排序”的双重目标。

| 资产类型样例 | 资产重要性 | 技术栈价值信号 | 风险线索 | 业务关联性 | 证据完整度 | 预期结果 |
| --- | --- | --- | --- | --- | --- | --- |
| 主站/统一认证类 | 高 | 高 | 中 | 高 | 高 | 即使风险线索一般，也应进入高价值或皇冠资产 |
| 普通业务子站但带高危线索 | 中 | 中 | 高 | 中 | 中 | 可进入重点关注或高价值资产，但通常不应压过统一入口 |
| 边缘服务/低业务关联类 | 低 | 中或低 | 高 | 低 | 中 | 可获得较高处置优先级，但不应直接进入最高价值档 |
