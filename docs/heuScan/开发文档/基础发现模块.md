# 一、模块定位与任务描述

基础发现模块（Asset Discovery Agent / 基础发现模块）的目标：围绕“用户给定的种子目标（域名 / IP / CIDR / URL）”，在严格的范围约束（scope）与低侵扰默认策略下，完成 **子域发现、DNS 解析、CDN 识别、端口开放探测、HTTP(S) 服务确认、最小可审计样本留存**，并以统一的 `ObservationEvent + EvidenceRef` 形式进入一期聚合后端的统一数据面（Event Bus / State Store），供指纹识别、JS 解析、图谱聚合、价值评估等下游模块消费。

- 范式：**REACT**
- 技术栈：**Go 执行器 + Python 适配器 / Agent 封装**
- 聚合后职责：**子域、IP、端口、CDN、HTTP 服务确认**
- 在聚合后端中的位置：**网络资产发现模块**

## 1.1 输入目标与典型场景

- 输入线索（来自总控标准化后的 TaskContract/AgentTask）：
  - `domain`：根域或子域（例：`example.com`、`api.example.com`）
  - `ip`：单个 IP（例：`1.2.3.4`）
  - `cidr`：网段（例：`1.2.3.0/24`，默认会做规模限制）
  - `url`：完整 URL（例：`https://example.com:8443/`，仅用于提取 host/port/协议）
- 输出目标：
  - 资产实体：domain / ip / port / url（服务）
  - 关系线索：domain→ip、ip→port、url→service
  - 最小证据：DNS 记录、端口探测输出摘要、HTTP 响应头/标题/证书摘要、截断后的正文片段引用

## 1.2 边界、约束与默认策略（必须写死，避免 AI 自行发挥）

- 授权约束：仅用于授权场景下的信息收集与资产盘点，不执行漏洞利用与破坏性操作。
- 低侵扰默认策略（一期默认 profile）：
  - 子域：优先被动发现 + 小规模补充（不做大规模暴力枚举）
  - 端口：默认仅扫描“常见 Web 端口集”（见 3.3），不默认全端口
  - HTTP：只做服务确认与最小采样（`GET /` + headers + 64KB 截断）
- 范围约束（scope）默认严格：
  - `allow_third_party=false`：不自动扩展到第三方外联域名；任何“新域名”必须满足 allowed_domains 或属于同根域
  - `deny_domains/deny_ips` 优先级高于 allow
- 预算与上限：
  - 达到 `time_budget_s` / `max_assets` / `max_events` 任一上限：必须停止继续探测，输出 `partial` + `LIMIT_REACHED` warnings

# 二、职责拆解

## 2.1 核心职责

1. 输入规范化：将 domain/ip/cidr/url 规范化为统一 `TargetSeed` 集合并去重
2. 子域发现（domain 种子）：产生候选子域列表（candidate），并按 scope 过滤
3. DNS 解析：解析 A/AAAA/CNAME，输出可审计的 DNS 证据
4. 端口探测：对解析得到的 IP（及用户给定 IP/CIDR）做受控端口扫描（常见 Web 端口集）
5. HTTP(S) 服务确认：对开放端口进行 httpx 探测，确认可访问 URL、状态码、标题、重定向链、TLS 摘要
6. 路径探测（dirsearch）
7. 最小样本留存：保存 headers + 截断 body（≤64KB）+ TLS 摘要引用，供指纹识别/JS 解析/流量审查复用
8. 事件化输出：所有结果以 `ObservationEvent` 发往 Event Bus / State Store，禁止直接写 Neo4j

## 2.2 非职责（明确不做）

- 不做指纹识别（由 Fingerprint Agent 负责）
- 不做 JS AST/语义解析（由 JS Semantic Agent 负责）
- 不做文件下载与敏感信息抽取（由 File Intel Agent 负责）
- 不做被动流量审查与漏洞线索判定（由 Passive Traffic Audit Agent 负责）
- 不做价值评估/攻击建议/报告输出（由 Value Assessment / Chain Hint / Report Agent 负责）
- 不做图谱聚合建模与 Neo4j 写入（由 Graph & Topology Agent 负责）

## 2.3 与其他 Agent/模块的职责边界（避免覆盖与混淆）

- 与指纹识别模块：
  - 本模块产出：`HTTP_SERVICE_FOUND` + `HTTP_SAMPLE_STORED`
  - 指纹模块消费这些样本并产出技术栈 Finding
- 与 JS 解析模块：
  - 本模块只做“最小样本留存”；不得在本模块做 AST/接口提取
  - 可选输出 `JS_URL_CANDIDATE`（仅当样本中解析出 script src），但不做深度解析
- 与扩展词典模块：
  - 本模块提供“已确认活跃 URL/服务”的事实底座，扩展词典据此生成更大词典并回流到后续探测（不在本模块发生）
- 与图谱聚合模块：
  - 本模块只输出资产事实与 evidence；跨域关系拼装、拓扑展示与图谱写入由图谱聚合模块统一完成

## 3.4 结构化类型（可生成代码）

### 3.4.1 `AssetRef`

```json
{
  "asset_id": "url:https://example.com:443/",
  "asset_type": "url",
  "value": "https://example.com:443/",
  "normalized": "https://example.com/",
  "parents": ["domain:example.com", "ip:1.2.3.4", "port:1.2.3.4:443"]
}
```

建议 `asset_id` 规范（必须稳定，便于幂等与去重）：

- domain：`domain:{fqdn}`
- ip：`ip:{ipv4_or_ipv6}`
- port：`port:{ip}:{port}`
- url（服务级）：
  url:{scheme}://{host}:{port}/
- url（路径级）：
  url:{scheme}://{host}:{port}{normalized_path}
  说明：
- 服务级 URL（/）用于表示站点入口
- 路径级 URL（/admin /api 等）作为独立资产存在

### 3.4.2 `HttpResponseSampleRef` / `EvidenceRef`

EvidenceRef 最小字段集（用于可审计与回放）：

```json
{
  "evidence_id": "ev-uuid",
  "task_id": "uuid",
  "source": "httpx",
  "query": "https://example.com:443/",
  "fetched_at": "2026-04-08T12:00:00Z",
  "payload_ref": "minio://raw/{task_id}/httpx/2026-04-08/{evidence_id}.json",
  "digest": "sha256:...",
  "http_status": 200,
  "cost_ms": 1200
}
```

MinIO 路径规范建议：

- `raw/{task_id}/{tool}/{yyyy-mm-dd}/{evidence_id}.json`

# 四、统一事件模型（ObservationEvent）与事件白名单

对齐 `docs/heuScan/项目介绍与功能设计/2.第一阶段多Agent功能拆解与协同架构设计.md` 的事件字段，本模块发出的 `ObservationEvent` 必须包含至少：

- `task_id`、`agent_id`、`event_type`、`payload_ref`、`confidence`、`priority`、`scope`、`lineage`、`dedupe_key`、`retry_count`、`timestamp`

## 4.1 事件类型白名单（禁止发明新类型）

- `DISCOVERY_SEED_NORMALIZED`
- `SUBDOMAIN_CANDIDATE_FOUND`
- `DNS_RESOLVED`
- `IP_ASSET_FOUND`
- `PORT_OPEN`
- `HTTP_SERVICE_FOUND`
- `HTTP_SAMPLE_STORED`

## 4.3 `dedupe_key` 规则（必须稳定）

建议格式（示例）：

- `SUBDOMAIN_CANDIDATE_FOUND`：`{task_id}:subdomain:{candidate}`
- `DNS_RESOLVED`：`{task_id}:dns:{domain}:{sha256(a+aaaa+cname)}`
- `PATH_FOUND`：`{task_id}:path:{normalized_url}:{status_code}`
- `PORT_OPEN`：`{task_id}:port:{ip}:{port}`
- `HTTP_SERVICE_FOUND`：`{task_id}:http:{final_url}:{status_code}`
- `HTTP_SAMPLE_STORED`：`{task_id}:sample:{final_url}:{digest}`

# 五、成熟工具链与 Adapter 设计（一期默认）

> 原则：Agent 不直接 shell 调工具；必须通过 `adapters/` 封装，便于限流、审计、mock 测试与跨平台。

## 5.1 默认工具链（一期写死）

- 子域：oneforall（启用爆破）
- DNS：dnsx
- IP扫描：masscan
- 端口：naabu
- HTTP：httpx
- 路径：dirsearch
- 路由：mtr

## 5.2 Go 执行器 stdin/stdout 协议（建议）

Go 执行器作为“工具编排器”，接受 JSON 输入，输出 JSON 结果；Python Adapter 负责：

- 参数校验与默认值填充
- scope 过滤与去重
- 结果标准化为事件 payload，并写入 Evidence 存储

stdin（示例）：

```json
{
  "task_id": "uuid",
  "tools": ["oneforall", "dnsx", "naabu", "httpx", "dirsearch"],
  "targets": ["example.com"],
  "ports": [80, 443, 8080, 8443],
  "limits": { "time_budget_s": 300, "max_concurrency": 50 }
}
```

stdout（示例）：

```json
{
  "subdomains": ["api.example.com"],
  "dns": [{ "domain": "api.example.com", "a": ["1.2.3.4"], "cname": ["..."] }],
  "ports": [{ "ip": "1.2.3.4", "port": 443, "open": true }],
  "http": [
    {
      "url": "https://api.example.com/",
      "status_code": 200,
      "title": "Example"
    }
  ]
}
```

## 5.3 Python Adapter / Protocol（建议直接照此生成代码）

```python
class ToolExecutor(Protocol):
    def run_oneforall(self, domain: str, *, timeout_s: int) -> list[str]: ...
    def run_dnsx(self, domains: list[str], *, timeout_s: int): ...
    def run_masscan(self, cidr: str, *, timeout_s: int): ...
    def run_naabu(self, ips: list[str], *, timeout_s: int): ...
    def run_httpx(self, urls: list[str], *, timeout_s: int, max_body_bytes: int): ...
    def run_dirsearch(self, url: str, *, timeout_s: int): ...


class EvidenceStore(Protocol):
    def put_json(self, task_id: str, source: str, payload: dict) -> "EvidenceRef": ...
```

# 六、证据与样本存储（统一数据面）

## 6.1 存储落点（一期默认）

- 原始输出/回包：MinIO（对象存储）
- Evidence 元数据：PostgreSQL（便于索引、检索、回放）
- 事件流：Event Bus / State Store（持久化事件与状态）

## 6.2 裁剪策略（必须写死）

- 单样本正文最大：`max_body_bytes=65536`（64KB）
- 允许路径探测生成新的 URL
- 路径扫描结果作为独立资产存储
- 重定向上限：`max_redirects=3`
- 样本总量上限：`max_total_samples` 达到后停止存样本，但仍可发 `HTTP_SERVICE_FOUND`

# 七、错误处理、告警与降级策略（必须写死）

## 7.1 warnings 工程化格式（统一）

```json
{
  "code": "TOOL_UNAVAILABLE",
  "severity": "error",
  "message": "naabu 不可用或执行失败",
  "retryable": true,
  "hint": "检查 Go 执行器或工具安装状态",
  "evidence_refs": []
}
```

## 7.2 warning code 白名单（一期至少覆盖）

- `TARGET_INVALID`
- `OUT_OF_SCOPE`
- `DNS_TIMEOUT`
- `TOOL_UNAVAILABLE`
- `RATE_LIMITED`
- `TIME_BUDGET_EXCEEDED`
- `ASSET_LIMIT_REACHED`
- `EVENT_BUDGET_EXCEEDED`
- `HTTP_SAMPLE_TRUNCATED`
- `LIMIT_REACHED`
- `PATH_SCAN_FAILED`

## 7.3 重试/超时/降级（默认必须固定）

- 单工具默认超时：`timeout_s=30`（可按工具覆盖：dns/http 更小）
- 重试：最多重试 1 次；仅对“工具执行失败/超时/临时 IO 错误”重试
- 对 `429/风控`：不重试，立即降级（降低并发/QPS 或跳过该工具）并输出 warning
- 任一工具失败不得导致整任务失败：应输出 partial + warnings，由总控决定是否补跑/重试

# 八、测试计划与验收标准

## 8.1 单元测试（建议覆盖）

- 目标规范化：domain/url/ip/cidr 的校验与归一
- scope 判定：allowed/deny/third_party 过滤逻辑
- `asset_id` 与 `dedupe_key` 稳定性（同输入多次运行一致）
- evidence 裁剪：正文截断与 `HTTP_SAMPLE_TRUNCATED` warning
- 事件 payload schema 校验：event_type 白名单与字段完整性

## 8.2 集成测试（建议覆盖）

- 使用固定目标夹具（本地域名/IP）跑通：oneforall → dnsx → naabu → httpx
- 校验事件序列：至少出现 `DNS_RESOLVED/PORT_OPEN/HTTP_SERVICE_FOUND/HTTP_SAMPLE_STORED/DISCOVERY_BATCH_DONE`
- 校验去重：重复运行不产生重复事件风暴
- 路径扫描测试：验证 dirsearch 工具是否正常生成 `PATH_FOUND` 事件
- 路径扫描限制：验证路径扫描受限于 `max_assets` 配置，不会超出资产上限
- IP 扫描协同：验证 masscan + naabu 协同是否正常工作

## 8.3 回归测试（建议覆盖）

- 预算/上限触发：输出 `partial`，并包含 `TIME_BUDGET_EXCEEDED/ASSET_LIMIT_REACHED/EVENT_BUDGET_EXCEEDED`
- scope 收紧：禁止扩展 third-party 时不输出越界域名（并给 `OUT_OF_SCOPE` warnings）

## 9.3 规范化与 ID 生成（必须一致）

- domain 规范化：去协议、去路径、去端口、转小写（`https://Example.com/a` → `example.com`）
- URL 规范化：
  - 服务级：保留 scheme/host/port 与 `/`
  - 路径级：保留规范化后的路径（用于路径资产）
- CIDR 限制：默认最大可展开主机数 `max_cidr_hosts=4096`；超过则 `TARGET_INVALID` 或 `LIMIT_REACHED`（由总控决定是否允许）
- `asset_id` 规则见 3.4.1（必须稳定）

路径规范化规则（新增）：

- 去除 query 参数（?a=1）
- 去除 fragment（#xxx）
- 空路径统一为 "/"
- 统一末尾斜杠（/admin → /admin）
- 可选：统一小写

示例：
原始：
https://example.com/admin?id=1

规范化：
https://example.com/admin

## 9.4 触发阶段建议（供 Orchestrator）

- 全局状态机进入 `DISCOVERING` 阶段时触发基础发现
- 每批输出 `DISCOVERY_BATCH_DONE`，由总控决定是否：
  - 触发指纹识别（对新增 URL）
  - 触发扩展词典（基于新增活跃服务）
  - 或继续扩展（提升 profile/补跑端口范围）
