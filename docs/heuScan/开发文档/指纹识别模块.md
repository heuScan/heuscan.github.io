# 一、模块定位与任务描述

指纹识别模块（Fingerprint Agent）的目标：围绕“已发现的网络资产（域名/IP/端口/URL）与其 HTTP/HTTPS 交互证据”，对前后端技术栈、中间件、组件、CMS、框架等进行**可审计、可去重、可回放**的指纹识别，并以结构化 `Finding + EvidenceRef` 产出 `ObservationEvent`（由 Event Bus / State Layer 负责落库进入统一数据面），供 Orchestrator 聚合为任务快照（TaskSnapshot）、供扩展词典/价值评估/图谱聚合等下游模块消费。


## 1.1 输入目标与典型场景

- 输入线索（来自总控规范化后的任务契约与上游事件）：
  - 资产：`domain` / `ip` / `port` / `url`（上游通常由基础发现模块产出）
  - 证据：HTTP 响应样本、HTML 片段、JS 引用、TLS 握手信息、服务 Banner（来自被动流量留存或上游模块已采集的样本）
- 输出目标：
  - 技术栈标签：如 `nginx`、`Apache`、`Spring Boot`、`Django`、`Vue`、`WordPress`、`Shiro`、`Jenkins` 等
  - 组件细粒度识别：如 `Shiro`、`Swagger UI`、`Actuator`、`Druid`、`Grafana`、`Kibana`、`Struts2` 等
  - 版本/构建信息（若证据足够且可稳定提取）
  - （建议）风险提示线索（仅“线索”，不做漏洞验证），例如：
    - `Apache Shiro` → `shiro_550`、`rememberMe`（低置信提示，需复核/下游验证）

## 1.2 边界、约束与默认策略

- 仅用于授权场景下的信息收集与技术栈识别，不执行漏洞利用与破坏性操作。
- 不把推测写成事实：所有识别结果必须附 `EvidenceRef`；若仅基于弱特征（例如单个 `Server` 头）则必须降低 `confidence` 并输出 `warnings`。
- 受控取样与证据裁剪（默认值必须写死，避免 AI 自行发挥）：
  - 单资产最多取样：`max_samples_per_asset=3`
  - 单响应最多保留正文：`max_body_bytes=65536`（64KB），仅用于指纹匹配与证据片段
  - 单资产指纹预算：`time_budget_s_per_asset=10`
- 幂等与去重：
  - 同一任务内，同一资产 + 同一组件 + 同一版本/候选，必须生成稳定 `dedupe_key`，避免事件风暴与重复下游触发。
- 结果冲突处理：
  - 允许同一资产识别到多个组件（例如 `nginx + Spring Boot + Vue`）
  - 若出现“同一组件多个版本候选”，必须输出 `candidates[]` 并标注冲突原因；不做强行合并。

# 二、职责拆解

## 2.1 核心职责

1. 证据消费：只从上下文/存储读取已有 `HttpResponseSample`（来源：基础发现模块抓取留存 / 被动流量留存等）；若缺失则输出 `EVIDENCE_INSUFFICIENT`（并附 `need_more_evidence=true` 的 warning 事件），交由 Orchestrator 补跑上游采样模块，本模块不主动抓取。
2. 指纹匹配：基于头、Cookie、HTML、JS、路径特征、TLS/证书、服务 Banner 等多维特征匹配指纹库。
3. 版本提取与归一：将原始版本字符串规范化为可比较形式（例如 `v1.2.3` → `1.2.3`），并标注提取来源与置信度。
4. 合并与去重：对同资产多证据、多 matcher 输出进行合并，产出稳定的 `Finding` 与 `ObservationEvent`。
5. 输出可审计证据链：所有结论必须可回溯到 `EvidenceRef`（原始回包/片段/位置）。
6. 结构化输出：输出 `findings + evidence + warnings + stats`，供 Result Aggregator 与下游模块消费。

## 2.2 非职责（明确不做）

- 不做漏洞验证/利用（例如不主动触发 Shiro 漏洞探测、不发 payload）。
- 不做目录爆破/路径扫描（由基础发现/扩展词典/扫描模块负责）。
- 不直接写 Neo4j 图谱（由图谱聚合模块统一建模与写入）；本模块仅产出可关联的结构化发现。
- 不做前端渲染与报告排版（由 Result Aggregator + 报告输出模块负责）。

## 2.3 与其他 Agent/模块的职责边界（避免覆盖与混淆）

结合 `docs/heuScan/项目介绍与功能设计/1.项目介绍与解决痛点.md` 与一期聚合架构：

- 与“基础发现模块（Asset Discovery Agent）”边界：
  - 基础发现负责：子域/IP/端口/URL 发现与基础探测。
  - 指纹识别负责：对“已发现的资产”进行技术栈识别，并输出结构化标签与证据。
- 与“JS 解析模块（JS Semantic Agent）”边界：
  - JS 解析负责：深度语义（API/参数/token/外联域名等）。
  - 指纹识别仅做：JS 文件名/路径/特征字符串的轻量指纹匹配（例如 `webpackJsonp`、`__NEXT_DATA__`、`/static/js/main.*.js`）。
- 与“流量审查模块（Passive Traffic Audit Agent）”边界：
  - 流量审查负责：风险线索（未授权/CORS/错误栈等）。
  - 指纹识别可消费其留存的 HTTP 样本，但不输出漏洞结论，仅输出“技术栈/组件/可能风险线索标签”。
- 与“扩展词典模块（Dynamic Dictionary Agent）”边界：
  - 指纹识别输出的标签（如 `spring-boot`、`jenkins`）可作为下游扩展词典生成的条件输入，但本模块不生成字典。
- 与“价值评估模块（Value Assessment Agent）”边界：
  - 指纹识别提供高价值组件线索（如 `Jenkins`、`Grafana`、`Admin Console`），价值评估负责优先级排序与资源倾斜建议。

# 三、输入/输出契约（工程化：字段、类型、默认值、示例）

> 说明：一期推荐使用 Pydantic（或 dataclass）定义 Schema。总控/服务层负责把前端输入转成 `TaskCommand`，再由 Orchestrator 生成 `AgentTask`；指纹模块只消费 `AgentTask + AgentContext`，并输出 `ObservationEvent`（见 `docs/heuScan/项目介绍与功能设计/2.第一阶段多Agent功能拆解与协同架构设计.md`）。

## 3.1 FingerprintCommand（模块输入命令）

### 3.1.1 字段定义

| 字段 | 类型 | 必填 | 默认值 | 说明 |
| --- | --- | --- | --- | --- |
| `task_id` | `str` | 是 | - | 任务 ID（全链路追踪） |
| `assets` | `list[AssetRef]` | 是 | - | 待识别资产集合（由基础发现模块或任务输入提供） |
| `sample_policy` | `SamplePolicy` | 否 | 见下 | 证据取样策略（本模块不主动抓取） |
| `match_policy` | `MatchPolicy` | 否 | 见下 | matcher 开关、阈值与冲突策略 |
| `limits` | `FingerprintLimits` | 否 | 见下 | 时间/体量/并发等约束 |
| `evidence_policy` | `EvidencePolicy` | 否 | 见下 | 证据裁剪与脱敏策略 |

### 3.1.2 相关结构

#### AssetRef

| 字段 | 类型 | 必填 | 默认值 | 说明 |
| --- | --- | --- | --- | --- |
| `asset_id` | `str` | 是 | - | 资产稳定 ID（推荐：`asset:{normalized_url}` 或由资产发现模块生成） |
| `url` | `str` | 否 | - | 目标 URL（`https://example.com:8443/`） |
| `host` | `str` | 否 | - | 主机（domain 或 ip） |
| `port` | `int` | 否 | - | 端口 |
| `scheme` | `Literal[\"http\",\"https\"]` | 否 | - | 协议 |
| `tags` | `list[str]` | 否 | `[]` | 上游标签（如 `cdn_suspected`、`login_page`） |

> 约束：`url` 与 (`host`+`port`+`scheme`) 至少提供一种；缺失则输出 warning `ASSET_REF_INVALID`。

#### SamplePolicy（证据取样策略：不抓取）

| 字段 | 类型 | 必填 | 默认值 | 说明 |
| --- | --- | --- | --- | --- |
| `prefer_existing_samples` | `bool` | 否 | `true` | 优先使用上下文/存储中已有样本，减少网络开销 |
| `max_samples_per_asset` | `int` | 否 | `3` | 单资产最多分析的样本数 |
| `min_samples_per_asset` | `int` | 否 | `1` | 单资产最少需要的样本数；不足则输出 `EVIDENCE_INSUFFICIENT` 并提示上游补采样 |
| `allowed_sources` | `list[str]` | 否 | `["asset_discovery","passive_traffic"]` | 允许消费的样本来源（不在列表内的样本忽略或降权） |
| `allow_header_only` | `bool` | 否 | `true` | 允许“仅响应头/无正文”的样本参与匹配，但必须降低置信并输出 warnings |

#### MatchPolicy（匹配策略）

| 字段 | 类型 | 必填 | 默认值 | 说明 |
| --- | --- | --- | --- | --- |
| `enabled_matchers` | `list[str]` | 否 | `["http_header","cookie","html_meta","html_body","js_hint","tls","banner"]` | 启用的 matcher 列表 |
| `min_confidence` | `float` | 否 | `0.6` | 进入最终 findings 的最低置信度 |
| `allow_low_confidence_hints` | `bool` | 否 | `true` | 是否输出低置信“线索标签”（不进入最终结论） |
| `conflict_strategy` | `Literal["keep_all_candidates","pick_highest_confidence"]` | 否 | `keep_all_candidates` | 版本冲突处理 |

#### FingerprintLimits（资源约束）

| 字段 | 类型 | 必填 | 默认值 | 说明 |
| --- | --- | --- | --- | --- |
| `time_budget_s_per_asset` | `int` | 否 | `10` | 单资产预算（超时则输出部分结果 + warning） |
| `max_assets` | `int` | 否 | `200` | 单次运行最多处理资产数量 |
| `max_total_events` | `int` | 否 | `2000` | 单次运行最多产出事件数量（防事件风暴） |

#### EvidencePolicy（证据策略）

| 字段 | 类型 | 必填 | 默认值 | 说明 |
| --- | --- | --- | --- | --- |
| `max_body_bytes` | `int` | 否 | `65536` | 单响应正文最多保留字节数（用于匹配与证据片段） |
| `hash_only_for_binary` | `bool` | 否 | `true` | 二进制响应（如图片/zip）仅留 hash，不留正文 |
| `redact_secrets` | `bool` | 否 | `true` | 证据片段脱敏（token/key/session 等） |
| `store_raw_response` | `bool` | 否 | `false` | 是否持久化原始回包（默认不存，避免敏感与体量膨胀；如需审计可开） |

### 3.2.2 ObservationEvent（事件输出约定）

> 事件通用字段（`task_id/agent_id/event_type/payload_ref/confidence/priority/scope/lineage/dedupe_key/...`）见一期架构文档第 7 章。指纹模块只需严格生成 `event_type + payload + dedupe_key`，并保证 payload 可落库可回放。

建议事件类型（`event_type`）：

1. `fingerprint.finding.created`：产出一个 `Finding`
2. `fingerprint.stack.summary`：针对资产输出一次技术栈摘要（供 TaskSnapshot 快速展示）
3. `fingerprint.hint.created`：产出一个 `RiskHint`（低置信线索）
4. `fingerprint.warning`：产出告警（样本读取失败/证据不足/疑似 CDN/WAF 干扰等）

`dedupe_key` 建议（必须稳定）：

```text
task_id + agent_id + asset_id + event_type + hash(product + category + version_or_candidate_set)
```

### 3.2.3 EvidenceRef（证据引用）

EvidenceRef 用于“可回放、可审计”。建议字段：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `evidence_id` | `str` | 是 | 稳定 ID（hash） |
| `kind` | `str` | 是 | `http_response` / `tls` / `banner` / `html_snippet` / `js_snippet` |
| `source` | `str` | 是 | `passive_traffic` / `asset_discovery` |
| `ref` | `str` | 是 | 原始证据定位（如对象存储 key、流量样本 id、响应记录 id） |
| `location` | `str` | 否 | 证据位置（如 header 名、正则命中片段范围） |
| `snippet` | `str` | 否 | 脱敏后的短片段（用于报告与前端展示） |
| `hash` | `str` | 否 | 原始数据 hash（便于去重/一致性校验） |
| `timestamp` | `str` | 是 | 采集时间（ISO8601） |

# 四、处理流程设计（Stage 化，便于 Orchestrator 编排与回放）

## Stage 0：输入装配与样本选择（prepare）

1. 从 `AgentContext` 读取上游事件（资产发现、被动流量留存、历史采样缓存）。
2. 资产规范化：统一生成 `asset_id`（建议按 `normalized_url` 稳定化）。
3. 样本选择：
   - 若 `prefer_existing_samples=true`，优先选择“最近、成功状态码、正文可用”的样本；
   - 若样本不足（`min_samples_per_asset` 不满足），则对该资产输出 `fingerprint.warning`（`code=EVIDENCE_INSUFFICIENT`，`need_more_evidence=true`），并停止对该资产的进一步匹配；由总控/Orchestrator 决定是否补跑上游采样模块。

## Stage 1：证据缺失处理（request_upstream_evidence）

本模块**不主动抓取**。当样本不足时：

1. 产出 `fingerprint.warning` 事件：
   - `code=EVIDENCE_INSUFFICIENT`
   - `need_more_evidence=true`
   - `asset_id` 与缺失原因（无样本/仅 header/正文为空等）
2. 由 Orchestrator 根据任务画像与预算，选择补跑：
   - 基础发现模块的“HTTP 采样/页面抓取”子能力（若未来在基础发现文档中拆分）
   - 或等待被动流量审查模块提供更多样本
3. 指纹模块对该资产输出空 findings（或仅低置信 hints），不得为了补证据而生成任何网络请求。

## Stage 2：多维 matcher 匹配（match）

按 `enabled_matchers` 顺序执行并输出中间结果（内部结构即可，但必须可解释）：

1. `http_header`：`Server`、`X-Powered-By`、`Via`、`X-Generator`、`X-AspNet-Version` 等
2. `cookie`：`JSESSIONID`、`PHPSESSID`、`laravel_session`、`rememberMe` 等
3. `html_meta`：`<meta name="generator">`、`<meta name="application-name">` 等
4. `html_body`：关键路径、特征字符串、注释、错误页模板
5. `js_hint`：脚本文件名/路径与轻量字符串特征（不做 AST）；仅基于样本中已有的 HTML/正文片段做匹配，不下载 JS、不解析 AST、不提取 API/参数/token（这些归 JS 解析模块）
6. `tls`：证书 Subject/Issuer、SAN、ALPN、JA3（若可得）、常见云/网关证书特征
7. `banner`：端口 banner（仅消费上游已采集的 banner，例如 22/3306/6379 等；本模块不做端口探测/连接探测；注意这类属于“服务指纹”，与 HTTP 分开分类）

## Stage 3：合并、冲突处理与置信度计算（merge）

合并规则必须确定化（建议）：

1. 同一 `product+category` 的证据合并：累积证据数与来源多样性提升 `confidence`
2. 强特征优先：`meta generator`、特定路径与正文命中 > `Server` 头这类弱特征
3. 版本冲突：
   - `keep_all_candidates`：保留候选列表，并生成 `fingerprint.warning` 提醒冲突
   - `pick_highest_confidence`：仅保留最高置信候选，但仍保留冲突 evidence 在 warning 中
4. CDN/WAF 干扰：
   - 若命中 CDN/WAF 指纹（如 `cloudflare`、`akamai`），必须额外输出 `fingerprint.warning`，并降低对后端框架识别的置信（除非正文/JS 强特征命中）

## Stage 4：结构化输出与事件写入（emit）

1. 将最终 `Finding` 转换为：
   - `fingerprint.finding.created`（payload 为 Finding）
   - 可选 `fingerprint.stack.summary`（payload 为资产级摘要）
2. 对低置信线索输出：
   - `fingerprint.hint.created`（payload 为 RiskHint，`requires_review=true`）
3. 统一生成 `dedupe_key`，写入 Event Bus / State Layer（由运行时层完成落库）

## Stage 5：供快照聚合的最小摘要（aggregate）

为 Result Aggregator 提供可直接展示的摘要字段（建议在 Finding 上就能得到）：

- `stack_top`：最高置信组件 Top-N（如 5）
- `admin_surface`：疑似后台/登录页标签（仅线索）
- `waf_cdn`：命中的 WAF/CDN 标签（若有）

# 五、指纹库（Signature DB）与 matcher 设计约束

## 5.1 指纹库组织（建议）

指纹数据建议以“可版本化的静态文件”组织（便于迭代与审计），每条指纹至少包含：

- `product`：产品名（稳定）
- `category`：分类
- `patterns`：匹配规则集合（header/cookie/meta/body/js/tls/banner）
- `version_extract`：版本提取规则（正则/捕获组）
- `confidence_weight`：该规则权重（强/中/弱）
- `notes`：规则解释（用于审计与调试）

> 一期落地建议：优先实现覆盖率高且稳定的规则（server/framework/cms），再补充组件与细粒度特征。

## 5.2 matcher 输出必须可解释

matcher 内部产出必须能说明“为什么识别到它”，最低要求：

- 命中的规则 id（或规则描述）
- 命中的位置（header 名 / cookie 名 / 正则命中片段）
- 命中的证据引用（EvidenceRef）

# 六、与状态机/事件流的协作要点（防循环、防膨胀）

1. 事件去重：严格使用 `dedupe_key`，同一资产同一结论不重复触发下游。
2. 上下文裁剪：对正文/JS 只保留裁剪片段 + hash，避免上下文膨胀。
3. 触发节奏：建议由 Orchestrator 在“发现阶段”后触发一次指纹识别；若后续出现新资产或新证据，再增量触发（以 asset_id 为粒度）。

# 七、错误处理、告警与降级策略（必须写死）

## 7.1 错误码与 warning 建议

| code | 触发条件 | 处理策略 |
| --- | --- | --- |
| `ASSET_REF_INVALID` | 资产引用缺失/格式不合法 | 跳过该资产，记录 warning |
| `SAMPLE_STORE_UNAVAILABLE` | 样本存储不可用/读取失败 | 输出空结果 + warning；由总控决定重试或降级跳过该模块 |
| `SAMPLE_NOT_FOUND` | 资产无可用样本（或被 allowed_sources 过滤后为空） | 输出 `EVIDENCE_INSUFFICIENT` + `need_more_evidence=true`，交由总控补采样 |
| `EVIDENCE_INSUFFICIENT` | 样本正文为空/仅弱特征 | 仅输出低置信 hints 或空 findings |
| `MATCH_CONFLICT` | 同组件出现多个版本候选 | 按 conflict_strategy 输出 candidates + warning |
| `EVENT_BUDGET_EXCEEDED` | 超出 `max_total_events` | 截断输出并 warning，避免事件风暴 |

## 7.2 重试/超时/降级（默认必须固定）

- 本模块不发起网络请求，因此不存在“抓取超时/风控重试”的逻辑。
- 重试策略（仅针对内部依赖，如样本存储/指纹库读取）：最多 `max_attempts=2`；仅对 `timeout/5xx/网络错误`（存储/服务层）重试
- 降级策略：
  1. 优先消费已有样本
  2. 样本不足则返回空 findings + warnings，并提示总控补采样
  3. 不因单资产或单依赖失败导致整任务 `FAILED`

# 八、测试计划与验收标准

## 8.1 单元测试（建议覆盖）

- `asset_id`/URL 规范化与去重
- matcher：header/cookie/meta/body/js/tls/banner 的命中与解释输出
- 版本提取与归一（含冲突候选）
- CDN/WAF 干扰下的置信度下调逻辑
- `dedupe_key` 稳定性（同输入多次运行不产生不同 key）

## 8.2 集成测试（建议覆盖）

- 用固定 HTTP 样本夹具（headers + body + tls）跑通：产出 Findings、EvidenceRef、warnings
- 与基础发现/被动流量留存的上下文装配：能消费上游样本并增量识别
- 事件预算限制：在大量资产下不产生事件风暴，能正确截断并告警

## 8.3 一期验收标准（最小可用）

1. 对常见 Web 技术栈能稳定识别（server/framework/cms 至少覆盖核心场景）。
2. 每条结论都能回溯到 EvidenceRef（可审计）。
3. 幂等：同一任务同一证据重复运行不产生重复 findings/事件。
4. 输出可被 Result Aggregator 汇总展示为 TaskSnapshot 的“技术栈摘要”。
