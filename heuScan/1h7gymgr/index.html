链路提示模块

约 3945 字大约 13 分钟

2026-04-19

链路提示模块 是整个扫描系统的" 关联发现引擎 "，它像一名侦探，在其他10个Agent收集的海量离散数据中，发现隐藏的、非直观的实体间关联关系。它不产生新的原始数据，而是在已有数据之上做二阶推理

二、核心功能

  1. 放大情报价值 ：从N个孤立点，发现O(N²)种可能关联
  2. 指导扫描方向 ：基于关联提示可以扩展扫描范围
  3. 溯源支撑 ：为企业归属、资产归并提供证据链
  4. 降低漏判率 ：单一维度证据不足时，多维度交叉可显著提升可信度

2.1 功能一：网络层关联推断

  • 触发条件 ：获得≥2个IP地址列表时
  • 处理逻辑 ：
  1. 按/24 CIDR前缀对IP进行分组
  2. 对组大小≥2的生成旁站关联提示
  3. 标记证据强度为"weak"（同机房可能是多租户）
  • 输出物 ： same_ip_block 类型提示
  • 质量要求 ：无遗漏、无误报，IP分组算法正确

2.2 功能二：SSL证书关联推断

  • 触发条件 ：获得证书指纹 → 域名映射表时
  • 处理逻辑 ：
  1. 按证书指纹聚合域名
  2. 聚合大小≥2的生成强关联提示
  3. 标记证据强度为"strong"（同一证书是强关联信号）
  4. 自动附加建议"纳入同一目标集合"
  • 输出物 ： same_certificate 类型提示
  • 质量要求 ：准确识别泛域名、多域名证书覆盖范围

2.3 功能三：前端资产关联推断

  • 触发条件 ：获得静态资源哈希 → 站点映射时
  • 处理逻辑 ：
  1. 计算JS/CSS文件内容哈希
  2. 哈希碰撞≥2个站点时生成前端关联
  3. 标记证据强度为"medium"
  4. 自动附加建议"扩大前端扫描范围"
  • 输出物 ： shared_js_hash 类型提示
  • 质量要求 ：区分通用库（jQuery等）与定制化代码

2.4 功能四：DNS链路推断

  • 触发条件 ：获得CNAME解析结果时
  • 处理逻辑 ：
  1. 反向构建：CNAME目标 → 来源域名列表
  2. 来源数量≥2的生成CDN/云服务关联提示
  3. 标记证据强度为"medium"
  • 输出物 ： cname_chain 类型提示
  • 质量要求 ：识别知名CDN域名，区分公有云与自建

2.5 功能五：HTTP跳转链路追踪

  • 触发条件 ：获得3xx重定向链数据时
  • 处理逻辑 ：
  1. 还原完整跳转路径：A→B→C→D
  2. 链长度≥3跳时生成隐蔽链路提示
  3. 高亮展示完整跳转路径供分析
  4. 标记证据强度为"medium"
  • 输出物 ： redirect_chain 类型提示
  • 质量要求 ：识别短链、引流、防爬虫跳转模式

2.6 功能六：第三方服务关联推断

  • 触发条件 ：获得第三方资源引用数据时
  • 处理逻辑 ：
  1. 统计资源URL被引用站点数
  2. ≥3个站点共享同一资源时生成"基础设施共性"提示
  3. 标记证据强度为"weak"
  • 输出物 ： third_party_resource 类型提示
  • 质量要求 ：排除Google Analytics、Font Awesome等公共服务

2.7 功能七：多维度交叉验证

  • 触发条件 ：已有≥2个单维度提示后
  • 处理逻辑 ：
  1. 对所有提示按实体进行反向索引
  2. 发现出现在≥2个维度中的实体簇
  3. 聚合所有维度的证据链
  4. 生成"多因子交叉验证"复合提示
  • 输出物 ： multi_factor_correlation 类型提示
  • 质量要求 ：证据去重、高分优先、不重复不遗漏

2.8 功能八：标准化输出对接

  • 处理逻辑 ：
    1. 为所有提示分配全局唯一ID
    2. 调用报告模块的 add_link_hint 接口
    3. 传入类型、描述、关联实体、证据数、置信度
  • 输出物 ：报告模块中的链路提示章节

三、边界、约束与默认策略

3.1 输入边界

维度有效值范围超出边界时的默认行为
IP列表分析至少2个IPv4地址少于2个IP不生成提示
证书关联同一证书对应≥2个域名单域名证书自动跳过
JS资源关联同一哈希对应≥2个站点单站点哈希不生成关联
CNAME链路同一目标对应≥2个来源单来源不生成聚合提示
重定向链路跳转链长度≥3跳两跳以内重定向不告警
第三方资源同一资源被≥3个站点引用少于3站不认为有共性
证据数量单条提示支持1-10条超过10条只取前10条强证据

3.2 输出边界

  • 单分析批次最大生成 64条 推断性提示
  • 单条提示关联实体数 2-32个
  • 证据分区间 [0.0, 1.0] ，保留3位小数
  • 置信度分级：very_low / low / medium / high
  • is_inferred字段 恒为True （本模块所有结论均为推断）

3.3 默认策略

  • 保守推断原则 ：宁漏勿错，所有结论标记为推断性，建议人工复核
  • 无状态计算 ：所有分析基于当前输入快照，不维护历史状态
  • 证据降级 ：跨源关联证据强度自动降一级处理
  • 去重机制 ：同一实体组合不重复生成同类型提示
  • 超时保护 ：单批次分析最长执行时间 5秒 ，超出截断返回已生成结果

四、职责拆解

4.1 核心职责

  • 多维度关联推断引擎 ：IP网段、SSL证书、JS哈希、DNS解析、重定向链、第三方资源等维度的关联识别
  • 证据链加权评分 ：基于证据类型、强度、数量的置信度量化计算
  • 多维度交叉验证 ：跨维度证据聚合，生成复合关联结论
  • 推断结论标准化输出 ：统一格式、统一置信度分级、统一建议话术
  • 图谱关系传播 ：基于种子实体的间接关联发现

4.2 非职责

  • 不做确定性结论 ：所有输出均为"提示"而非"事实"，不替代人工研判
  • 不维护实体图谱 ：不持久化实体和关系数据，只做单次批处理
  • 不进行原始数据采集 ：不主动扫描、不爬取、不做DNS/HTTP探测
  • 不实现去重逻辑 ：输入数据去重由上游调用方保证
  • 不与外部系统交互 ：纯内存计算，无IO、无网络调用

六、关键算法分析

6.1 加权置信度评分算法

6.1.1 数学公式

最终得分 = 基础分 * 0.6 + 证据平均分 * 0.3 + 数量奖励

其中：
- 基础分 = TYPE_BASE_SCORE[提示类型]  范围[0.2, 0.8]
- 证据平均分 = Σ(证据强度权重 * 自定义权重) / N
- 数量奖励 = min(0.2, N * 0.03)  每增加1条证据+0.03，上限0.2

6.1.2 权重配置

证据强度权重值说明
very_strong1.0确定性事实，如证书哈希精确匹配
strong0.75强统计学关联
medium0.5中等可信度关联
weak0.25弱相关性，需谨慎对待
very_weak0.1仅供参考的巧合性相似

6.1.3 置信度映射阈值

最终得分区间置信度等级
[0.8, 1.0]medium
[0.5, 0.8)low
[0.0, 0.5)very_low

关键设计决策 ： 本模块最高置信度为medium，不输出high等级，强制所有推断经过人工复核。

6.2 多维度交叉验证算法

6.2.1 伪代码

def cross_validation(all_hints):
    # 第一步：构建反向索引
    entity_to_hints = defaultdict(list)
    for hint in all_hints:
        for entity in hint.entities:
            entity_to_hints[entity.value].append(hint)
    
    # 第二步：筛选出现在≥2个提示中的节点
    for entity_value, related_hints in entity_to_hints.items():
        if len(related_hints) >= 2:
            # 该实体存在多维度证据
            all_linked_entities = collect_all_entities(related_hints)
            all_evidences = collect_all_evidence(related_hints)
            
            # 第三步：生成复合提示
            yield create_multifactor_hint(
                central_entity = entity_value,
                linked_entities = all_linked_entities - {entity_value},
                evidence = all_evidences[:10]
            )

6.2.2 算法价值

  • 发现"隐藏的强关联" ：单一维度证据弱但多维度同时出现时，可信度显著提升
  • 例 ：A和B既共享证书，又共享JS哈希 → 关联可信度远高于单一维度
