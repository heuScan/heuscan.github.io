企业情报模块

约 5714 字大约 19 分钟

2026-04-19

企业情报模块（Enterprise Intel Agent）的目标：围绕“目标域名/企业主体”，通过多数据源（ICP、企业信息平台等）获取备案主体、企业画像、关键人员、股权与投资链路、历史变更等信息，并以可审计证据链形式写入图谱与结构化结果，供总控agent聚合为任务快照与最终报告。

  • 范式：PLAN + REFLECT
  • 技术栈：Python

1.1 输入目标与典型场景

  • 输入线索：公司名、备案线索、外部情报响应
  • 输出目标：
    • 备案主体识别（domain → ICP → 主体公司）
    • 企业画像（基础信息/状态/地址/法人/股东/对外投资/主要人员/历史变更等，视数据源能力）
    • 关系拓扑（公司↔公司、公司↔个人、个人↔个人的关系图谱）

1.2 边界、约束与默认策略

  • 仅用于授权场景下的信息收集、分析与图谱构建，不涉及自动化攻击执行设计。
  • 必须受控扩展：默认 max_depth=2，并受 max_nodes/max_edges/time_budget_s 任一约束；触发上限必须停止扩展并产出“部分结果 + 告警”。
  • 人名同名属于高歧义：除非数据源提供稳定标识（如内部 id、证件号脱敏标识等），不得强行跨公司合并为同一自然人；必要时触发复核。
  • 不得把推测写成事实：不确定结论输出为“疑似/可能”，并附证据引用与置信度。

二、职责拆解（对齐总控文档的要素）

2.1 核心职责

  1. 主体解析：域名 → ICP → 备案主体公司（或主体类型判定）
  2. 企业画像采集：公司 → 基础信息 / 法人 / 股东 / 对外投资 / 任职 / 历史变更（视数据源能力）
  3. 关系扩展：围绕“控制/投资/任职”关系递归扩展相关公司与人员（可配置策略）
  4. 实体消歧与去重：公司优先按 credit_code，人物默认不跨公司合并
  5. 图谱写入：Neo4j 写入节点与关系，保证幂等与去重
  6. 证据与审计：保留来源、查询时间、原始回包引用（source_refs），支持回放
  7. 结构化输出：输出 companies/persons/relations + evidence + warnings + stats 供总控聚合

2.2 与其他 Agent/模块的职责边界

结合 docs/heuScan/项目介绍与功能设计/1.项目介绍与解决痛点.md 与 docs/heuScan/项目介绍与功能设计/2.第一阶段多Agent功能拆解与协同架构设计.md 的一期目标（模块化单体 + 编排优先 + 统一数据面），企业情报模块的边界必须明确如下：

  • 与“基础发现/资产发现”能力边界：
    • 本模块不负责子域名枚举、IP/端口扫描、路径探测、CDN 解析等资产发现工作。
    • 本模块产出的是企业主体与关系；域名/IP/端口等资产实体与其关联关系应由资产发现模块产出，并由图谱聚合层统一关联。
  • 与“指纹识别”能力边界：
    • 本模块不识别前后端技术栈、中间件、CMS、组件等指纹信息；该类信息由指纹识别模块负责。
  • 与“动态扫描字典生成（扩展词典）”能力边界：
    • 本模块不直接生成目录/路径/参数字典；它最多提供下游可用的企业别名线索（如规范化公司名、简称、曾用名），由扩展词典模块生成字典并回流到发现/扫描阶段。
  • 与“JS 语义解析”能力边界：
    • 本模块不解析 JS，不提取接口/参数/外联域名；该类能力由 JS 语义解析模块负责。
  • 与“被动流量审查/漏洞线索标记”能力边界：
    • 本模块不分析 HTTP 流量、不输出未授权/CORS/漏洞线索；该类能力由流量审查模块负责。
  • 与“文件情报”能力边界：
    • 本模块不下载文件/抽取文档内容；该类能力由文件情报模块负责。
  • 与“网络拓扑结构生成/图谱聚合”能力边界：
    • 本模块写入/输出的图谱仅包含 Company/Person 及其工商关系（LEGAL_PERSON/SHAREHOLDER/INVEST/EMPLOY）。
    • “网络拓扑结构”（域名→IP→端口→路径/服务）与跨域关系拼装（企业↔域名↔资产）由图谱聚合模块负责统一建模与展示。
  • 与“价值评估/攻击建议/报告输出”能力边界：
    • 本模块不做资产价值排序、不产出攻击链路建议、不生成最终报告；这些由价值评估模块与报告输出模块负责。
  • 与“复核模块”协作边界：
    • 本模块只负责产出 review_required + warnings + evidence_refs，不做人工复核决策闭环；复核流程由复核模块与总控统一推进。

三、与总控Agent的对接契约（模块级输入输出）

总控的 task_contract 是全局约束；企业情报模块必须在契约范围内执行。

3.1 模块输入（建议内部 EnterpriseIntelCommand）

{
  "task_id": "uuid",
  "targets": [
    { "type": "domain", "value": "example.com" }
  ],
  "max_depth": 2,
  "limits": {
    "max_nodes": 1000,
    "max_edges": 3000,
    "time_budget_s": 180
  },
  "source_policy": {
    "preferred_sources": ["icp", "aiqicha", "qichacha", "zhanzhang"],
    "allow_paid_sources": true,
    "allow_fallback": true
  },
  "evidence_policy": {
    "store_raw_payload": true,
    "store_source_refs": true
  }
}

3.2 模块输出（建议内部 EnterpriseIntelResult）

{
  "task_id": "uuid",
  "status": "succeeded",
  "companies": [],
  "persons": [],
  "relations": [],
  "graph_delta": { "nodes": [], "edges": [] },
  "evidence": [
    {
      "source": "icp",
      "query": "example.com",
      "fetched_at": "2026-04-08T12:00:00Z",
      "source_ref": "minio://.../raw/icp/xxx.json"
    }
  ],
  "warnings": [],
  "review_required": false,
  "stats": { "depth_reached": 2, "nodes": 120, "edges": 260 }
}

3.3 字段解释与校验规则

3.3.1 EnterpriseIntelCommand 字段说明

字段类型必填默认值说明校验/约束
task_idstring是-任务唯一标识，贯穿全链路（状态、证据、图谱、结构化结果）非空；建议 UUID
targetsarray是-输入线索集合（域名/企业/统一社会信用代码）至少 1 项；同类去重
targets[].typeenum是-线索类型domain / company_name / credit_code
targets[].valuestring是-线索值domain：去协议去路径；credit_code：18 位校验；company_name：trim
max_depthint否2递归扩展深度取值建议 0~3；超过上限按上限截断并 warning
limits.max_nodesint否1000图谱节点上限（单 task）>=1；触发即停止扩展并 warning
limits.max_edgesint否3000图谱边上限（单 task）>=1；触发即停止扩展并 warning
limits.time_budget_sint否180模块级时间预算（秒）>=1；触发即停止扩展并 warning
source_policy.preferred_sourcesarray否["icp"]数据源优先级顺序仅允许白名单；未知 source 直接判定输入非法并拒绝执行（由总控/应用层处理）
source_policy.allow_paid_sourcesbool否true是否允许使用付费源（若已配置密钥）若 false，遇到仅付费可得字段应降级并 warning
source_policy.allow_fallbackbool否true上游失败时是否允许其他源兜底false 时，上游失败应直接进入 review 或 fail（按场景）
evidence_policy.store_raw_payloadbool否true是否保存原始回包false 时也必须保留 source_ref 的最小元数据
evidence_policy.store_source_refsbool否true是否保存证据引用（回放索引）建议一期恒为 true

3.3.2 EnterpriseIntelResult 字段说明

字段类型说明
statusenumsucceeded / failed / partial（达到限制或降级导致不完整）
companies/persons/relationsarray结构化实体与关系（供聚合层/展示层消费）
graph_deltaobject本次任务写入/新增的图谱增量（用于增量聚合/快照构建；也可返回全量）
evidencearray证据索引（见第六章）；任一关键结论必须能回溯到 evidence
warningsarray可解释的风险/降级/冲突信息（见 3.5）
review_requiredbool是否必须进入复核模块（高风险/冲突/歧义/越界）
statsobjectdepth_reached/nodes/edges 等统计，便于聚合层与展示层消费

3.3.3 status/stage 枚举（建议统一）

  • status：pending / running / succeeded / failed / partial
  • stage：resolve / fetch / expand / write / aggregate

3.3.4 结构化实体与关系模型（companies/persons/relations）

目标：聚合层/展示层可直接消费、可排序、可复核；任意关键关系必须能回溯到 evidence_refs。

companies[]（建议最小字段集）：

字段类型必填说明
idstring是内部唯一 id（建议：company:{credit_code} 或 company:hash(name)）
namestring是公司名称（标准化后的展示名）
credit_codestring | null否统一社会信用代码（若可得）
statusstring | null否企业状态（存续/注销/吊销等，来源决定）
source_tagsstring[]否命中的数据源列表（如 ["icp","qichacha"]）
confidencenumber | null否主体置信度（0~1；可选）

persons[]（建议最小字段集）：

字段类型必填说明
idstring是内部唯一 id（优先使用来源稳定 id，否则 task 内临时 id）
namestring是姓名
source_tagsstring[]否命中的数据源列表
ambiguityboolean | null否是否存在同名歧义风险

relations[]（建议最小字段集）：

字段类型必填说明
typeenum是LEGAL_PERSON / SHAREHOLDER / INVEST / EMPLOY
source_idstring是起点实体 id（对应 company/person 的 id）
target_idstring是终点实体 id
attrsobject否关系属性：ratio、role_title、confidence 等
evidence_refsstring[]否与该关系直接相关的证据引用（source_ref 或 evidence_id）

3.3.5 图谱数据结构（graph_delta）

nodes[]（GraphNode）建议字段：

字段类型必填说明
idstring是与 companies[].id / persons[].id 对齐
typeenum是company / person
namestring是展示名

edges[]（GraphEdge）建议字段：

字段类型必填说明
sourcestring是起点节点 id
targetstring是终点节点 id
typeenum是与 relations[].type 对齐（如 LEGAL_PERSON）
attrsobject否ratio/role_title/confidence/source_ref 等

3.4 复核触发（对齐总控的“复核模块”路由）

满足任一条件应输出 review_required=true 并给出 warnings（供总控路由到复核模块）：

  • 同一公司出现冲突 credit_code 或主体关键信息（如名称/状态）在不同来源明显不一致
  • 人物仅凭姓名产生跨公司合并（应禁止或降级为“疑似同名”并触发复核）
  • 股权比例/控制关系来自不同来源且冲突明显
  • 扩展过程中出现异常膨胀（接近/达到规模上限）

3.5 warnings 工程化格式（建议）

建议 warnings 统一结构，便于总控聚合、展示层呈现与复核路由：

{
  "code": "SOURCE_RATE_LIMIT",
  "severity": "warn",
  "message": "qichacha 触发限流，已降级使用 aiqicha 补齐",
  "source": "qichacha",
  "retryable": true,
  "hint": "降低并发或延长重试间隔",
  "evidence_refs": ["minio://.../raw/qichacha/error-xxx.json"]
}

建议 code 白名单（一期至少覆盖）：

  • SUBJECT_UNRESOLVED：主体无法确认
  • SOURCE_RATE_LIMIT：上游限流/风控
  • SOURCE_TIMEOUT：上游超时/网络异常
  • DATA_CONFLICT：多源关键字段冲突
  • IDENTITY_AMBIGUOUS：同名歧义无法消解
  • LIMIT_REACHED：触发深度/规模/时间预算上限

3.6 模块标准输出模板(待议)

当企业情报模块作为 LLM Agent 运行时，建议约束其只输出以下结构化 JSON（便于总控/状态机直接消费），并确保字段含义与本章一致：

{
  "task_id": "",
  "status": "succeeded",
  "stage": "aggregate",
  "companies": [],
  "persons": [],
  "relations": [],
  "graph_delta": { "nodes": [], "edges": [] },
  "evidence": [],
  "warnings": [],
  "review_required": false,
  "stats": {
    "depth_reached": 0,
    "nodes": 0,
    "edges": 0,
    "time_used_s": 0
  }
}

四、内部接口设计（供总控/Orchestrator 调用）

企业情报模块作为后端内部 Specialist Agent 存在：不对外暴露 HTTP API，只提供进程内可调用的能力接口与可审计的事件输出；对外唯一 API 入口由 Flask API Layer / Application Layer 统一承载。

4.1 能力标识与注册

  • capability_tag = "enterprise_intel"（建议固定，用于 AgentRegistry.get_by_capability(...)）
  • Orchestrator 基于任务计划选择本能力执行，不允许前端或 API 层直接点名调用本模块。

4.2 调用入口（函数级契约）

对齐总控文档建议的内部接口：企业情报模块实现 BaseAgent，由运行时执行器调用 run(...)。

class EnterpriseIntelAgent(BaseAgent):
    capability_tag = "enterprise_intel"

    def run(self, task: AgentTask, context: AgentContext) -> list[ObservationEvent]:
        """
        - task.payload: EnterpriseIntelCommand（见 3.1 / 3.3）
        - 返回：ObservationEvent 列表（见 4.3），供 Event Bus / State Store 与 Result Aggregator 消费
        """
        ...

约定：

  • AgentTask.payload 必须承载本模块输入契约 EnterpriseIntelCommand，字段含义以第 3 章为准（不再引入面向 HTTP 的 DTO）。
  • 调用方（通常是 Orchestrator / 执行器）负责：
    • 任务生命周期（创建/排队/超时/取消/重试）的系统级管理
    • 把本模块输出事件写入 Event Bus / State Store
    • 调用聚合器构建任务快照/图谱快照/报告材料

4.3 事件输出约定（最小事件集）

企业情报模块输出以 ObservationEvent.event_type 区分；大 payload 建议用 payload_ref 指向存储（例如 MinIO/PostgreSQL），事件内只携带摘要字段与引用。

建议最小事件类型（一期固定）：

  • enterprise_intel.progress
    • 用途：驱动状态机推进与阶段统计
    • payload（建议）：{ "status": "running|partial|failed|succeeded", "stage": "resolve|fetch|expand|write|aggregate", "progress": 0-100?, "stats": {...}? }
  • enterprise_intel.warning
    • 用途：对上游限流、数据冲突、降级、越界等给出可解释原因
    • payload：单条 warning（沿用 3.5 的结构，retryable/hint/evidence_refs 等语义不变）
  • enterprise_intel.result
    • 用途：产出最终结构化结果，供聚合器入库/生成快照
    • payload：EnterpriseIntelResult（见 3.3.2；可通过 payload_ref 存放全量，事件内保留必要索引）
  • enterprise_intel.review_required
    • 用途：显式触发复核路由（或由 result/warnings 中的 review_required=true 间接触发）
    • payload（建议）：{ "reason_codes": [...], "evidence_refs": [...] }

4.4 错误、降级、重试与幂等（内部语义）

  • 输入校验失败：直接拒绝执行本模块能力（由调用方在应用层记录失败原因）；必要时可产出 enterprise_intel.warning 说明原因，并让任务进入 failed 或 review_required（按总控策略）。
  • 上游失败/限流/验证码/风控：不得中断系统级任务；本模块应优先“降级补齐”，并通过 warnings[] 明确：
    • code/severity/source/retryable/hint/evidence_refs
    • 若最终不完整：以 status=partial 输出结果并附 warnings
  • 不可恢复失败：以 enterprise_intel.progress（status=failed）+ enterprise_intel.warning（说明原因）结束，并保证已有 evidence/日志可回放。
  • 幂等：
    • 事件层：为关键事件设置稳定的 ObservationEvent.dedupe_key（例如 task_id + stage + source + query + normalized_company_id），便于 Event Bus 去重。
    • 数据层：图谱写入必须幂等（节点/边按 id 与关键属性去重）；evidence 建议按 digest 或 source_ref 去重，避免同一回包重复落库。

五、数据源适配层（一期：可插拔多数据源）

5.1 Adapter 抽象（建议）

将爱企查/企查查/ICP/站长之家等实现为多个 Adapter，统一为同一接口：

  • resolve_subject(target)：域名/公司名 → 主体（公司名/credit_code）
  • fetch_company_profile(company)：公司画像（基础信息、状态、地址等）
  • fetch_company_relations(company)：法人/股东/对外投资/任职等
  • fetch_change_history(company)：历史变更（若可用）

Adapter 统一返回结构建议包含：source、fetched_at、raw_ref（可回放引用）、confidence（可选）。

5.2 优先级与降级策略（一期默认）

  1. ICP（域名备案）优先用于“主体确认”
  2. 企业画像来源按 preferred_sources 顺序尝试
  3. 任一来源不可用：不中断任务，记录 warning，并继续用其他来源补齐
  4. 若主体无法确认：停止扩展并 review_required=true

5.3 配额/限流/风控处理（建议）

  • 每个 source 单独的并发与 QPS（如 qps=1~3）配置
  • 触发上游限流/验证码/风控：快速降级到其他源，并记录失败原因（warnings）与失败证据引用（若可保存）

5.4 配置项与密钥管理（工程落地必须）

建议以环境变量或配置文件集中管理各数据源的密钥与限流参数（不得硬编码在代码与文档示例里）：

  • INTEL_SOURCE_ICP_ENABLED=true/false
  • INTEL_SOURCE_AIQICHA_API_KEY=...
  • INTEL_SOURCE_QICHACHA_API_KEY=...
  • INTEL_SOURCE_ZHANZHANG_API_KEY=...
  • INTEL_SOURCE_<SOURCE>_QPS=1
  • INTEL_SOURCE_<SOURCE>_CONCURRENCY=2
  • INTEL_EVIDENCE_STORE=minio（或 postgres）
  • INTEL_EVIDENCE_RETENTION_DAYS=90

六、数据面：图谱 + 证据 + 审计回放

6.1 证据（Evidence）落库建议

为支持“可回放、可审计”，建议每次拉取外部数据源时生成 evidence：

  • source：来源（icp/aiqicha/qichacha/...）
  • query：查询参数（域名/公司名/credit_code）
  • fetched_at：拉取时间
  • source_ref：原始回包的引用（如 MinIO 路径或数据库主键）
  • digest：原始回包摘要（可选，便于去重）

证据不一定要全部写入 Neo4j；更建议写入独立存储（如 PostgreSQL/MinIO），Neo4j 节点/边仅保留 source_ref 关联。

建议 evidence 最小字段集（AI 生成代码时可直接作为数据结构）：

{
  "evidence_id": "ev-uuid",
  "task_id": "uuid",
  "source": "icp",
  "query": "example.com",
  "fetched_at": "2026-04-08T12:00:00Z",
  "source_ref": "minio://.../raw/icp/xxx.json",
  "digest": "sha256:...",
  "http_status": 200,
  "cost_ms": 1200
}

MinIO 路径规范建议（便于检索与回放）：

  • raw/{task_id}/{source}/{yyyy-mm-dd}/{evidence_id}.json

6.2 实体去重与消歧（必须说明）

  • Company 去重：
    • 优先按 credit_code；将 company.id 设为 company:{credit_code}，确保全局唯一
    • 无 credit_code 时：将 company.id 设为 company:hash(normalized_name)（低置信），并在 warnings 标注主体不稳定
  • Person 去重：
    • 默认不跨公司合并；将 person.id 设为 person:{source}:{source_person_id}（若来源提供）
    • 若无稳定 id：使用 task 内部临时 id（不做全局合并），并 ambiguity=true

7.1 图模型设计

7.1.1 节点设计（Node）

系统中定义两类核心节点：

（1）Company（企业节点）

字段类型说明

namestring公司名称

（2）Person（个人节点）

字段类型说明

7.2 查询模型设计

1 查询企业关联人物

MATCH (p:Person)-[:LEGAL_PERSON|SHAREHOLDER|EMPLOY]->(c:Company {name: "目标公司"})
RETURN p

2 查询企业对外投资

MATCH (c1:Company {name: "目标公司"})-[:INVEST]->(c2:Company)
RETURN c2

3 查询关系拓扑（核心）

MATCH (n)-[r]-(m)
WHERE n.name = "目标公司"
RETURN n, r, m
LIMIT 100

4 多跳关系查询（Agent 核心能力）

MATCH path = (c:Company {name: "目标公司"})-[*1..2]-(n)
RETURN path

7.3 数据写入设计（幂等入库策略）

1 Company 节点去重（建议统一以 id 作为主键）

MERGE (c:Company {id: $id})
SET c.name = $name, c.credit_code = $credit_code

无 credit_code 的兜底（仍使用 id，但 credit_code 为空且 confidence 更低）：

MERGE (c:Company {id: $id})
SET c.name = $name

2 关系创建（附带来源与证据引用）

MATCH (p:Person {id: $person_id})
MATCH (c:Company {id: $company_id})
MERGE (p)-[r:LEGAL_PERSON]->(c)
SET r.source = $source, r.source_ref = $source_ref, r.fetched_at = $fetched_at

3 更新属性（示例）

SET c.status = $status

7.4 图规模控制（必须与停止条件一致）

  • 最大扩展深度：默认 2（可配置 2~3）
  • 最大节点数限制：默认 1000（可配置）
  • 最大边数限制：默认 3000（可配置）
  • 过滤低价值节点：无新增关系的叶子节点不继续扩展
  • 循环去重：同一公司在同一 task 内只扩展一次

八、核心处理流程（Agent 逻辑，阶段化）

Stage 1：主体解析（resolve）

域名 → ICP → 备案主体公司：

  1. 调用 ICP Adapter 获取备案信息与主体线索
  2. 若无法确认主体：输出 warning，review_required=true，结束或降级为“仅备案信息结果”

Stage 2：企业画像采集（fetch）

  1. 按 preferred_sources 顺序拉取企业画像与关系数据
  2. 每次拉取都生成 evidence（source_ref/raw_ref）
  3. 对关键字段（credit_code、法人、主要股东、对外投资）做最小一致性校验

Stage 3：递归扩展（expand，重点）

扩展入口建议（可配置）：

  • 对外投资企业（INVEST）
  • 关键股东/控制方关联公司（SHAREHOLDER）
  • 关键人员任职关联公司（EMPLOY，谨慎）

停止条件：depth==0 或任一 limits 触发或命中异常膨胀/循环。

伪代码（示意）：

def expand(company_id, depth):
    if depth == 0 or reached_limits():
        return
    if visited(company_id):
        return
    mark_visited(company_id)

    rels = fetch_relations(company_id)
    write_graph(company_id, rels)

    for next_company in pick_next_companies(rels):
        expand(next_company, depth - 1)

Stage 4：图谱写入与一致性校验

  1. 写入 Neo4j：节点/边/属性/证据引用
  2. 冲突检测：明显冲突输出 warnings，并 review_required=true

Stage 5：结构化输出与图谱输出（aggregate）

  1. 输出结构化结果：companies/persons/relations/evidence/warnings/stats
  2. 输出图谱数据：nodes/edges 或 graph_delta
  3. 输出复核标记：review_required

九、测试计划与验收标准

9.1 单元测试（建议覆盖）

  • Adapter 接口契约与字段映射（含缺失字段容错）
  • Company 去重/Person 不合并策略
  • 扩展停止条件（depth/规模/time_budget）
  • 错误码/错误场景映射为 warnings 的规则

9.2 集成测试（建议覆盖）

  • 模拟链路：ICP → 公司 → 法人/股东 → 对外投资 → 扩展 2 层
  • 校验：图谱节点/边数量受控、重复边去重、结构化输出包含 evidence/source_refs
  • 降级：某数据源不可用时仍能输出可用快照（warnings 可见）

9.3 回归测试（建议覆盖）

  • 同一 task_contract 多次执行幂等（不重复膨胀图谱）
  • 同名人物场景不产生错误合并，并能触发复核标记

十、参考实现骨架（用于 AI 直接生成 Agent 代码）

一期建议落地语言与框架：Python + Flask；数据结构使用 Pydantic（或 dataclass），与 docs/heuScan/项目介绍与功能设计/2.第一阶段多Agent功能拆解与协同架构设计.md 的后端形态保持一致。

10.1 模块入口（建议）

  • 模块入口函数：run_enterprise_intel(command: EnterpriseIntelCommand, ctx: RuntimeContext) -> EnterpriseIntelResult
  • 入口必须只依赖 command + ctx，不得直接读取 Flask request 对象（由总控/服务层装配输入）

10.2 Adapter 接口（建议直接照此生成代码）

class IntelSourceAdapter(Protocol):
    source_name: str

    def resolve_subject(self, target: str, *, timeout_s: int) -> SubjectResolveResult: ...
    def fetch_company_profile(self, company_key: CompanyKey, *, timeout_s: int) -> CompanyProfileResult: ...
    def fetch_company_relations(self, company_key: CompanyKey, *, timeout_s: int) -> CompanyRelationsResult: ...
    def fetch_change_history(self, company_key: CompanyKey, *, timeout_s: int) -> ChangeHistoryResult: ...

10.3 重试/超时/降级（决策必须写死，避免 AI 自行发挥）

  • 单次调用默认超时：timeout_s=10（可按 source 单独配置）
  • 重试策略：最多 max_attempts=2；仅对 timeout/上游服务端错误/网络错误 重试；对 限流/风控 不重试，直接降级
  • 降级策略：按 preferred_sources 顺序切换；所有候选均失败则输出 SUBJECT_UNRESOLVED 或 SOURCE_TIMEOUT 等 warning，并在必要时置 review_required=true

10.4 规范化与 ID 生成（必须一致，否则图谱与结构化结果会漂移）

  • domain 规范化：去协议、去路径、去端口、转小写（如 https://Example.com/a → example.com）
  • company_name 规范化：trim、全角半角归一（若有）、连续空白压缩
  • credit_code 校验：长度 18；若不合法不得作为强主键使用（只能作为弱字段，并 warning）
  • company.id：
    • 有 credit_code：company:{credit_code}
    • 无 credit_code：company:hash({normalized_name})
  • person.id：
    • 有来源稳定 id：person:{source}:{source_person_id}
    • 无稳定 id：person:temp:{task_id}:{seq}

十一、Neo4j 约束与索引（一期建议）

目的：保证幂等写入、提升查询性能，避免同一实体重复膨胀。

CREATE CONSTRAINT company_id_unique IF NOT EXISTS
FOR (c:Company) REQUIRE c.id IS UNIQUE;

CREATE CONSTRAINT person_id_unique IF NOT EXISTS
FOR (p:Person) REQUIRE p.id IS UNIQUE;
