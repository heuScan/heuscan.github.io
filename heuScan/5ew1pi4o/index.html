资格赛项目说明文档

约 14709 字大约 49 分钟

2026-04-19

1、项目介绍

针对传统渗透测试信息收集存在的工具碎片化、决策静态化、数据孤岛化、报告人工化等痛点，本项目设计并实现了一款基于智能 Agent 的主动 - 被动融合式信息收集系统。 该系统突破了传统工具的局限性：通过 Agent 集群实现分布式协同扫描与动态任务调度，根据目标环境实时调整扫描策略与探测粒度；利用机器学习算法生成场景化专属字典，显著提升弱口令与目录爆破的成功率；结合主动指纹识别与被动流量分析技术，精准发现常规工具难以探测的影子资产、内网服务与未授权访问漏洞。 系统内置多维度数据关联引擎，能够自动梳理资产间的依赖关系与攻击面，生成可视化的网络拓扑与攻击路径图，并输出结构化的综合情报报告。本项目将极大解放渗透测试人员的重复性劳动，实现信息收集环节的智能化、自动化与标准化，为构建完整的渗透测试流程提供核心基础能力。

2、解决痛点

2.1 传统信息收集的痛点

渗透测试前期的信息收集并不是单一维度的资产枚举，而是覆盖企业主体、股权结构、法人信息、备案信息、子域名、IP、端口、敏感路径、前后端技术栈、网页内容、JS 代码、可下载文件、HTTP 流量等多类信息的综合性工作。目标网站体量越大，涉及的页面、接口、脚本、文件和关联资产越多，信息收集的范围和工作量也会迅速膨胀。

在传统工作方式中，上述信息通常需要依赖多个相互独立的工具和平台完成。例如，企业信息需要分别查询工商平台、备案平台和站长类平台；子域名、端口、路径和指纹识别需要分别使用 oneforall、端口扫描器、dirsearch、指纹识别工具等；网页代码、JS 文件、可下载文件和 HTTP 流量又需要人工审查或借助其他分析工具。工具之间缺少统一任务入口、统一数据格式和统一结果视图，导致信息分散、上下文断裂，渗透测试人员需要在多个工具结果之间反复切换、复制、整理和比对。

传统扫描器普遍依赖固定字典或通用规则，扫描策略缺乏对目标实际环境的理解。以敏感目录扫描为例，固定字典会在大量无关路径上消耗时间，却无法根据目标的技术栈、业务命名、企业别名、JS 中出现的接口路径等线索生成更有针对性的探测路径。当目标表现出 Java、Spring Boot、Django、Vue、WordPress 等特征时，传统工具往往不能自动调整扫描重点，容易出现无效扫描多、有效命中少、关键路径遗漏的问题。

资产发现结果之间也普遍缺少关联性。子域名、IP、端口、服务、路径、证书、备案主体和公司关系通常按照发现顺序或工具输出顺序排列，缺少从“公司主体 -> 域名 -> IP -> 端口 -> 服务 -> 路径 -> 证据”的统一建模。后续整理时，测试人员需要手动判断资产归属、解析关系、服务依赖和业务边界，容易出现重复记录、关系缺失、归属不清和证据无法追溯等问题。

企业主体、法人、股东、对外投资、历史变更等工商情报本身也存在来源分散、字段不统一、信息噪声多和人工整理成本高的问题。传统方式往往只能得到零散的企业信息列表，难以自动形成公司与公司、公司与个人、个人与个人之间的关系图谱，也难以进一步与网站资产、备案主体和网络拓扑建立关联。

在网页和前端代码审查环节，传统信息收集高度依赖人工阅读页面、查看源码、审查 JS 文件和记录可疑接口。大型网站通常包含大量压缩、混淆或分包加载的 JS 文件，其中可能隐藏 API 路径、外联 URL、测试接口、调试信息、内部命名、固定密钥、Token 片段、接口参数和环境标识。人工审查不仅工作量大，而且容易遗漏隐藏在代码分支、注释、配置对象或拼接逻辑中的高价值线索。

网页可下载文件和敏感文件泄露同样容易被忽视。诸如 .git、.env、配置文件、日志文件、备份文件、数据库导出、接口文档、SourceMap、压缩包、CI/CD 配置和证书密钥文件等，可能包含源码结构、数据库连接信息、接口说明、部署信息、账号凭据和内部路径。传统流程中，这类文件往往需要人工发现、下载、识别和解析，缺乏自动化的类型判断、敏感信息抽取和结构化输出能力。

此外，部分风险线索并不直接体现在静态资产列表中，而是隐藏在 HTTP 请求与响应过程里。例如未授权访问、CORS 配置异常、错误栈泄露、DEBUG 信息、内网 IP 暴露、敏感参数、弱凭证传输、SQL 报错和异常回显等，往往需要在流量层面进行持续审查。传统信息收集通常把主动扫描和流量审查割裂开来，导致这些隐藏于交互过程中的线索难以在信息收集阶段被及时发现和标记。

传统结果整理方式也缺少价值排序能力。大量资产和线索通常按发现时间或工具输出顺序排列，高价值目标可能淹没在普通资产、静态资源、低价值子域和重复路径之中。测试人员需要依靠经验判断哪些资产更接近核心业务、哪些接口更值得优先关注、哪些后台或控制面具有更高价值，这不仅耗时，也容易受到个人经验差异影响。

最后，传统信息收集缺少攻击链路层面的辅助分析。企业关系、资产归属、域名解析、端口服务、JS 接口、文件泄露、流量风险和价值评分之间往往存在潜在利用链路，但传统工具只输出单点结果，无法自动提示资产之间可能存在的依赖关系、跳转关系、认证关系或证据链。渗透测试人员需要在收集完成后重新梳理所有信息，手动推断可能的后续测试路径，既消耗时间，也可能遗漏非直观但价值较高的关联路径。

2.2 本项目针对传统痛点的解决思路

针对传统信息收集工具碎片化、结果孤岛化和人工整理成本高的问题，本项目采用多 Agent 协同的信息收集架构，将企业情报、基础发现、指纹识别、JS 解析、动态字典、文件情报、被动流量审查、图谱聚合、价值评估、链路提示和报告输出等能力纳入统一任务流程。系统由总控 Agent 负责目标标准化、任务拆解、能力路由和状态推进，各能力 Agent 只负责自身边界内的专业分析，从而形成可编排、可扩展、可追踪的信息收集闭环。

在基础资产发现层面，系统统一收集子域名、DNS 解析、IP、端口、HTTP 服务、路径和最小响应样本，并将发现结果沉淀为标准化事件和证据引用。不同于传统工具各自输出独立结果，本项目通过统一数据面保存任务、事件、资产、证据和状态，使后续指纹识别、JS 解析、图谱构建、价值评估和报告输出都能基于同一份可追溯事实进行协作。

针对固定字典效率低和命中率不足的问题，系统引入动态扫描字典生成能力。动态字典 Agent 可以综合企业别名、备案主体、技术栈指纹、历史发现路径、JS 中提取的接口和业务命名特征，生成更贴合目标环境的增量字典。例如识别到 Spring Boot 后，优先关注 actuator、swagger、druid 等相关路径；识别到企业简称或业务系统命名后，生成更符合目标命名习惯的目录和接口候选，从而减少无效探测，提高有效线索覆盖率。

针对网页和 JS 中隐藏线索难以人工完整审查的问题，系统设计 JS 语义解析 Agent，对前端脚本中的组件结构、接口路径、外联 URL、认证逻辑、加密/签名行为、固定材料、调试残留、注释和环境标识进行结构化提取。该模块不只是简单匹配 URL，而是从脚本语义中补全页面、接口、路由、状态字段和业务功能之间的关系，为后续图谱聚合、价值评估和链路提示提供应用层线索。

针对可下载文件和敏感文件难以系统化分析的问题，系统引入文件情报 Agent，对扫描过程中发现的 .git、.env、SourceMap、配置文件、日志文件、备份文件、数据库文件、接口文档、密钥证书、压缩包和部署配置等进行只读解析。系统会根据文件类型提取账号、Token、密钥、接口、路径、数据库配置、部署信息和源码结构等线索，并对敏感内容进行脱敏和证据化记录，避免人工逐个文件打开和整理。

针对隐藏在 HTTP 交互过程中的风险线索，本项目设计被动流量审查 Agent，对采集到的请求与响应样本进行被动分析，识别未授权访问、CORS 异常、错误栈泄露、DEBUG 信息、敏感参数、异常回显和弱凭证传输等风险迹象。该模块与主动发现结果形成闭环：当流量审查发现高风险线索时，可以将相关资产推送给价值评估和复核流程，使系统在信息收集阶段就能标记值得优先关注的目标。

针对资产、企业和证据之间缺少关联的问题，本项目通过图谱聚合 Agent 构建统一事实底座，将公司主体、法人、股东、备案信息、域名、IP、端口、服务、路径、文件、JS 线索、流量线索和证据引用统一建模。系统能够形成企业关系图谱与网络资产拓扑，展示“企业主体 - 网站域名 - IP - 端口 - 服务 - 路径 - 证据”之间的连接关系，解决传统工具输出难以关联、难以追溯的问题。

针对结果缺乏价值排序的问题，系统引入价值评估 Agent，对资产重要性、技术栈特征、风险线索、业务关联性和证据完整度进行综合评估。系统不再只按发现顺序展示结果，而是对主站、统一认证、管理后台、核心 API、关键数据面、CI/CD 控制面、数据库管理面等高价值目标进行标记和排序，帮助测试人员优先关注更可能影响核心业务的资产。

针对传统流程缺少攻击链路辅助分析的问题，系统设计链路提示 Agent，在已有图谱和证据基础上进行保守的关联推断。该模块可以结合 IP 网段、SSL 证书、DNS 链路、HTTP 跳转、JS 资源哈希、第三方资源引用、接口调用关系和多维度交叉证据，提示可能存在的资产关联、业务依赖和后续测试方向。链路提示只作为辅助研判结果输出，并附带证据来源、置信度和复核标记，不替代人工判断，也不执行自动化攻击。

最终，报告输出 Agent 会将各模块产生的企业情报、资产发现、技术指纹、JS 线索、文件情报、流量风险、图谱关系、价值排序和链路提示进行统一聚合，生成结构化、可审计、可阅读的信息收集报告。通过统一入口、统一数据面、统一证据链和统一输出，本项目能够显著降低传统信息收集中的重复劳动和整理成本，提高信息收集的完整性、准确性、关联性和后续可用性。

三、系统架构

3.1 总体架构设计

本项目采用“一个 Vue3 前端 + 一个 Flask API 后端 + 后端内部多 Agent Runtime”的总体架构。系统不将各 Agent 拆分为多个独立后端服务，而是在统一 Flask 后端中完成任务创建、状态管理、Agent 编排、工具适配、数据沉淀和结果聚合，保证开发、部署、调试和后续扩展的复杂度可控。

前端作为统一展示与操作入口，主要负责任务提交、目标配置、执行进度查看、资产发现结果展示、企业关系图谱与网络拓扑展示、报告查看和结果导出。前端不承担 Agent 调度、状态推理和结果拼装逻辑，所有数据均通过 Flask API 获取。

后端作为系统唯一对外服务入口，负责把用户输入转换为标准化任务，并由 Orchestrator 统一调度各能力 Agent。各 Agent 按能力域划分，分别承担企业情报、基础资产发现、指纹识别、JS 解析、动态字典生成、文件情报、被动流量审查、图谱聚合、价值评估、链路提示和报告输出等工作。所有 Agent 的执行结果统一进入事件流和数据存储层，再由结果聚合器生成前端可查询的任务快照、图谱视图和报告产物。

3.2 后端分层架构

Flask 后端内部采用分层设计，避免 API、Agent、工具调用和数据写入相互耦合。整体划分为 API Layer、Application Layer、Agent Runtime Layer、Event Bus / State Layer、Tool Adapter Layer 和 Data Layer 六个层次。

API Layer 负责接收前端请求，包括任务创建、任务状态查询、发现结果查询、图谱查询、报告查询和系统状态查询等。API 层只面向前端提供稳定接口，不直接调用某个具体 Specialist Agent，也不暴露后端内部 Agent 的执行细节。

Application Layer 负责把 HTTP 请求转换为内部任务命令，管理任务生命周期，并聚合查询结果供前端使用。该层主要承担任务服务、执行服务、结果服务、图谱查询服务和报告服务等应用级职责，不直接实现扫描、解析或研判逻辑。

Agent Runtime Layer 是系统的核心运行层，负责 Agent 注册、任务调度、上下文装配、Agent 执行、超时重试和执行记录。Orchestrator、Specialist Agents、Review Agent、Agent Registry、Agent Executor 和 Result Aggregator 均位于该层。所有 Agent 都由 Runtime 管理，不允许在服务层中自由实例化和互相调用。

Event Bus / State Layer 负责接收 Agent 输出的 ObservationEvent，执行去重、状态推进、失败隔离、重试记录和任务快照更新。系统的任务状态不是由前端推断，而是由 Orchestrator 和事件状态层共同维护，确保任务过程可回放、可审计、可恢复。

Tool Adapter Layer 用于屏蔽底层工具和外部数据源的差异。企业信息查询、子域名发现、端口探测、HTTP 采样、指纹识别、JS 解析、文件抽取和图谱写入等能力都通过适配器接入。Agent 只能调用适配器，不能直接拼接命令行参数或绕过统一工具接口。

Data Layer 负责持久化任务、事件、资产、证据、图谱、评分和报告。该层围绕统一领域模型设计，不允许每个 Agent 自行定义一套孤立数据结构，从而保证图谱、价值评估、报告输出和审计回放都能基于同一份事实底座工作。

3.3 多 Agent 协同架构

系统中的多 Agent 并不是多个独立系统，而是同一个后端 Runtime 内部的能力域拆分。Orchestrator Agent 作为总控核心，负责任务拆解、能力路由、状态推进、上下文裁剪、错误处理和复核触发；Specialist Agents 负责各自能力域内的专业分析；Review Agent 负责高风险、低置信、冲突结论和最终报告草稿的复核；Result Aggregator 负责将底层事件和发现结果聚合为前端可读的 TaskSnapshot。

在情报与发现层，Enterprise Intel Agent 负责公司主体、法人、股东、备案、历史变更和企业关系采集；Asset Discovery Agent 负责子域名、DNS、IP、端口、CDN、HTTP 服务和基础路径发现；Fingerprint Agent 负责对已发现资产进行前后端技术栈、中间件、组件、CMS 和服务特征识别。

在解析与扩展层，JS Semantic Agent 负责从 JS/TS、HTML 和脚本上下文中提取 API 路径、参数、外联 URL、认证逻辑、固定材料和泄露线索；Dynamic Dictionary Agent 根据企业别名、指纹特征、业务命名和 JS 路径生成增量扫描字典；File Intel Agent 对可下载文件、配置文件、日志文件、SourceMap、备份文件和接口文档等进行只读解析；Passive Traffic Audit Agent 对 HTTP 请求与响应样本进行被动审查，识别未授权、CORS 异常、错误栈、DEBUG 信息、敏感参数和异常回显等风险线索。

在关联与输出层，Graph & Topology Agent 将企业实体、网络资产、服务、路径、文件、JS 线索、流量线索和证据引用统一建模，生成企业关系图谱和网络资产拓扑；Value Assessment Agent 基于资产重要性、技术栈特征、风险线索、业务关联性和证据完整度进行价值分级和优先级排序；Chain Hint Agent 基于图谱事实和多维证据进行保守的链路提示；Report Agent 将各模块结果汇总为结构化、可审计、可阅读的信息收集报告。

各 Agent 之间不直接通过 HTTP 调用彼此，也不直接共享内部内存对象。它们统一通过 AgentTask、AgentContext、ObservationEvent、EvidenceRef、Finding、ReviewTicket、TaskSnapshot 和持久化数据对象协作。这样既能保持模块边界清晰，也能保证每个发现都有来源、每次调度都有记录、每个结论都能追溯。

3.4 任务状态机与事件流

系统围绕任务生命周期建立统一状态机。一次信息收集任务从用户输入开始，首先由 API 层接收目标信息，再由应用层转换为 TaskCommand，由 Orchestrator 完成目标标准化、范围界定和任务拆解。之后系统按照“范围界定 -> 资产发现 -> 分析解析 -> 策略扩展 -> 图谱关联 -> 价值排序 -> 报告生成 -> 复核输出”的流程推进。

在资产发现阶段，系统收集企业主体、子域名、DNS、IP、端口、HTTP 服务和基础路径等事实，并为后续模块保存最小可审计样本。在分析解析阶段，系统对 HTTP 样本、指纹特征、JS 脚本、可下载文件和流量样本进行结构化分析，提取接口、路径、技术栈、敏感文件、风险线索和证据引用。在策略扩展阶段，系统根据已发现的企业别名、技术栈、业务命名和 JS 路径生成更有针对性的增量字典，并在受控范围内补充探测。

在图谱关联阶段，系统将多源结果写入统一图谱，形成公司、人员、域名、IP、端口、服务、路径、文件和证据之间的关系。在价值排序阶段，系统对资产和线索进行价值分级、优先级排序和核心业务判定。在报告生成阶段，系统将任务快照、图谱摘要、价值排序、链路提示和证据索引聚合为最终报告。在复核阶段，高风险、低置信、冲突或推断性较强的结论会进入 Review 流程，避免未经证据支撑的结论直接输出。

所有 Agent 的执行结果都以 ObservationEvent 形式进入事件流，并通过 dedupe_key、lineage、confidence、priority、scope 等字段维护去重、来源、置信度和作用范围。原始 HTML、JS、流量、文件和报告正文不直接塞入事件，而是以 EvidenceRef 引用方式关联到对象存储。前端查询时不直接消费底层事件表，而是读取 Result Aggregator 生成的 TaskSnapshot、发现列表、图谱视图和报告引用。

3.5 统一数据与存储架构

系统采用统一数据面承载任务、事件、资产、证据、图谱和报告，避免各 Agent 形成独立数据孤岛。PostgreSQL 用于保存任务主表、状态流转、Agent 执行记录、ObservationEvent、Finding、价值评分、审计记录和报告元数据，是系统结构化事实和状态回放的主要存储。

Redis 用于保存短期调度数据，包括任务队列、执行锁、去重键、Agent 心跳、上下文缓存和临时调度状态。它承担运行时加速和并发控制职责，但不作为最终事实来源；当 Redis 数据失效时，系统仍应能够依赖 PostgreSQL 中的事件和状态记录恢复任务视图。

Neo4j 用于保存企业关系图、资产图和网络拓扑图。企业主体、法人、股东、域名、IP、端口、服务、路径、证据和风险线索可以通过图节点与关系边组织起来，支撑前端图谱展示、资产归属分析、链路提示和后续人工研判。

MinIO 用于保存非结构化证据对象，包括 HTML 页面、JS 文件、HTTP 流量样本、可下载文件、解析后的文本片段、截图、工具原始输出和报告正文等。结构化结果只保存对象引用、哈希、来源和摘要，避免大文件直接进入事件表或关系库。

通过 PostgreSQL、Redis、Neo4j 和 MinIO 的分工，系统能够同时满足任务状态管理、调度效率、关系建模和证据留存需求。所有关键结论都必须能够回溯到 ObservationEvent、EvidenceRef、ToolCallRecord 和对应的 Agent 执行记录，保证结果可审计、可复核、可重建。

3.6 前端展示架构

前端采用 Vue3 构建统一交互界面，围绕任务创建、任务详情、资产发现、图谱拓扑、报告查看和系统状态展示组织页面。用户在前端输入公司名、域名、IP、URL 或任务配置后，前端只调用 Flask API 创建任务，不直接接触任何 Agent，也不直接调用扫描器或解析器。

任务执行过程中，前端通过 API 获取任务状态、阶段进度、当前运行 Agent、最近事件摘要、错误与重试信息。资产与发现列表展示来自 Result Aggregator 聚合后的结果，包括资产类型、技术栈、风险线索、置信度、价值等级和证据引用。图谱与拓扑页面展示 Neo4j 中的企业关系图、网络资产图和关键链路视图。报告页面展示 Report Agent 生成的结构化报告、摘要和导出结果。

前端展示层不承担状态推理、事件聚合和 Agent 协同逻辑。新增或调整某个后端 Agent 时，只要任务快照、发现列表、图谱和报告接口保持稳定，前端无需感知内部能力模块的变化。该设计保证了系统后续扩展 Agent 能力时，不会频繁破坏用户界面和对外协议。

3.7 架构安全边界与可扩展性

系统整体架构以授权范围、证据可追溯和低侵扰为基本边界。所有任务在开始前都应形成明确的目标范围和任务契约，各 Agent 只能在 Orchestrator 下发的范围内执行，不允许自行扩大目标边界。文件情报、JS 解析和流量审查等模块只做只读分析、结构化提取和风险线索标记，不执行自动化攻击利用。

在质量控制方面，系统通过事件记录、证据引用、置信度标记、复核工单和报告审查机制降低误判风险。高价值但证据不足、多个模块结论冲突、链路提示推断性较强或报告中出现高风险表述时，应进入 Review 流程，由复核模块或人工介入确认。非核心模块失败时，系统支持降级输出和部分完成，避免单个能力不可用导致整个任务完全失败。

在扩展性方面，第一阶段采用模块化单体后端以降低落地复杂度，但架构上保留异步 Worker 演进空间。后续当基础发现、JS 解析、文件情报或流量审查等高负载模块需要独立扩展时，可以将 Agent Executor 迁移到后台 Worker 或任务队列中执行；对外仍保持同一个 Flask API 后端，前端协议、任务接口、事件模型和 Agent 契约保持稳定。

因此，本项目的系统架构既能支撑第一阶段快速形成可运行闭环，也能为后续高并发、多任务、多 Agent 扩展和更复杂的图谱分析能力预留空间。

四、项目功能设计

本项目围绕“发现—解析—研判—输出”的主链路进行功能设计，在保留传统信息收集基础能力的同时，引入多 Agent 协同、统一数据面、图谱聚合和智能排序能力，使信息收集不再停留在单点扫描和零散结果堆积，而是形成面向后续渗透工作的完整信息闭环。

4.1 统一任务接入与智能编排

系统首先提供统一任务接入能力，将公司名、域名、IP、URL 等输入进行标准化处理，形成统一的任务入口。总控 Agent 负责对目标进行规范化、范围界定和任务建模，避免前端输入直接散落到各个功能模块中，保证系统执行边界清晰、目标范围可控。

在任务进入系统后，总控 Agent 会根据输入目标类型、信息收集目的和当前阶段结果，对各能力模块进行统一编排。对于仅需要企业情报收集的任务，可以优先调度企业情报能力；对于需要更深入应用层分析的任务，则可以联动基础发现、JS 解析、文件情报和流量审查等模块。通过这种统一编排机制，系统能够根据任务场景动态选择能力路径，减少无效流程和重复工作。

同时，系统具备任务状态推进、错误处理、降级恢复和结果聚合能力。即使某一子模块执行失败，也不会直接导致整任务终止，而是可以通过部分完成、重试、补跑或复核等方式继续推进，确保最终输出尽可能完整、稳定、可审计。

4.2 企业情报与主体关系发现

在企业情报层面，系统支持围绕目标域名或企业主体开展备案信息、公司主体信息、法人信息、股东信息、对外投资、历史变更等内容的采集与整合。该能力解决的是传统工商信息来源分散、字段不统一、人工整理成本高的问题，使企业情报能够直接进入后续分析流程，而不是停留在零散查询结果层面。

系统不仅关注单一企业实体，还强调主体关系的发现与组织。通过对公司与公司、公司与个人、个人与个人之间关系的整理，系统能够形成企业主体关系图谱，为资产归属分析、业务背景判断和后续攻击面梳理提供背景支撑。

该部分功能与后续图谱聚合、价值评估和报告输出紧密衔接。企业情报结果不仅可以作为独立输出使用，也可以进一步与备案主体、网站资产、域名和网络拓扑建立对应关系，为后续“企业主体—网络资产—风险线索”的统一关联打下基础。

4.3 基础资产发现与技术特征识别

在基础发现层面，系统具备子域名收集、DNS 解析、IP 发现、端口探测、CDN 识别、HTTP 服务确认和敏感路径发现等能力，用于建立目标最基础的资产画像。该部分继承并整合传统信息收集中的常见能力，但重点不在于简单重复工具功能，而在于将基础发现结果纳入统一任务流和统一数据面中。

在此基础上，系统进一步对发现到的资产进行技术特征识别。通过对 HTTP 响应、页面片段、服务特征、头信息和脚本特征的分析，系统能够识别前后端技术栈、中间件、组件、CMS 和关键服务特征，帮助用户快速理解目标系统所采用的技术体系。

基础资产发现与技术特征识别共同构成整个系统的事实底座。前者回答“发现了什么资产”，后者回答“这些资产是什么类型、具有什么特征”，二者为动态字典生成、JS 深度解析、价值评估和后续图谱建模提供关键输入。

4.4 应用层深度解析与动态扩展

针对传统信息收集在应用层容易停留于页面表象的问题，系统设计了应用层深度解析能力。其核心包括对 JS/TS 代码、页面上下文和脚本依赖关系的语义分析，用于提取隐藏 API 路径、参数、外联 URL、认证逻辑、调试信息、环境标识、固定材料和敏感线索等高价值内容。

这部分功能不仅能补全网页层面无法直接观察到的系统结构，还能帮助识别隐藏接口、业务依赖面和测试/调试痕迹，使应用层信息收集从“看页面”升级为“理解页面背后的行为逻辑”。这些结果可以进一步反馈到图谱聚合和价值评估中，提高后续判断的完整性和准确性。

基于应用层和技术特征的已有发现，系统还具备动态扩展能力。动态扫描字典可以结合企业别名、业务命名、技术栈特征和已提取路径信息生成更有针对性的增量扫描线索，从而减少固定字典带来的冗余探测，提高敏感路径和关键接口的命中效率。这样，系统可以在已有发现基础上进行受控补跑和扩展，而不是盲目扩大扫描范围。

4.5 文件情报与被动流量审查

在文件情报层面，系统对扫描过程中发现的可下载文件和敏感文件进行识别、下载和只读解析。其关注对象包括配置文件、日志文件、备份文件、数据库导出、SourceMap、接口文档、压缩包、版本控制残留和证书密钥文件等。通过自动识别文件类型并提取其中的账号、路径、接口、部署信息和敏感配置，系统可以大幅降低人工逐个文件分析的成本。

在流量审查层面，系统通过对 HTTP 请求与响应样本进行被动式分析，识别未授权访问、CORS 异常、错误栈泄露、DEBUG 信息、敏感参数、异常回显和弱凭证传输等风险线索。与传统仅关注静态页面和资产列表的方式不同，该能力能够把隐藏在交互过程中的高价值迹象提前暴露出来。

文件情报与被动流量审查分别从“静态文件内容”和“动态交互过程”两个维度补足传统信息收集的盲区。前者强调文件载荷中的敏感信息，后者强调通信过程中的风险行为，二者共同为价值评估、链路提示和最终报告输出提供更丰富的证据来源。

4.6 图谱聚合、价值评估与链路提示

系统在多源信息收集完成后，并不直接将结果按发现顺序堆叠输出，而是通过图谱聚合能力将企业实体、网络资产、技术特征、应用线索、文件情报和流量风险统一组织为可关联、可查询的事实底座。这样，系统能够形成企业关系图谱、资产关系图谱和网络拓扑结构，帮助用户从整体视角理解目标。

在此基础上，系统进一步引入价值评估能力，对资产重要性、技术特征、风险线索、业务关联性和证据完整度进行综合判断，形成高价值目标标记和优先级排序。这使得信息收集结果不再只是“发现了哪些资产”，而是进一步回答“哪些资产最值得优先关注、为什么值得关注”。

对于传统流程中依赖人工梳理的潜在利用关系，系统通过链路提示能力给出保守的关联分析。该能力结合 IP 网段、证书、DNS、跳转关系、前端资源特征、图谱事实和多维度交叉证据，对可能存在的业务依赖、资产关联和后续测试方向进行提示。链路提示强调辅助研判和证据支撑，不替代人工判断，但可以显著降低信息收集之后的二次整理成本。

4.7 报告输出与结果交付

在结果输出层，系统对任务全过程中的发现结果、图谱关系、价值排序、风险线索和证据引用进行统一汇总，形成结构化的任务快照和综合信息收集报告。报告不只是简单罗列扫描结果，而是围绕企业情报、资产发现、技术特征、应用层线索、文件情报、流量风险、图谱关系和优先目标等维度进行组织，使后续渗透测试人员可以直接基于报告开展下一步工作。

同时，系统支持图谱视图、发现列表、证据引用和报告摘要等多种交付方式。这样既能满足机器处理和平台展示的需要，也能满足人工阅读、复核和归档的需求。所有重要结论都保留来源和证据引用，保证输出结果不仅可读，而且可追溯、可审计。

通过这一功能设计，本项目实现了从任务接入、资产发现、应用解析、情报补全、风险识别、关系建模、价值排序到结果交付的完整闭环。系统既具备传统信息收集所需的基础功能，又通过多 Agent 协同和统一结果组织方式强化了信息之间的关联性、优先级和后续可用性。

五、项目创新与技术亮点

本项目的创新点不只体现在单个扫描能力的增强，而是体现在信息收集流程整体范式的升级：从“多个工具分别执行、人工整理结果”的传统模式，转向“统一任务入口、Agent 协同编排、多源证据沉淀、图谱化关联、价值化输出”的智能信息收集闭环。系统在授权范围内对企业情报、网络资产、应用层线索、文件情报和流量风险进行统一分析，不执行自动化攻击利用，而是为后续渗透测试提供更完整、更可解释、更可复核的信息基础。

5.1 多 Agent 协同的信息收集闭环创新

传统信息收集往往依赖渗透测试人员手动串联多个工具，工具之间缺少统一调度、统一上下文和统一结果模型。本项目将信息收集拆分为企业情报、基础发现、指纹识别、JS 解析、动态字典、文件情报、流量审查、图谱聚合、价值评估、链路提示和报告输出等多个能力域，并由 Orchestrator 在统一 Flask 后端内部进行调度。各 Agent 不直接互相调用，而是通过任务、事件、上下文、证据引用和状态机协作，使系统能够在保持模块边界清晰的同时形成完整闭环。

该设计的创新价值在于：系统不再把 Agent 简单作为“会总结的工具外壳”，而是把 Agent 纳入可编排、可追踪、可降级、可复核的运行时体系中。Orchestrator 根据任务目标和阶段结果动态决定能力路由，Specialist Agent 只负责各自专业领域，Review Agent 负责高风险或低置信结论复核，Result Aggregator 负责生成前端可读的任务快照。这样既提升了自动化程度，也避免多个 Agent 自由调用导致流程失控。

5.2 主动探测与被动审查融合创新

本项目同时引入主动资产发现与被动流量审查，解决传统信息收集“只看静态结果、不看交互过程”的问题。主动发现模块负责子域名、DNS、IP、端口、HTTP 服务、路径和指纹等基础事实；被动流量审查模块则从请求与响应样本中识别未授权访问迹象、CORS 异常、错误栈泄露、DEBUG 信息、敏感参数、异常回显等风险线索。二者不是割裂运行，而是通过统一事件流形成反馈关系。

当被动审查发现高价值风险迹象时，相关资产会被送入价值评估、链路提示或复核流程；当主动发现产生新的页面、接口、文件或服务样本时，又会为被动审查提供新的分析对象。通过这种主动与被动融合的方式，系统能够在信息收集阶段提前发现隐藏于交互过程中的高价值线索，减少后续人工重复审查。

5.3 面向目标环境的动态字典与策略扩展

固定字典是传统目录扫描和路径发现效率低的重要原因。本项目的动态字典能力不是简单扩充通用字典，而是根据目标环境自动生成更具上下文关联的增量探测线索。系统会综合企业名称、企业简称、备案主体、业务命名、技术栈指纹、历史命中路径、JS 中提取的 API 路径和文件情报中出现的内部路径，形成贴合目标的候选路径、接口名和资源名。

该能力的关键创新在于“以发现反哺发现”。基础发现、指纹识别和 JS 解析产生的结果会进入动态字典 Agent，动态字典 Agent 再生成受控增量词典，交由 Orchestrator 判断是否补跑基础发现。补跑过程受任务边界、最大展开深度、最小信息增益和去重规则约束，避免盲目扩大扫描范围或进入循环。

5.4 JS 语义解析、文件情报与流量情报的多源融合

传统信息收集对网页和应用层的理解通常停留在页面标题、状态码、指纹和可见路径层面，难以系统性挖掘前端脚本、可下载文件和 HTTP 交互中的隐含线索。本项目将 JS 语义解析、文件情报和被动流量审查作为应用层深度理解的三类关键来源：JS 解析负责提取组件结构、接口路径、外联 URL、认证逻辑、固定材料、注释和调试残留；文件情报负责解析配置、日志、备份、SourceMap、接口文档和部署文件；流量审查负责识别交互过程中的风险行为和异常响应。

这三类能力共同解决“信息藏在代码里、文件里、流量里”的问题。系统不会把这些结果直接写成孤立告警，而是统一转换为标准发现、证据引用和图谱关系，使应用层线索能够与企业主体、网络资产、技术指纹和报告输出自然衔接。

5.5 统一数据面、证据链与图谱化事实底座

本项目的另一项核心技术亮点是统一数据面。系统不让各 Agent 各自保存零散结果，而是将任务、状态、事件、发现、证据、图谱、评分和报告纳入同一套数据模型中。PostgreSQL 保存任务、事件、发现、状态流转和报告元数据；Redis 承担短期调度、锁、去重和缓存；Neo4j 保存企业关系图、资产关系图和网络拓扑；MinIO 保存 HTML、JS、流量、文件和报告正文等非结构化证据对象。

统一数据面的价值在于让每个结论都能追溯到来源。系统中的重要发现不会只以自然语言描述存在，而是能够回到 ObservationEvent、EvidenceRef、ToolCallRecord、AgentExecutionRecord 和对应证据对象。这样既能支撑前端图谱展示，也能支撑报告复核、问题排查和后续审计。

5.6 价值评估与链路提示的智能研判能力

传统信息收集报告往往按发现顺序罗列结果，高价值目标容易被大量普通资产淹没。本项目引入价值评估 Agent，对资产重要性、技术栈特征、风险线索、业务关联性、证据完整度等因素进行综合研判，形成优先级标记和高价值目标排序。系统能够突出主站、统一认证入口、管理后台、核心 API、关键数据面、CI/CD 控制面和数据库管理面等值得优先关注的对象。

在价值排序基础上，链路提示 Agent 会结合图谱事实和多维证据进行保守关联推断，例如基于 IP 网段、SSL 证书、DNS 链路、HTTP 跳转、前端资源哈希、接口调用关系和第三方资源引用，提示可能存在的业务依赖或资产关联。链路提示只作为辅助研判，不替代人工判断；所有推断性结论均应附带证据来源、置信度和复核标记。

5.7 工程可落地性、安全边界与可扩展能力

本项目在设计上兼顾创新性与工程落地性。第一阶段采用模块化单体后端，而不是过早拆分多个独立服务，降低部署、调试和联调复杂度；同时通过 Agent Runtime、统一事件模型、工具适配层和数据层隔离，为后续异步 Worker、队列执行、多任务并发和能力扩展预留空间。对外始终保持一个 Flask API 后端和一个 Vue3 前端，内部执行方式变化不会破坏前端协议。

安全边界方面，系统以授权范围、低侵扰、只读分析和证据可追溯为基本原则。任务开始前由 Orchestrator 形成目标范围和任务契约，各 Agent 只能在契约范围内运行；JS 解析、文件情报和流量审查仅做信息提取、风险标记和辅助研判，不执行自动化攻击利用；当出现范围漂移、状态异常、证据冲突或高风险推断时，系统进入 Review 或隔离流程，防止错误结论扩散。

5.8 技术亮点总结

技术亮点核心作用相比传统方式的提升
多 Agent Runtime将多个能力域 Agent 纳入统一后端运行时避免工具割裂和 Agent 自由调用失控
Orchestrator 编排统一完成任务拆解、能力路由、状态推进和错误处理从人工串联工具升级为事件驱动协同
ObservationEvent 事件流将 Agent 输出沉淀为可去重、可回放、可审计事件解决结果不可追踪、状态不可恢复的问题
EvidenceRef 证据引用将结论与 HTML、JS、文件、流量等原始证据关联保证报告结论可复核、可解释
Neo4j 图谱建模构建企业关系图、资产图和网络拓扑从资产列表升级为关系化事实底座
MinIO 证据对象存储保存大体积非结构化证据和报告正文避免原始证据丢失或污染结构化事件表
Review 复核机制对高风险、低置信、冲突和推断性结论进行质量控制降低误判扩散，提高输出可信度
Result Aggregator 聚合输出生成任务快照、图谱视图、发现列表和报告引用前端无需拼接底层事件，展示结果更稳定
动态字典生成根据企业、技术栈、JS 路径和历史发现生成增量词典降低无效扫描，提高目标相关路径命中率
主动与被动融合联动资产发现、HTTP 样本和流量风险线索提前暴露隐藏于交互过程中的高价值迹象
价值评估与链路提示对资产优先级和潜在关联路径进行辅助研判从“发现资产”提升到“指导后续测试方向”
模块化单体可演进架构一期降低落地复杂度，后续可迁移到 Worker兼顾快速实现和长期扩展能力

综上，本项目的技术亮点可以概括为：以 Orchestrator 为核心建立可控的多 Agent 协同体系，以 ObservationEvent 和 EvidenceRef 建立可审计的数据流，以 Neo4j 和统一数据面建立可关联的事实底座，以价值评估和链路提示提升结果可用性，并以模块化单体到异步 Worker 的演进路线保证工程落地。该设计使信息收集从“工具集合”升级为“智能协同系统”，能够显著提升信息收集的完整性、准确性、关联性和后续渗透测试准备效率。

六、应用场景与商业价值

本项目定位为面向授权场景的信息收集与情报整理基础平台，其价值不仅体现在“替代若干单点工具”，更体现在把企业情报、资产发现、应用层线索、文件情报、流量风险、图谱关系、价值排序和报告交付组织为一个统一闭环。因此，它既适用于渗透测试前期的信息收集，也适用于企业安全团队的持续资产梳理、重点目标排查和安全服务的标准化交付。

6.1 授权渗透测试与红队前置信息收集

在授权渗透测试、攻防演练和红队评估场景中，前期信息收集往往决定后续测试效率。本项目可以围绕企业主体、备案关系、域名、IP、端口、路径、技术栈、JS 接口、文件泄露和流量异常等多个维度同步展开分析，并通过统一任务入口、统一结果视图和统一证据链减少人工在多工具之间切换、整理和比对的成本。对于测试工程师而言，系统不仅回答“发现了什么”，还能够进一步回答“哪些目标更值得优先关注、彼此之间可能存在什么关联、这些结论的证据来自哪里”，从而显著缩短进入后续验证阶段前的准备时间。

6.2 企业内部安全运营与攻击面梳理

对于企业内部安全团队、攻防实验室和安全运营部门而言，本项目可作为持续性攻击面梳理与资产认知平台使用。系统能够把原本分散在工商信息、备案信息、子域名、网络资产、前端脚本、可下载文件和 HTTP 流量中的线索沉淀到统一数据面和图谱模型中，帮助团队从“零散资产台账”升级为“可查询、可追溯、可排序的事实底座”。这类能力尤其适用于集团化企业、多业务线企业和资产边界复杂的组织，用于梳理主体归属、识别影子资产、发现高价值入口、支撑年度攻防演练准备和长期资产治理工作。

6.3 重点时期安全保障与专项排查

在重大活动保障、专项安全检查、上线前审查和高风险时期排查等场景下，安全团队通常需要在有限时间内快速完成资产收敛、重点目标识别和证据留存。本项目的价值在于，它能够通过动态字典、被动流量审查、价值评估和链路提示等能力，对主站、统一认证、管理后台、核心 API、关键数据面和控制面进行优先排序，将有限的排查资源倾斜到更可能影响核心业务的目标上。同时，系统保留完整事件记录、证据引用和报告输出链路，便于在重点时期形成可归档、可复核的专项成果。

6.4 安全服务标准化交付与平台化复用

对于安全服务商、测评机构和内部共享安全平台团队而言，本项目具备较强的标准化交付价值。其一，系统通过 Result Aggregator 和报告输出模块，将多 Agent 产生的异构数据统一为任务快照、发现列表、图谱视图和结构化报告，降低不同项目、不同人员之间的交付差异。其二，系统的统一数据面和证据链设计使结果能够被二次沉淀和复用，为后续项目形成经验库、字典库、规则库和案例库提供基础。其三，系统采用“一个 Vue3 前端 + 一个 Flask API 后端 + 后端内部多 Agent Runtime”的聚合形态，既便于一期快速落地，也便于后续在私有化部署、异步 Worker 扩展、内部平台对接等方向演进。

6.5 项目商业价值

从商业价值角度看，本项目主要体现在以下几个方面。第一，是效率价值。系统通过自动化编排、统一结果聚合和报告生成，显著降低信息收集阶段的重复劳动，提高单个项目的处理效率和团队的人均产出能力。第二，是质量价值。通过统一事件模型、证据引用、复核流程和价值排序，输出结果不再只是“工具结果堆积”，而是具备更强的完整性、关联性、解释性和可审计性。第三，是平台价值。系统并不依赖某一个单一工具，而是通过 Agent、适配器和统一数据面形成可扩展的平台底座，后续可以持续接入新的数据源、分析模块和交付形式。第四，是沉淀价值。企业情报、资产关系、技术特征、风险线索和报告结果能够持续积累，形成面向长期安全工作的知识资产，而不是一次性消耗型输出。

因此，本项目的应用价值不仅面向单次渗透测试任务，更面向安全服务标准化、企业攻击面治理和长期情报积累。它能够帮助用户从“人工驱动的信息收集流程”转向“系统驱动的智能信息收集闭环”，在提升效率的同时增强结果的后续可用性和业务支撑能力。
